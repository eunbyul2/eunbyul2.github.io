# Linux inode란 무엇인가, 그리고 거기에 저장되지 않는 정보는 무엇인가

## 1. 우리가 "파일"이라고 부르는 것은 사실 두 개다

터미널에서 파일을 다룰 때 우리는 보통 이렇게 생각합니다.

```text
/home/eunbyul/report.txt 라는 파일이 하나 있다
```

이름 하나, 파일 하나. 직관적으로는 이게 맞는 것 같지만, 리눅스 파일시스템 내부에서는 이 하나의 "파일"이 사실 서로 다른 두 가지로 나뉘어 관리됩니다.

- **이름(경로)** — `report.txt`라는 문자열. 이건 그냥 "어떤 대상을 가리키는 라벨"일 뿐입니다.
- **실체** — 그 파일의 권한, 크기, 수정 시간, 그리고 실제 데이터가 디스크 어디에 있는지 같은 정보. 이걸 관리하는 게 바로 **inode**입니다.

비유를 들면 이렇습니다. 택배 상자(실제 데이터 + 메타데이터)에 붙어 있는 "받는 사람 이름표(파일 이름)"를 생각해보세요. 이름표는 상자를 찾아가기 위한 수단일 뿐, 상자 자체가 아닙니다. 같은 상자에 이름표를 두 개 붙일 수도 있고(하드 링크), 이름표만 바꿔 붙일 수도 있습니다(이름 변경). 상자 안의 내용물은 그대로인데 말이죠.

즉 리눅스에서 파일에 접근하는 흐름은 이렇게 3단계로 이루어집니다.

```text
파일 이름 (디렉터리 엔트리)
   ↓
inode 번호로 연결
   ↓
inode (메타데이터 + 데이터 위치 정보)
   ↓
실제 데이터 블록
```
---

## 2. inode란 정확히 무엇인가

**inode**는 index node의 줄임말로, 파일이나 디렉터리 하나마다 만들어지는 **메타데이터 저장 구조**입니다. 
여기서 중요한 건 inode는 "파일 내용"이 아니라 "파일에 대한 설명서"라는 점입니다.

주민등록증에 비유하면 이해가 쉽습니다.

- 주민등록번호(inode 번호)는 사람(파일)을 식별하는 번호입니다.
- 주민등록증 안에는 생년월일, 주소, 사진 등 그 사람에 대한 정보(메타데이터)가 담겨 있습니다.
- 하지만 주민등록증 안에 "이 사람을 부르는 애칭"은 적혀 있지 않습니다. 애칭(파일 이름)은 주변 사람들(디렉터리)이 기억하는 것이지, 그 사람 자신의 고정된 속성이 아닙니다.

이 비유를 기억해두면 뒤에서 나올 "왜 inode에는 파일 이름이 없는가"가 자연스럽게 이해됩니다.

```bash
ls -li
```

```text
266659 drwx------+ 40 eunbyul staff 1280 7 11 06:06 Downloads
```

가장 왼쪽의 `266659`가 이 `Downloads` 디렉터리의 inode 번호입니다. 이 번호를 통해 커널은 실제 inode 객체(권한, 크기, 데이터 위치 등이 담긴 구조체)를 디스크에서 찾아갑니다.

한 가지 주의할 점: inode 번호는 **하나의 파일시스템 안에서만** 유일합니다. 디스크를 여러 파티션으로 나눴다면, 파티션 A의 inode 100번과 파티션 B의 inode 100번은 완전히 다른 파일입니다. 그래서 커널이 파일을 진짜로 구분할 때는 "이 파일시스템 + 이 inode 번호" 조합을 함께 사용합니다.

## 3. 파일 이름은 어디에 저장되는가 — 디렉터리 엔트리

앞서 inode에는 파일 이름이 없다고 했는데, 그럼 이름은 대체 어디 있을까요? 
답은 **디렉터리** 자체입니다.

디렉터리는 겉보기엔 그냥 파일들을 담는 "폴더"처럼 보이지만, 리눅스 내부에서 디렉터리는 사실 **"이름 → inode 번호" 매핑 정보를 담고 있는 특수한 파일**입니다. 이 매핑 하나하나를 **디렉터리 엔트리(directory entry)** 라고 부릅니다.

예를 들어 홈 디렉터리 안에 이런 항목들이 있다고 합시다.

```text
Desktop
Documents
Downloads
```

디렉터리 내부적으로는 다음과 같은 표가 저장되어 있는 셈입니다.

```text
이름          →  inode 번호
Desktop       →  266613
Documents     →  266657
Downloads     →  266659
```

즉 `Downloads`라는 문자열은 `Downloads` 디렉터리의 inode 안에 있는 게 아니라, **부모 디렉터리**가 들고 있는 매핑 테이블에 있는 겁니다. `Downloads`의 inode(266659) 자신은 "내가 Downloads라고 불린다"는 사실을 전혀 모릅니다.

## 4. inode에는 실제로 어떤 정보가 들어있나

파일시스템(ext4, XFS, Btrfs 등)마다 세부 구현은 다르지만, 일반적으로 inode에는 다음 정보가 저장됩니다.

| 항목 | 설명 |
| --- | --- |
| 파일 유형 | 일반 파일 / 디렉터리 / 심볼릭 링크 / 소켓 / FIFO 등 |
| 접근 권한 | 소유자·그룹·기타에 대한 읽기/쓰기/실행 권한 (`rwx`) |
| 소유자 UID, 그룹 GID | 이름이 아니라 **숫자**로 저장 |
| 파일 크기 | 논리적 크기 |
| 하드 링크 수 | 이 inode를 가리키는 디렉터리 엔트리의 개수 |
| 타임스탬프 | atime / mtime / ctime |
| 데이터 위치 | 실제 데이터 블록(또는 extent)의 주소 |
| 파일시스템 플래그 | immutable 등 특수 속성 |

### 4-1. UID/GID는 "숫자"로 저장된다

`ls -l`을 치면 소유자가 `eunbyul`처럼 사람이 읽을 수 있는 이름으로 나오지만, 실제로 inode 안에 저장된 값은 `UID 501`, `GID 20` 같은 **숫자**입니다. `eunbyul`이라는 문자열은 시스템이 `/etc/passwd` 같은 사용자 데이터베이스에서 501번을 조회해서 보여주는 변환 결과일 뿐입니다.

```bash
ls -ln    # 이름 대신 숫자 UID/GID를 그대로 보여줌
```

디스크를 다른 시스템으로 옮기거나 NFS로 공유할 때 **같은 UID 501번이 그 시스템에서는 다른 사람**일 수 있기 때문에 중요한 부분입니다.
파일 소유자가 갑자기 이상한 이름으로 바뀐 것처럼 보이는 사고가 여기서 발생합니다.

### 4-2. 파일 크기(inode)와 실제 디스크 사용량(블록)은 다를 수 있다

```bash
truncate -s 1G sparse.img
ls -lh sparse.img   # 1.0G  ← inode에 기록된 "논리적 크기"
du -h sparse.img     # 0     ← 실제로 디스크에 쓰인 블록의 양
```

`ls`는 inode에 저장된 논리적 크기를 보여주고, `du`는 진짜로 할당된 블록의 양을 보여줍니다. sparse file(듬성듬성 파일)처럼 "크기는 크다고 되어 있지만 실제로는 거의 안 쓴" 파일이 존재할 수 있는 이유입니다.

### 4-3. ctime은 생성 시간이 아니다

- **atime**: 파일을 마지막으로 읽은(access) 시간
- **mtime**: 파일 내용이 마지막으로 바뀐(modify) 시간
- **ctime**: inode 자체(권한, 소유자, 링크 수 등 메타데이터)가 마지막으로 바뀐(change) 시간

이름 때문에 ctime을 creation time(생성 시간)으로 착각하기 쉬운데, 사실은 change time입니다. 파일 생성 시각은 일부 최신 파일시스템에서 별도로 "birth time / crtime"이라는 이름으로 관리합니다.

## 5. inode에 저장되지 않는 정보

### 5-1. 파일 이름

앞서 봤듯 파일 이름은 inode가 아니라 부모 디렉터리의 디렉터리 엔트리에 저장됩니다. inode는 자기 자신이 무슨 이름으로 불리는지 알지 못합니다.

이게 왜 필요한 설계인지는 4장에서 본 "하나의 inode에 여러 이름"이 가능해야 하기 때문입니다. 만약 inode 안에 이름을 고정으로 저장해버리면, 하드 링크나 이름 변경 같은 기능 자체가 불가능해집니다.

### 5-2. 전체 경로

`/home/eunbyul/report.txt` 같은 전체 경로도 inode 어디에도 저장되어 있지 않습니다. 경로는 실체가 아니라, 루트 디렉터리부터 시작해서 디렉터리 엔트리를 하나씩 순서대로 따라가며 **매번 새로 조립되는 결과물**입니다.

```text
/  → "home" 찾기 → home의 inode
   → "eunbyul" 찾기 → eunbyul의 inode
   → "report.txt" 찾기 → report.txt의 inode
```

그래서 같은 파일시스템 안에서 파일을 다른 디렉터리로 옮겨도(`mv`) inode 번호는 그대로 유지됩니다. 옮겨진 파일이 "예전과 같은 파일"이라는 걸 이렇게 확인할 수 있습니다.

### 5-3. 사용자 이름 / 그룹 이름

4-1에서 다뤘듯, inode에는 `eunbyul`, `staff` 같은 문자열이 아니라 UID/GID 숫자만 있습니다. 이름은 어디까지나 OS가 보여주기 위해 변환한 결과입니다.

### 5-4. 일반적인 파일의 실제 내용

파일 내용(예: report.txt 안의 텍스트)은 보통 inode 안이 아니라 **별도의 데이터 블록**에 저장됩니다. inode는 그 데이터가 어디 있는지 가리키는 위치 정보만 가지고 있습니다.

```text
inode
 ├── (메타데이터)
 └── 데이터 블록 위치: "블록 5000~5099"
                          ↓
                      실제 파일 내용
```

다만 예외가 있습니다. 
아주 짧은 심볼릭 링크의 대상 경로처럼 데이터가 매우 작을 경우, 일부 파일시스템은 이를 별도 블록 대신 inode 내부의 여유 공간에 바로 저장하는 최적화를 합니다(이를 **fast symlink** 또는 **inline data**라고 부릅니다). 그래서 "파일 내용은 절대로 inode에 없다"보다는 "일반적으로는 없지만, 아주 작은 데이터는 예외적으로 있을 수 있다"가 더 정확한 설명입니다.

### 5-5. 그 파일을 열고 있는 프로세스 정보

어떤 프로세스가 지금 이 파일을 열어서 쓰고 있는지도 inode에는 없습니다. 이 정보는 커널이 메모리에서 별도로 관리하는 "파일 디스크립터 테이블"과 "열린 파일 객체"가 담당합니다.

```text
프로세스 → 파일 디스크립터 → 열린 파일 객체 → inode
```

## 6. 이름과 inode를 분리했기 때문에 가능한 것들

### 6-1. 하드 링크 — 하나의 inode, 여러 이름

```bash
ln original.txt hardlink.txt
```

이 명령은 파일 내용을 복사하지 않습니다. 그저 `hardlink.txt`라는 새 이름을 만들어서 `original.txt`와 **똑같은 inode**를 가리키게 할 뿐입니다.
즉, 하나의 inode를 여러 이름이 동시에 가리킬 수 있음을 알 수 있습니다.

```text
original.txt → inode 100
hardlink.txt → inode 100   (같은 inode!)
```

그래서 둘은 다음 특성을 완전히 공유합니다.

```text
같은 inode 번호, 같은 데이터, 같은 권한, 같은 소유자, 같은 크기, 같은 타임스탬프
```

한쪽으로 내용을 수정하면 다른 쪽에서도 바뀐 내용이 보입니다. 복사본이 아니라 진짜로 같은 실체를 가리키고 있기 때문입니다. 
(하드 링크는 서로 다른 파일시스템을 넘어서는 만들 수 없습니다. inode 번호가 그 파일시스템 안에서만 의미가 있기 때문)

### 6-2. 이름 변경이 순식간에 끝나는 이유

```bash
ls -id Downloads         # 266659 Downloads
mv Downloads MyDownloads
ls -id MyDownloads       # 266659 MyDownloads  ← inode 번호가 그대로!
```

`mv`로 이름을 바꿔도 inode 번호는 바뀌지 않습니다. 실제로 일어난 일은 "데이터를 다시 쓰는 것"이 아니라, 부모 디렉터리의 매핑 테이블에서 이름표만 갈아 끼운 것뿐입니다. 그래서 파일이 아무리 커도 같은 파일시스템 안에서의 이름 변경/이동은 거의 즉시 끝납니다.

![ls -id Downloads 결과](/img/inode-test-1.png)

`Downloads`의 inode 번호가 `266659`인 것을 확인한 뒤, `MyDownload`로 이름을 바꾸고 다시 `Downloads`로 되돌려도 inode 번호는 계속 `266659`로 동일합니다.

![mv로 이름을 바꿔도 inode 번호가 유지되는 결과](/img/inode-test-2.png)

이름표만 두 번 바뀌었을 뿐, 가리키는 실체(inode)는 처음부터 끝까지 그대로였다는 뜻입니다.

반대로 **다른 파일시스템으로** 옮기거나(`mv` across filesystems) `cp`로 복사하면, 새로운 inode를 만들고 데이터도 실제로 복사해야 하므로 시간이 걸립니다.

```text
같은 파일시스템 안 이동  → 디렉터리 엔트리만 변경 (빠름)
다른 파일시스템 간 이동  → 새 inode 생성 + 데이터 복사 (느림)
```

### 6-3. 삭제했는데 디스크 용량이 안 줄어드는 이유

```bash
rm application.log
```

`rm`이 실제로 하는 일은 "데이터를 지우는 것"이 아니라 **디렉터리 엔트리를 제거하고, 그 inode의 링크 수를 1 줄이는 것**입니다.

```text
링크 수가 0이 되고
+ 그 파일을 열고 있는 프로세스가 없어야
→ 비로소 inode와 데이터 블록이 회수된다
```

만약 어떤 프로세스가 그 로그 파일을 계속 열어놓고 쓰는 중이라면, 디렉터리에서는 이름이 사라졌어도 커널은 "열린 파일 객체"를 통해 여전히 그 inode를 참조하고 있습니다. 그래서 `rm`을 했는데도 `df -h`로 보면 디스크 공간이 회수되지 않는 상황이 생깁니다.

```bash
sudo lsof | grep deleted   # 삭제됐지만 여전히 열려 있는 파일 확인
```

이 현상은 5-5에서 본 "열고 있는 프로세스 정보는 inode가 아니라 커널이 별도로 관리한다"는 사실 때문에 벌어지는 겁니다.

## 7. 심볼릭 링크는 하드 링크와 완전히 다르다

심볼릭 링크는 겉보기엔 비슷해 보이지만 원리가 정반대입니다. 심볼릭 링크는 **자기 자신만의 별도 inode**를 가지고, 그 안에 원본의 경로를 "문자열"로 저장합니다.

```bash
ln -s original.txt symlink.txt
```

```text
symlink.txt → (별도의 새로운 inode) → 안에는 "original.txt"라는 문자열만 있음
```

즉 하드 링크는 "같은 실체를 공유"하는 것이고, 심볼릭 링크는 "다른 실체가 원본의 위치를 적어둔 메모"에 가깝습니다.

```text
하드 링크:   original.txt와 hardlink.txt가 동일한 inode 100을 직접 공유
심볼릭 링크:  symlink.txt는 별도의 inode를 가지며, 그 안에 "original.txt"라는 경로만 적혀 있음
```

그래서 원본 파일이 삭제되면 심볼릭 링크는 더 이상 유효하지 않은 대상을 가리키게 됩니다(dangling symbolic link). 반면 하드 링크는 원본이 지워져도 (링크 수가 남아있는 한) 멀쩡히 살아남습니다.

---

## 8. 디스크는 남았는데 "No space left on device"가 뜨는 이유 — inode 고갈

```bash
df -h    # 디스크 용량 확인
df -i    # inode 개수/사용량 확인
```

```text
Filesystem   Inodes   IUsed   IFree   IUse%
/dev/sda1   1000000 1000000      0    100%
```

파일 하나는 크기가 1바이트든 1기가바이트든 상관없이 기본적으로 **inode 하나를 소비**합니다. 그래서 아주 작은 파일을 수백만 개 만들면, 디스크 용량 자체는 여유가 있어도 inode가 먼저 바닥날 수 있습니다. 이때 뜨는 에러가 바로 `No space left on device`입니다 — 데이터 블록이 아니라 inode가 부족해서 나는 에러라는 게 헷갈리는 지점입니다.

이런 상황은 대량의 로그 파일, 세션 파일, 메일 큐, 컨테이너의 writable layer, CI/CD 빌드 캐시처럼 **작은 파일을 많이 생성하는 환경**에서 자주 발생합니다.

## 9. 정리

```text
사용자가 /home/eunbyul/report.txt를 열려고 함
   ↓
커널이 경로를 "/", "home", "eunbyul", "report.txt"로 쪼갬
   ↓
각 디렉터리 엔트리에서 다음 inode 번호를 찾아 나감
   ↓
최종적으로 report.txt의 inode를 찾음
   ↓
inode 안의 권한/유형을 확인
   ↓
inode가 가리키는 데이터 블록에 접근
   ↓
실제 파일 내용을 프로세스에 전달
```

> inode는 파일이나 디렉터리의 유형, 권한, 소유자 UID/GID, 크기, 링크 수, 타임스탬프, 실제 데이터 위치 등의 메타데이터를 관리하는 파일시스템 객체다. 파일의 내용 그 자체가 아니라, 그 파일을 설명하고 찾아가기 위한 "설명서"에 해당한다.

**inode에 저장되지 않는 것**

```text
1. 파일 이름          → 부모 디렉터리의 디렉터리 엔트리가 관리
2. 전체 경로          → 디렉터리 엔트리를 순차적으로 탐색한 결과물
3. 사용자/그룹 이름    → UID/GID 숫자만 저장, 이름은 OS가 변환
4. 일반적인 파일 내용  → 대개 별도의 데이터 블록에 저장 (아주 작은 데이터는 예외)
5. 파일을 열고 있는 프로세스 → 커널의 파일 디스크립터/열린 파일 객체가 관리
```
