## 1. Data Silo란 무엇인가?

### 1-1) Data Silo의 기본 의미

Data Silo는 조직이나 시스템 안에서 데이터가 특정 부서, 특정 서비스, 특정 데이터베이스, 특정 애플리케이션에 고립되어 다른 영역과 쉽게 연결되거나 공유되지 못하는 상태를 말한다.

여기서 중요한 표현은 “데이터가 따로 있다”는 말이다. 단순히 데이터베이스가 여러 개 있다는 뜻이 아니라, 각각의 데이터가 서로 연결되지 않고 독립적으로 관리되어 전체 흐름을 파악하기 어렵다는 뜻이다.

예를 들어 쇼핑몰 회사에 다음과 같은 시스템이 있다고 가정할 수 있다.

```text
회원 시스템      → 회원 정보 저장
주문 시스템      → 주문 내역 저장
결제 시스템      → 결제 내역 저장
배송 시스템      → 배송 상태 저장
고객센터 시스템  → 문의 내역 저장
마케팅 시스템    → 광고 유입 데이터 저장
```

고객 한 명의 전체 행동을 이해하려면 회원 정보, 주문 정보, 결제 정보, 배송 정보, 문의 정보, 광고 유입 정보를 함께 봐야 한다. 그러나 이 데이터가 각각 다른 시스템에 흩어져 있고 서로 연결되어 있지 않다면, 고객의 전체 흐름을 한 번에 분석하기 어렵다. 이것이 Data Silo 문제다.

### 1-2) Silo라는 표현이 사용되는 이유

Silo는 원래 곡물이나 사료를 보관하는 독립된 저장탑을 의미한다. 여러 개의 저장탑이 각각 분리되어 있고 내부 내용물이 쉽게 섞이지 않는 구조다.

데이터도 이와 비슷하게 부서별, 시스템별, 서비스별로 분리되어 저장되면 서로 섞이거나 연결되지 않는다.

```text
영업팀 데이터 저장소
마케팅팀 데이터 저장소
재무팀 데이터 저장소
고객센터 데이터 저장소
운영팀 로그 저장소
```

각 저장소가 따로 존재하고, 데이터 구조와 관리 기준도 다르고, 접근 권한도 다르면 조직 전체에서 데이터를 통합적으로 활용하기 어렵다. 그래서 이런 상태를 Data Silo라고 부른다.

### 1-3) 단순한 데이터 분산과 Data Silo의 차이

모든 데이터가 여러 시스템에 나뉘어 있다고 해서 무조건 문제가 되는 것은 아니다. 실제 기업 시스템은 업무 목적에 따라 데이터베이스가 나뉘는 것이 일반적이다.

문제는 데이터가 나뉘어 있다는 사실 자체가 아니라, 나뉜 데이터를 연결하고 해석하고 활용할 수 있는 체계가 없다는 데 있다.

정상적인 분산 구조는 다음과 같다.

```text
각 시스템이 독립적으로 운영됨
공통 식별자 존재
데이터 연동 파이프라인 존재
통합 분석 저장소 존재
데이터 카탈로그 존재
권한 정책 존재
```

반면 Data Silo 상태는 다음과 같다.

```text
각 시스템이 독립적으로 운영됨
공통 식별자 없음
데이터 연동 없음
부서별 기준이 다름
데이터 위치를 찾기 어려움
권한 요청과 공유 절차가 복잡함
분석할 때마다 수작업으로 취합해야 함
```

즉, Data Silo의 핵심은 데이터의 물리적 분리보다 연결성, 표준화, 접근성, 활용성의 부족이다.

## 2. Data Silo가 발생하는 원인

### 2-1) 부서별로 시스템을 따로 도입함

기업이 성장하면서 각 부서는 자신의 업무에 맞는 시스템을 따로 도입한다.

예를 들어 영업팀은 CRM을 사용하고, 재무팀은 ERP를 사용하고, 고객센터는 상담 솔루션을 사용하고, 마케팅팀은 광고 분석 도구를 사용한다.

```text
영업팀      → CRM
재무팀      → ERP
고객센터    → VOC / Helpdesk 솔루션
마케팅팀    → 광고 플랫폼 / 캠페인 관리 도구
운영팀      → 모니터링 시스템 / 로그 시스템
```

각 시스템은 해당 부서 업무에는 적합하지만, 조직 전체의 데이터 통합 관점에서는 서로 분리된 데이터 저장소가 된다.

### 2-2) 서비스별 데이터베이스가 분리됨

현대 애플리케이션은 기능별로 데이터베이스를 분리하는 경우가 많다. 특히 마이크로서비스 아키텍처에서는 서비스별로 독립적인 데이터베이스를 갖는 구조가 일반적이다.

```text
회원 서비스 DB
상품 서비스 DB
주문 서비스 DB
결제 서비스 DB
배송 서비스 DB
알림 서비스 DB
```

이 구조는 서비스 독립성과 확장성에는 유리하지만, 분석 관점에서는 데이터가 여러 곳에 흩어지는 문제가 생길 수 있다.

예를 들어 “어떤 광고를 보고 들어온 사용자가 어떤 상품을 보고, 무엇을 구매했으며, 이후 문의를 남겼는가?”를 분석하려면 여러 DB를 연결해야 한다. 이 연결 구조가 없으면 Data Silo가 된다.

### 2-3) 데이터 표준이 없음

같은 의미의 데이터가 시스템마다 다르게 표현되면 데이터 통합이 어려워진다.

예를 들어 고객 식별자가 다음처럼 다르게 관리될 수 있다.

```text
회원 시스템: user_id
주문 시스템: customer_id
고객센터: member_no
마케팅 시스템: client_id
```

날짜 형식도 다를 수 있다.

```text
2026-06-15
2026/06/15
15-06-2026
20260615
```

상태값도 시스템마다 다를 수 있다.

```text
결제 완료: PAID, paid, 1, Y, COMPLETE
취소 완료: CANCEL, canceled, 0, N, REFUND
```

이처럼 컬럼명, 데이터 타입, 코드값, 날짜 형식, 집계 기준이 다르면 데이터를 한곳에 모아도 바로 분석하기 어렵다.

### 2-4) 데이터 소유권이 부서 중심으로 고정됨

Data Silo는 기술 문제만이 아니라 조직 문제이기도 하다. 많은 조직에서 데이터는 전사 자산이라기보다 특정 부서의 소유물처럼 취급된다.

예를 들어 다음과 같은 상황이 발생할 수 있다.

```text
영업팀: CRM 데이터는 우리 팀 데이터다.
재무팀: 매출 데이터는 외부 공유가 어렵다.
고객센터: 상담 내용은 민감해서 공유할 수 없다.
운영팀: 로그 데이터는 우리만 관리한다.
```

물론 보안상 제한이 필요한 데이터는 반드시 통제해야 한다. 하지만 기준 없이 부서 단위로 데이터가 닫혀 있으면 필요한 분석과 협업이 어려워진다.

### 2-5) 보안과 개인정보 이슈로 인해 과도하게 차단됨

개인정보, 결제 정보, 계약 정보, 내부 기밀 데이터는 엄격하게 보호되어야 한다. 하지만 보안 정책이 세밀하지 않으면 조직은 데이터를 안전하게 공유하기보다 아예 공유하지 않는 방식을 선택하기 쉽다.

이 경우 다음과 같은 문제가 생긴다.

```text
데이터 접근 요청 절차가 복잡함
승인 시간이 오래 걸림
권한 기준이 명확하지 않음
필요한 사람도 데이터를 보지 못함
가명처리나 마스킹 체계가 없음
```

결국 보안을 이유로 데이터 활용이 막히고, Data Silo가 강화된다.

### 2-6) 데이터 연동 파이프라인이 없음

여러 시스템의 데이터를 분석하려면 데이터를 주기적으로 수집하고 변환하고 저장하는 파이프라인이 필요하다.

하지만 파이프라인이 없으면 실무자는 매번 수작업으로 데이터를 받아서 엑셀로 합치거나, 시스템 담당자에게 직접 요청해야 한다.

```text
DB에서 CSV 추출
메일로 전달
엑셀에서 수작업 병합
컬럼명 수정
중복 제거
피벗 테이블 작성
보고서 작성
```

이 방식은 느리고 오류가 많으며 반복 작업이 많다.

## 3. Data Silo로 인해 발생하는 문제

### 3-1) 데이터 기반 의사결정이 느려짐

Data Silo가 있으면 의사결정에 필요한 데이터를 빠르게 확보하기 어렵다. 보고서를 만들기 위해 여러 부서에 데이터를 요청하고, 형식을 맞추고, 수작업으로 병합해야 한다.

예를 들어 “이번 달 마케팅 캠페인이 실제 매출에 얼마나 기여했는가?”를 분석하려면 다음 데이터가 필요하다.

```text
광고 노출 데이터
광고 클릭 데이터
회원가입 데이터
상품 조회 데이터
주문 데이터
결제 데이터
환불 데이터
재구매 데이터
```

이 데이터가 각각 다른 시스템에 있고 연결되지 않는다면 분석에 많은 시간이 걸린다.

### 3-2) 부서마다 보는 숫자가 달라짐

Data Silo가 있으면 같은 지표라도 부서마다 다르게 계산할 수 있다.

예를 들어 신규 고객 수를 계산할 때 기준이 다를 수 있다.

```text
마케팅팀: 광고를 통해 유입된 신규 가입자 수
영업팀: 첫 구매를 완료한 고객 수
재무팀: 실제 결제가 발생한 고객 수
서비스팀: 계정을 생성한 사용자 수
```

각 기준이 모두 나름의 의미는 있지만, 공통 정의가 없으면 회의에서 숫자 자체가 충돌한다.

```text
마케팅팀: 신규 고객 10,000명
영업팀: 신규 고객 7,500명
재무팀: 신규 고객 6,800명
```

이 경우 의사결정보다 “어떤 숫자가 맞는가?”를 따지는 데 시간이 소모된다.

### 3-3) 고객 경험을 전체적으로 이해하기 어려움

고객 데이터가 분리되어 있으면 고객 여정 전체를 볼 수 없다.

고객 여정은 보통 다음처럼 이어진다.

```text
광고 노출
검색
웹사이트 방문
회원가입
상품 조회
장바구니 추가
결제
배송
문의
환불 또는 재구매
```

Data Silo 상태에서는 각 단계가 서로 다른 시스템에 저장되기 때문에 고객이 어디에서 이탈했는지, 어떤 문제가 반복되는지, 어떤 고객이 충성 고객인지 파악하기 어렵다.

### 3-4) AI와 머신러닝 활용이 어려워짐

AI/ML 모델은 다양한 데이터를 연결해서 학습할수록 더 유용한 패턴을 찾을 수 있다. 하지만 데이터가 사일로화되어 있으면 학습 데이터셋을 만들기 어렵다.

예를 들어 이탈 예측 모델을 만들려면 다음 데이터가 필요할 수 있다.

```text
고객 프로필
로그인 이력
구매 이력
문의 이력
환불 이력
이벤트 참여 이력
서비스 사용 로그
```

이 데이터가 각 시스템에 고립되어 있으면 모델 학습용 Feature를 만들기 어렵다. 결국 AI 도입을 시도해도 데이터 준비 단계에서 막히게 된다.

### 3-5) 운영과 장애 대응이 비효율적임

인프라나 클라우드 운영에서도 Data Silo는 문제가 된다.

예를 들어 특정 고객의 VM 성능 저하 원인을 분석한다고 가정한다.

필요한 데이터는 다음과 같다.

```text
VM 생성 정보
Hypervisor 상태
스토리지 IOPS
네트워크 트래픽
보안그룹 정책
모니터링 메트릭
애플리케이션 로그
장애 티켓
고객 계약 정보
```

이 데이터가 Nova, Neutron, Cinder, Ceph, Prometheus, Grafana, Loki, 티켓 시스템, CRM 등에 흩어져 있으면 원인 분석이 느려진다.

### 3-6) 데이터 중복과 관리 비용이 증가함

부서마다 필요한 데이터를 별도로 복사하고 관리하면 중복 데이터가 많아진다.

```text
마케팅팀 고객 목록
영업팀 고객 목록
고객센터 고객 목록
재무팀 고객 목록
```

각 목록이 따로 관리되면 최신 상태가 다를 수 있고, 개인정보 변경이나 삭제 요청이 들어왔을 때 모든 시스템에 반영하기 어렵다.

## 4. Data Silo 해결의 핵심 방향

### 4-1) 핵심은 “모든 데이터를 무조건 한곳에 복사”가 아니다

Data Silo 해결이라고 하면 모든 데이터를 하나의 데이터베이스에 넣는 것으로 오해하기 쉽다. 하지만 실무적으로는 모든 데이터를 한곳에 강제로 몰아넣는 것이 항상 정답은 아니다.

중요한 것은 다음 네 가지다.

```text
데이터가 어디에 있는지 찾을 수 있어야 한다.
데이터의 의미와 기준을 이해할 수 있어야 한다.
필요한 데이터끼리 연결할 수 있어야 한다.
권한에 맞게 안전하게 활용할 수 있어야 한다.
```

따라서 Data Silo 해결은 저장소 통합, 데이터 표준화, 파이프라인 자동화, 거버넌스, 보안, 조직 문화 개선을 함께 다루는 문제다.

### 4-2) 기술적 해결과 조직적 해결이 함께 필요함

기술적으로는 Data Warehouse, Data Lake, Lakehouse, ETL/ELT, API, Kafka, 데이터 카탈로그 같은 도구를 사용할 수 있다.

하지만 도구만 도입한다고 해결되는 것은 아니다. 조직 차원에서 데이터 소유자, 접근 권한, 품질 기준, 표준 용어, 공통 지표 정의도 함께 정해야 한다.

Data Silo 해결의 네 가지 축은 다음과 같다.

```text
기술: 데이터 통합 저장소, 파이프라인, API, 이벤트 연동
표준: 공통 식별자, 스키마, 코드값, 지표 정의
거버넌스: 데이터 소유자, 품질 관리, 변경 관리, 카탈로그
보안: 권한 관리, 마스킹, 암호화, 감사 로그
```

## 5. 해결 방안 1: 데이터 현황 조사와 데이터 맵 작성

### 5-1) 가장 먼저 해야 할 일

Data Silo를 해결하려면 먼저 조직 안에 어떤 데이터가 어디에 있는지 파악해야 한다. 이를 데이터 인벤토리 또는 데이터 맵이라고 부를 수 있다.

조사해야 할 항목은 다음과 같다.

```text
시스템명
데이터베이스 종류
주요 테이블 또는 데이터셋
데이터 소유 부서
관리 담당자
데이터 생성 주기
데이터 갱신 주기
데이터 사용 목적
민감정보 포함 여부
접근 권한 정책
외부 연동 가능 여부
API 제공 여부
백업 및 보관 정책
```

### 5-2) 데이터 맵 예시

```text
CRM 시스템
- 소유 부서: 영업팀
- 주요 데이터: 고객 정보, 영업 기회, 계약 정보
- DB 유형: PostgreSQL
- 갱신 주기: 실시간
- 민감정보: 이름, 연락처, 이메일 포함
- 연동 방식: API 가능

결제 시스템
- 소유 부서: 재무팀
- 주요 데이터: 결제 내역, 환불 내역, 결제 수단
- DB 유형: MySQL
- 갱신 주기: 실시간
- 민감정보: 결제 관련 정보 포함
- 연동 방식: 제한적 DB Replica 가능

고객센터 시스템
- 소유 부서: CS팀
- 주요 데이터: 문의 티켓, 상담 내용, 처리 상태
- DB 유형: SaaS
- 갱신 주기: 실시간
- 민감정보: 상담 내용에 개인정보 포함 가능
- 연동 방식: REST API 가능
```

이 작업을 해야 어떤 데이터를 먼저 통합할지, 어떤 데이터는 보안 검토가 필요한지, 어떤 데이터는 API로 연결할 수 있는지 판단할 수 있다.

## 6. 해결 방안 2: 공통 식별자와 데이터 표준화

### 6-1) 공통 식별자가 중요한 이유

데이터를 연결하려면 공통 키가 필요하다. 고객 데이터라면 customer_id, 상품 데이터라면 product_id, 주문 데이터라면 order_id가 대표적인 공통 식별자다.

예를 들어 다음과 같이 연결할 수 있다.

```text
customer_id → 고객 정보와 주문 정보 연결
order_id    → 주문 정보와 결제 정보 연결
product_id  → 주문 정보와 상품 정보 연결
ticket_id   → 고객 문의 이력 연결
```

공통 식별자가 없으면 데이터 연결이 매우 어려워진다.

### 6-2) 공통 식별자가 없는 경우

현실에서는 시스템마다 ID가 다를 수 있다.

```text
CRM 고객 ID: C-1001
쇼핑몰 회원 ID: U-8891
고객센터 회원 번호: M-7721
광고 플랫폼 사용자 ID: A-5529
```

이 경우 같은 사람인지 매칭하기 위한 기준이 필요하다.

```text
이메일
전화번호
사업자등록번호
로그인 계정
외부 인증 ID
디바이스 ID
쿠키 ID
```

다만 이메일, 전화번호 같은 개인정보를 사용할 때는 보안과 개인정보 보호 기준을 반드시 적용해야 한다.

### 6-3) 데이터 표준화 대상

표준화해야 할 항목은 다음과 같다.

```text
테이블명
컬럼명
데이터 타입
날짜 형식
시간대 기준
상태 코드
상품 코드
지역 코드
고객 등급
집계 기준
지표 계산식
```

예를 들어 결제 상태 코드를 다음처럼 통일할 수 있다.

```text
PAID      → 결제 완료
CANCELED  → 결제 취소
REFUNDED  → 환불 완료
FAILED    → 결제 실패
PENDING   → 결제 대기
```

### 6-4) 표준 용어 사전 구축

부서마다 같은 단어를 다르게 쓰면 지표 충돌이 발생한다. 따라서 주요 용어를 표준화해야 한다.

예를 들어 “활성 사용자”를 다음처럼 정의할 수 있다.

```text
활성 사용자 = 특정 기간 동안 로그인 또는 핵심 행동을 1회 이상 수행한 사용자
```

“신규 고객”도 기준을 명확히 해야 한다.

```text
신규 가입자 = 해당 기간에 계정을 최초 생성한 사용자
신규 구매자 = 해당 기간에 첫 결제를 완료한 사용자
신규 유료 고객 = 해당 기간에 유료 상품을 최초 결제한 사용자
```

이렇게 정의하지 않으면 부서마다 다른 숫자를 사용하게 된다.

## 7. 해결 방안 3: Data Warehouse 구축

### 7-1) Data Warehouse의 개념

Data Warehouse는 여러 업무 시스템에서 데이터를 모아 분석하기 좋게 정리한 중앙 저장소다. 주로 정형 데이터 분석, 리포트, BI 대시보드에 사용된다.

일반적인 구조는 다음과 같다.

```text
업무 시스템 DB
CRM / ERP / 주문 / 결제 / 고객센터
        ↓
ETL 또는 ELT 파이프라인
        ↓
Data Warehouse
        ↓
BI / 리포트 / SQL 분석
```

### 7-2) Data Warehouse가 적합한 경우

Data Warehouse는 다음과 같은 경우에 적합하다.

```text
매출 분석
고객 분석
영업 실적 분석
재무 리포트
정형 데이터 기반 BI
경영 대시보드
공통 KPI 관리
```

### 7-3) Data Warehouse의 장점

Data Warehouse를 구축하면 부서별로 흩어진 정형 데이터를 한곳에서 분석할 수 있다.

장점은 다음과 같다.

```text
SQL 분석이 쉬움
BI 도구와 연동이 쉬움
공통 지표 관리에 유리함
정제된 데이터 제공 가능
보고서 자동화에 적합함
부서별 Data Mart 구성 가능
```

### 7-4) 대표 기술

```text
Amazon Redshift
Google BigQuery
Snowflake
Azure Synapse Analytics
Oracle Exadata
Teradata
PostgreSQL 기반 분석 DB
ClickHouse
```

온프레미스나 프라이빗 클라우드 환경에서는 PostgreSQL, Greenplum, ClickHouse, Trino, Hive 등을 조합할 수 있다.

## 8. 해결 방안 4: Data Lake 구축

### 8-1) Data Lake의 개념

Data Lake는 정형, 반정형, 비정형 데이터를 원본에 가깝게 저장하는 대규모 데이터 저장소다.

저장 가능한 데이터는 다음과 같다.

```text
CSV
JSON
Parquet
Avro
로그 파일
이미지
음성
문서
웹 클릭 로그
IoT 센서 데이터
서버 로그
애플리케이션 이벤트
```

Data Warehouse가 정리된 분석용 데이터에 강하다면, Data Lake는 다양한 원천 데이터를 대량으로 보관하고 AI/ML, 로그 분석, 장기 보관에 활용하기 좋다.

### 8-2) Data Lake가 필요한 경우

다음과 같은 경우 Data Lake가 유용하다.

```text
대규모 로그 데이터를 저장해야 함
AI/ML 학습 데이터를 모아야 함
원본 데이터를 장기 보관해야 함
정형 데이터뿐 아니라 이미지, 문서, 음성도 저장해야 함
여러 시스템의 Raw Data를 먼저 모아두고 싶음
```

### 8-3) Data Lake의 장점

```text
다양한 형식의 데이터 저장 가능
원본 데이터 보존 가능
대용량 저장 비용 절감 가능
AI/ML 학습 데이터 구축에 유리
Spark, Flink 등 분산 처리와 연동 가능
```

### 8-4) Data Lake 구축 시 주의점

Data Lake는 잘 관리하지 않으면 Data Swamp가 될 수 있다. Data Swamp는 데이터는 많이 쌓여 있지만 무엇이 어디에 있는지 모르고 품질도 보장되지 않는 상태를 말한다.

따라서 Data Lake에는 다음 관리 체계가 필요하다.

```text
데이터 카탈로그
메타데이터 관리
스키마 관리
데이터 품질 검증
접근 권한 관리
저장 경로 표준화
수명 주기 정책
민감정보 분류
```

### 8-5) 대표 기술

```text
Amazon S3
Azure Data Lake Storage
Google Cloud Storage
HDFS
MinIO
Ceph Object Gateway
```

온프레미스 환경에서는 MinIO나 Ceph Object Gateway를 이용해 S3 호환 Object Storage 기반 Data Lake를 구성할 수 있다.

## 9. 해결 방안 5: Lakehouse 아키텍처 도입

### 9-1) Lakehouse의 개념

Lakehouse는 Data Lake와 Data Warehouse의 장점을 결합한 아키텍처다.

Data Lake는 다양한 원천 데이터를 저렴하게 저장하는 데 강하고, Data Warehouse는 정제된 데이터를 빠르게 분석하는 데 강하다. Lakehouse는 Data Lake 위에서 테이블 관리, 트랜잭션, 스키마 관리, SQL 분석을 가능하게 하여 두 구조의 장점을 함께 사용하려는 방식이다.

### 9-2) Lakehouse가 해결하려는 문제

기존 Data Lake는 원본 데이터 저장에는 좋지만 데이터 품질, 트랜잭션, 스키마 변경 관리, BI 분석 측면에서 약점이 있었다.

Lakehouse는 다음 기능을 제공한다.

```text
ACID 트랜잭션
스키마 진화
테이블 버전 관리
Time Travel
SQL 분석 지원
대용량 데이터 처리
BI와 ML 동시 활용
```

### 9-3) 대표 기술

```text
Delta Lake
Apache Iceberg
Apache Hudi
Databricks Lakehouse
Apache Spark
Trino
Presto
```

### 9-4) 적합한 사용 사례

```text
로그 데이터와 업무 데이터를 함께 분석해야 하는 경우
BI와 AI/ML을 같은 데이터 기반에서 운영해야 하는 경우
Data Lake를 이미 운영 중이지만 품질과 테이블 관리가 필요한 경우
클라우드와 온프레미스 모두에서 유연한 데이터 플랫폼이 필요한 경우
```

## 10. 해결 방안 6: ETL/ELT 파이프라인 구축

### 10-1) ETL의 개념

ETL은 Extract, Transform, Load의 약자다.

```text
Extract   → 원천 시스템에서 데이터 추출
Transform → 분석에 맞게 변환
Load      → 목적 저장소에 적재
```

예를 들어 MySQL의 주문 데이터를 추출해서 날짜 형식을 통일하고, 결제 상태 코드를 정리한 뒤 Data Warehouse에 적재하는 방식이다.

### 10-2) ELT의 개념

ELT는 Extract, Load, Transform의 약자다.

```text
Extract   → 원천 시스템에서 데이터 추출
Load      → 먼저 저장소에 적재
Transform → 저장소 안에서 분석용으로 변환
```

최근에는 클라우드 DW, Spark, Lakehouse 성능이 좋아지면서 원본 데이터를 먼저 저장하고 나중에 변환하는 ELT 방식도 많이 사용된다.

### 10-3) 파이프라인이 필요한 이유

Data Silo를 해결하려면 데이터 수집과 통합이 반복 가능하고 자동화되어야 한다. 사람이 매번 CSV를 다운로드하고 엑셀로 합치는 방식은 확장성이 없다.

파이프라인을 구축하면 다음 작업을 자동화할 수 있다.

```text
데이터 추출
데이터 정제
데이터 검증
데이터 적재
실패 재시도
스케줄링
로그 기록
알림 발송
```

### 10-4) 파이프라인 도구

```text
Apache Airflow
Dagster
Prefect
Apache NiFi
dbt
Spark
Flink
Kafka Connect
AWS Glue
Azure Data Factory
Google Cloud Dataflow
```

### 10-5) 예시 구조

```text
MySQL / PostgreSQL / Oracle / API / 로그
        ↓
Airflow / NiFi / Kafka Connect
        ↓
S3 / MinIO / HDFS / Data Warehouse
        ↓
Spark / dbt / SQL Transform
        ↓
Data Mart / BI / ML Feature Table
```

## 11. 해결 방안 7: API 기반 데이터 연동

### 11-1) API 연동의 개념

API 기반 연동은 데이터를 직접 복사하지 않고 시스템 간에 필요한 데이터를 요청하고 응답받는 방식이다.

예를 들어 주문 시스템이 고객 정보를 직접 복사해서 보관하지 않고, 필요할 때 회원 시스템 API를 호출할 수 있다.

```text
주문 시스템 → 회원 API 호출 → 고객 정보 조회
```

### 11-2) API 연동이 적합한 경우

API 연동은 다음 상황에 적합하다.

```text
실시간 조회가 필요한 경우
데이터 원본을 한 시스템에서만 관리하고 싶은 경우
서비스 간 기능 연동이 필요한 경우
대량 분석보다 업무 처리 목적이 큰 경우
```

### 11-3) API 연동의 한계

API는 실시간 업무 처리에는 유리하지만 대규모 분석에는 한계가 있다.

예를 들어 1년치 고객 행동 데이터를 분석하기 위해 API를 수백만 번 호출하는 것은 비효율적이다. 이런 경우에는 Data Warehouse나 Data Lake에 데이터를 적재한 뒤 분석하는 것이 더 적합하다.

### 11-4) API 관리 시 필요한 요소

```text
API 인증과 권한 관리
Rate Limit
응답 스키마 표준화
API Gateway
버전 관리
장애 대응
호출 로그
감사 로그
```

## 12. 해결 방안 8: 이벤트 기반 아키텍처 도입

### 12-1) 이벤트 기반 아키텍처의 개념

이벤트 기반 아키텍처는 시스템에서 발생한 사건을 이벤트로 발행하고, 필요한 시스템이 이를 구독해서 처리하는 구조다.

예를 들어 사용자가 회원가입을 하면 다음과 같은 이벤트가 발생할 수 있다.

```text
UserSignedUp
```

상품을 주문하면 다음 이벤트가 발생할 수 있다.

```text
OrderCreated
PaymentCompleted
DeliveryStarted
```

### 12-2) 이벤트 기반 구조 예시

```text
회원 서비스
주문 서비스
결제 서비스
배송 서비스
        ↓
Kafka / RabbitMQ / Pulsar
        ↓
Data Lake
알림 시스템
추천 시스템
분석 시스템
모니터링 시스템
```

### 12-3) Data Silo 해결에 도움이 되는 이유

서비스별 DB가 분리되어 있어도 각 서비스가 이벤트를 발행하면 중앙에서 전체 흐름을 수집할 수 있다.

예를 들어 고객 여정을 이벤트로 연결할 수 있다.

```text
PageViewed
ProductClicked
CartAdded
OrderCreated
PaymentCompleted
TicketCreated
RefundRequested
```

이 이벤트를 수집하면 고객 행동 분석, 실시간 대시보드, 추천 모델, 이상 탐지에 활용할 수 있다.

### 12-4) 대표 기술

```text
Apache Kafka
RabbitMQ
Apache Pulsar
Amazon Kinesis
Google Pub/Sub
Azure Event Hubs
```

## 13. 해결 방안 9: Master Data Management 도입

### 13-1) MDM의 개념

Master Data Management, 즉 MDM은 조직의 핵심 기준 데이터를 일관되게 관리하는 체계다.

대표적인 Master Data는 다음과 같다.

```text
고객
상품
거래처
조직
직원
계정
지역
장비
서비스
계약
```

### 13-2) MDM이 필요한 이유

여러 시스템에서 같은 고객을 서로 다른 ID로 관리할 수 있다.

```text
CRM 고객 ID: C001
쇼핑몰 회원 ID: U100
고객센터 ID: CS777
계약 시스템 ID: CT900
```

MDM은 이 데이터들이 같은 고객을 의미하는지 매칭하고, 전사 기준이 되는 고객 정보를 만든다.

```text
Global Customer ID: G000001
```

이렇게 하면 여러 시스템의 데이터를 하나의 고객 기준으로 연결할 수 있다.

### 13-3) MDM의 주요 기능

```text
중복 데이터 식별
동일 고객 매칭
기준 데이터 정제
표준 코드 관리
Golden Record 생성
데이터 변경 이력 관리
시스템 간 기준 데이터 동기화
```

### 13-4) Golden Record

Golden Record는 여러 시스템에 흩어진 동일 엔티티 정보를 통합해 만든 신뢰 가능한 대표 레코드다.

예를 들어 고객 전화번호가 시스템마다 다르게 저장되어 있을 수 있다.

```text
CRM: 010-1111-2222
고객센터: 01011112222
계약 시스템: 010-1111-2222
```

MDM은 정제 규칙과 신뢰 우선순위에 따라 대표 값을 결정한다.

```text
표준 전화번호: 010-1111-2222
```

## 14. 해결 방안 10: 데이터 카탈로그 구축

### 14-1) 데이터 카탈로그의 개념

데이터 카탈로그는 조직 내 데이터가 어디에 있고, 어떤 의미이며, 누가 관리하고, 어떻게 사용할 수 있는지 알려주는 데이터 검색 및 메타데이터 관리 시스템이다.

쉽게 말하면 데이터의 지도다.

### 14-2) 데이터 카탈로그에 포함되는 정보

```text
데이터셋 이름
데이터 위치
시스템 출처
스키마
컬럼 설명
데이터 타입
데이터 소유자
관리 담당자
갱신 주기
품질 상태
접근 권한
민감정보 여부
관련 지표
사용 예시
Lineage
```

### 14-3) Data Lineage

Data Lineage는 데이터가 어디에서 생성되어 어떤 과정을 거쳐 현재 테이블이나 리포트에 도달했는지 보여주는 흐름 정보다.

예를 들어 매출 대시보드의 매출 값이 어디에서 왔는지 추적할 수 있다.

```text
결제 DB payments 테이블
        ↓
Airflow ETL
        ↓
DW fact_payment 테이블
        ↓
dbt mart_sales_monthly 모델
        ↓
BI 매출 대시보드
```

Lineage가 있으면 지표 오류가 발생했을 때 원인을 추적하기 쉽다.

### 14-4) 대표 도구

```text
OpenMetadata
DataHub
Apache Atlas
Amundsen
Collibra
Alation
```

오픈소스 기반으로는 OpenMetadata, DataHub, Apache Atlas가 많이 거론된다.

## 15. 해결 방안 11: 데이터 거버넌스 수립

### 15-1) 데이터 거버넌스의 개념

데이터 거버넌스는 조직의 데이터를 정확하고 안전하고 일관되게 관리하기 위한 정책, 역할, 프로세스의 집합이다.

Data Silo를 해결하려면 데이터가 단순히 모이는 것만으로는 부족하다. 누가 책임지고, 어떤 기준으로 관리하고, 누가 접근할 수 있고, 문제가 생기면 어떻게 처리할지 정해야 한다.

### 15-2) 데이터 거버넌스의 주요 요소

```text
데이터 소유자 지정
데이터 관리자 지정
데이터 품질 기준 정의
데이터 접근 권한 정책
개인정보 처리 기준
표준 용어 사전
공통 지표 정의
데이터 변경 관리
데이터 보존 기간 정책
감사 로그 관리
데이터 활용 승인 프로세스
```

### 15-3) 데이터 소유자와 데이터 관리자

Data Owner는 데이터에 대한 업무적 책임을 갖는 사람 또는 부서다. 예를 들어 고객 데이터의 Data Owner는 영업팀 또는 고객관리 부서일 수 있다.

Data Steward는 데이터 품질, 표준, 메타데이터를 관리하는 실무 담당자다.

```text
Data Owner   → 데이터의 업무적 책임
Data Steward → 데이터 품질과 표준 관리
Data Engineer → 데이터 파이프라인과 플랫폼 구현
Data Analyst → 데이터 분석과 지표 활용
Security Officer → 보안과 접근 통제 관리
```

### 15-4) 거버넌스가 없는 경우의 문제

```text
누가 데이터 정의를 결정하는지 모름
테이블 변경 시 영향 범위를 모름
지표 계산식이 부서마다 다름
민감정보 관리 기준이 불명확함
품질 문제가 발생해도 책임 소재가 불분명함
```

따라서 Data Silo 해결에는 거버넌스가 필수다.

## 16. 해결 방안 12: 데이터 보안과 권한 관리

### 16-1) Data Silo 해결과 보안은 함께 가야 함

Data Silo를 해결한다고 해서 모든 데이터를 모든 사람에게 공개하면 안 된다. 데이터 통합의 목적은 무분별한 공개가 아니라 권한에 맞는 안전한 공유다.

### 16-2) 필요한 보안 기능

```text
RBAC
ABAC
컬럼 단위 권한
행 단위 권한
데이터 마스킹
암호화
가명처리
익명처리
접근 로그
감사 로그
권한 승인 프로세스
```

### 16-3) RBAC와 ABAC

RBAC는 Role-Based Access Control의 약자로 역할 기반 접근 제어다.

```text
마케팅팀 분석가 → 고객 세그먼트 데이터 조회 가능
재무팀 담당자 → 결제 금액 데이터 조회 가능
운영팀 엔지니어 → 시스템 로그 조회 가능
```

ABAC는 Attribute-Based Access Control의 약자로 속성 기반 접근 제어다. 사용자 부서, 직급, 데이터 민감도, 접속 위치, 업무 목적 등을 기준으로 접근을 제어한다.

```text
사용자 부서 = 재무팀
데이터 등급 = 민감
접근 목적 = 월간 정산
접속 위치 = 사내망
```

조건을 만족할 때만 접근을 허용할 수 있다.

### 16-4) 데이터 마스킹 예시

```text
원본 이메일: user@example.com
마스킹 이메일: u***@example.com

원본 전화번호: 010-1234-5678
마스킹 전화번호: 010-****-5678
```

분석에는 전체 개인정보가 필요하지 않은 경우가 많다. 따라서 민감정보는 마스킹하거나 가명처리한 뒤 제공하는 것이 좋다.

## 17. 해결 방안 13: 데이터 품질 관리

### 17-1) 데이터 품질이 중요한 이유

Data Silo를 해결해서 데이터를 한곳에 모아도 데이터 품질이 낮으면 분석 결과를 신뢰할 수 없다.

예를 들어 다음 문제가 발생할 수 있다.

```text
고객 ID 누락
주문 금액 음수
결제 완료인데 결제 시간이 없음
중복 주문 발생
날짜 형식 오류
상품 코드 불일치
참조 무결성 깨짐
```

### 17-2) 품질 검증 항목

```text
NULL 비율
중복 데이터
데이터 타입 오류
값의 범위 오류
날짜 범위 오류
참조 무결성
스키마 변경
집계값 차이
이상치
정합성 검증
```

### 17-3) 품질 검증 예시

```text
order_id는 NULL이면 안 된다.
payment_amount는 0보다 크거나 같아야 한다.
order_status는 PAID, CANCELED, FAILED, PENDING 중 하나여야 한다.
created_at은 현재 시각보다 미래일 수 없다.
customer_id는 customer 테이블에 존재해야 한다.
```

### 17-4) 대표 도구

```text
Great Expectations
Soda
dbt tests
Amazon Deequ
Apache Griffin
```

데이터 품질 검증은 파이프라인 안에 포함되어야 한다. 검증 실패 시 적재를 중단하거나 알림을 보내야 한다.

## 18. 해결 방안 14: 공통 지표와 Semantic Layer 구축

### 18-1) 공통 지표 정의의 필요성

Data Silo의 대표적인 문제는 부서마다 지표 기준이 다르다는 것이다. 이를 해결하려면 공통 지표 정의가 필요하다.

예를 들어 매출을 정의할 때도 여러 기준이 존재한다.

```text
총매출 = 전체 결제 금액
순매출 = 결제 금액 - 환불 금액 - 할인 금액
인식 매출 = 회계 기준에 따라 인식된 매출
```

각 지표를 명확하게 정의해야 회의와 보고에서 혼란이 줄어든다.

### 18-2) Semantic Layer의 개념

Semantic Layer는 데이터베이스의 복잡한 테이블 구조를 비즈니스 사용자가 이해하기 쉬운 지표와 용어로 추상화하는 계층이다.

예를 들어 데이터베이스에는 다음과 같은 테이블이 있을 수 있다.

```text
fact_order
fact_payment
dim_customer
dim_product
dim_campaign
```

Semantic Layer에서는 이를 다음처럼 비즈니스 용어로 제공한다.

```text
신규 고객 수
월간 활성 사용자
순매출
재구매율
고객 생애 가치
캠페인 전환율
```

### 18-3) 기대 효과

```text
부서별 지표 불일치 감소
BI 대시보드 기준 통일
분석 속도 향상
비즈니스 사용자 이해도 향상
데이터 신뢰도 향상
```

## 19. 해결 방안 15: BI와 셀프서비스 분석 환경 제공

### 19-1) 데이터 활용 환경이 필요한 이유

데이터를 통합해도 사용자가 쉽게 활용하지 못하면 효과가 제한적이다. 따라서 BI 도구나 셀프서비스 분석 환경을 제공해야 한다.

### 19-2) BI 도구의 역할

BI 도구는 통합된 데이터를 기반으로 대시보드, 리포트, 시각화를 제공한다.

```text
매출 대시보드
고객 분석 대시보드
서비스 사용량 대시보드
운영 장애 대시보드
마케팅 성과 대시보드
재무 리포트
```

### 19-3) 대표 도구

```text
Tableau
Power BI
Looker
Apache Superset
Metabase
Grafana
```

운영 메트릭 중심이면 Grafana가 적합하고, 비즈니스 분석 중심이면 Tableau, Power BI, Superset, Metabase 등을 사용할 수 있다.

### 19-4) 셀프서비스 분석의 장점

```text
데이터 요청 대기 시간 감소
분석가 업무 부담 감소
현업 부서의 데이터 활용 증가
반복 보고서 자동화
의사결정 속도 향상
```

## 20. 인프라 및 클라우드 환경에서의 Data Silo 예시

### 20-1) 프라이빗 클라우드 환경의 데이터 분산

OpenStack, CloudStack, Kubernetes 같은 인프라 환경에서도 Data Silo가 발생할 수 있다.

예를 들어 OpenStack 기반 프라이빗 클라우드에는 다음 데이터가 존재한다.

```text
Nova DB      → VM 인스턴스 정보
Neutron DB   → 네트워크, 포트, 라우터, 보안그룹 정보
Cinder DB    → 볼륨 정보
Glance DB    → 이미지 정보
Keystone DB  → 사용자, 프로젝트, 권한 정보
Ceilometer   → 사용량 데이터
Prometheus   → 메트릭
Loki/ELK      → 로그
Ceph          → 스토리지 상태
티켓 시스템  → 장애 및 요청 이력
CRM/계약 시스템 → 고객 계약 정보
```

이 데이터가 연결되지 않으면 운영자가 장애 원인을 분석할 때 여러 시스템을 직접 확인해야 한다.

### 20-2) 예시: VM 성능 저하 분석

고객이 “VM이 느리다”고 문의했다고 가정한다.

필요한 데이터는 다음과 같다.

```text
VM이 어느 Compute Node에서 실행 중인지
해당 Node의 CPU, Memory 사용률은 어떤지
VM의 Disk I/O는 어떤지
Cinder Volume과 Backend Storage 상태는 어떤지
네트워크 패킷 손실이나 지연은 없는지
보안그룹이나 라우팅 문제는 없는지
최근 장애 티켓이 있었는지
동일 고객의 다른 VM도 문제가 있는지
```

이 정보가 각각 다른 시스템에 있으면 장애 분석이 오래 걸린다.

### 20-3) 인프라 환경에서의 해결 방향

```text
모니터링 메트릭 통합
로그 중앙화
이벤트 수집
CMDB 구축
사용량 데이터 통합
고객/계약 정보와 리소스 정보 연결
대시보드 제공
알림과 티켓 시스템 연동
```

구성 예시는 다음과 같다.

```text
OpenStack / Kubernetes / Ceph / Network 장비
        ↓
Exporter / Log Collector / Event Collector
        ↓
Prometheus / Loki / Elasticsearch / Kafka
        ↓
Data Lake / Data Warehouse
        ↓
Grafana / Superset / Alertmanager / Ticket System
```

## 21. 온프레미스 또는 프라이빗 클라우드 기반 구축 예시

온프레미스 환경에서는 다음과 같은 구성으로 Data Silo 해결 플랫폼을 만들 수 있다.

```text
Data Catalog: OpenMetadata 또는 DataHub

Monitoring: Prometheus, Grafana
```

```text
[Source Systems]

        ↓

        ↓

        ↓

```

### 21-3) 단계적 도입이 중요한 이유

처음부터 모든 데이터를 통합하려고 하면 범위가 너무 커지고 실패 가능성이 높다. 우선순위를 정해 작은 성공 사례를 만든 뒤 확장하는 것이 현실적이다.

예를 들어 다음 순서로 진행할 수 있다.

```text
1단계: 핵심 데이터 현황 조사
2단계: 고객/주문/결제 데이터 우선 통합
3단계: 공통 식별자와 지표 정의
4단계: Data Warehouse 또는 Data Lake 구축
5단계: BI 대시보드 제공
6단계: 로그/이벤트 데이터 확장
7단계: 데이터 카탈로그와 거버넌스 강화
8단계: AI/ML 활용으로 확장
```

## 22. Data Silo 해결을 위한 실무 추진 절차

### 22-1) 1단계: 문제 정의

먼저 어떤 Data Silo 문제가 실제 업무에 영향을 주는지 정의해야 한다.

예시는 다음과 같다.

```text
고객 데이터가 여러 시스템에 흩어져 고객별 통합 분석이 어렵다.
부서별 매출 지표가 달라 회의 때마다 숫자가 충돌한다.
장애 분석 시 로그, 메트릭, 티켓 데이터를 따로 확인해야 한다.
AI 모델 학습용 데이터를 만들기 위해 수작업이 너무 많다.
보고서 작성에 매주 많은 시간이 소요된다.
```

### 22-2) 2단계: 데이터 자산 조사

조직 내 데이터 자산을 조사하고 데이터 맵을 만든다.

```text
어떤 시스템에 어떤 데이터가 있는가?
누가 관리하는가?
어떤 형식인가?
얼마나 자주 갱신되는가?
민감정보가 포함되어 있는가?
연동 방식은 무엇인가?
```

### 22-3) 3단계: 우선순위 선정

모든 데이터를 한 번에 통합하지 않고, 업무 효과가 큰 데이터부터 시작한다.

우선순위가 높은 데이터 예시는 다음과 같다.

```text
고객 데이터
주문 데이터
결제 데이터
상품 데이터
서비스 사용 로그
장애 티켓 데이터
운영 메트릭
```

### 22-4) 4단계: 표준 정의

데이터 통합 전에 표준을 정의해야 한다.

```text
공통 ID
컬럼명 규칙
데이터 타입
날짜와 시간대 기준
코드값
지표 계산식
민감정보 분류 기준
```

### 22-5) 5단계: 통합 저장소 설계

목적에 맞게 저장소를 선택한다.

```text
정형 리포트와 BI 중심 → Data Warehouse
로그, 파일, AI 학습 데이터 중심 → Data Lake
BI와 AI/ML을 함께 고려 → Lakehouse
실시간 이벤트 분석 중심 → Kafka + Stream Processing
```

### 22-6) 6단계: 파이프라인 구축

수동 취합을 없애고 자동화된 수집, 변환, 적재 구조를 만든다.

```text
Batch ETL
CDC
API 수집
Log 수집
Event Streaming
dbt Transform
Spark Processing
```

### 22-7) 7단계: 품질 검증과 모니터링

데이터 파이프라인은 장애가 발생할 수 있으므로 모니터링이 필요하다.

확인해야 할 항목은 다음과 같다.

```text
파이프라인 성공/실패 여부
데이터 적재 건수
스키마 변경 여부
NULL 비율
중복 데이터
처리 지연 시간
원천 데이터와 적재 데이터의 건수 차이
```

### 22-8) 8단계: 권한과 보안 적용

데이터를 통합한 뒤에는 접근 권한을 세분화해야 한다.

```text

직무별 권한
컬럼 단위 권한
민감정보 마스킹
접근 로그
승인 프로세스
```

### 22-9) 9단계: 데이터 카탈로그와 문서화

사용자가 데이터를 찾고 이해할 수 있도록 카탈로그와 문서를 제공한다.

```text

컬럼 설명

소유자

품질 상태
```

### 22-10) 10단계: 활용 확산

최종적으로 데이터가 실제 업무에 쓰이도록 대시보드, 리포트, API, AI/ML 학습 데이터셋 등을 제공한다.


## 24. 기대 효과

### 24-1) 의사결정 속도 향상

데이터가 통합되고 지표가 표준화되면 보고서 작성 시간이 줄고, 필요한 데이터를 빠르게 확인할 수 있다.

### 24-2) 데이터 신뢰도 향상

공통 지표와 품질 검증 체계가 있으면 부서마다 다른 숫자를 사용하는 문제가 줄어든다.

### 24-3) 고객 이해도 향상

고객의 가입, 구매, 결제, 문의, 재구매 흐름을 연결해서 볼 수 있으므로 고객 경험 분석이 가능해진다.

### 24-4) AI/ML 활용 기반 마련

여러 시스템의 데이터를 연결해 모델 학습용 Feature를 만들 수 있으므로 추천, 이탈 예측, 이상 탐지, 수요 예측 등에 활용할 수 있다.

### 24-5) 운영 효율 향상

인프라 환경에서는 로그, 메트릭, 이벤트, 티켓 데이터를 통합해 장애 대응 시간을 줄일 수 있다.

### 24-6) 보안과 데이터 활용의 균형 확보

권한 관리, 마스킹, 감사 로그를 적용하면 데이터를 안전하게 공유하면서도 활용성을 높일 수 있다.

## 25. 최종 정리

Data Silo는 데이터가 여러 시스템에 나뉘어 있다는 단순한 상태를 의미하지 않는다. 핵심은 데이터가 서로 연결되지 않고, 기준이 다르고, 접근이 어렵고, 조직적으로 공유되지 않아 전체 관점의 분석과 활용이 어려운 상태다.

Data Silo 문제를 해결하려면 기술적 통합만으로는 부족하다. 데이터 저장소, 파이프라인, API, 이벤트 연동 같은 기술적 접근과 함께 공통 식별자, 표준 용어, 공통 지표, 데이터 품질, 거버넌스, 보안 정책이 함께 마련되어야 한다.

핵심 해결 방향은 다음과 같이 정리할 수 있다.

```text
1. 데이터가 어디에 있는지 파악한다.
2. 중요한 데이터부터 우선 통합한다.
3. 공통 식별자와 데이터 표준을 정의한다.
4. Data Warehouse, Data Lake, Lakehouse 등 목적에 맞는 저장소를 구축한다.
5. ETL/ELT, API, 이벤트 기반 파이프라인으로 데이터를 자동 연동한다.
6. 데이터 카탈로그와 Lineage로 데이터를 찾고 추적할 수 있게 한다.
7. 데이터 거버넌스와 품질 관리 체계를 만든다.
8. 보안과 권한 관리를 적용해 안전하게 공유한다.
9. BI, 리포트, AI/ML, 운영 대시보드로 실제 활용을 확산한다.
```

결국 Data Silo 해결의 목표는 모든 데이터를 무조건 하나의 저장소에 몰아넣는 것이 아니다. 조직 전체가 데이터를 신뢰할 수 있고, 필요한 데이터를 찾을 수 있으며, 서로 연결해서 분석할 수 있고, 권한에 맞게 안전하게 활용할 수 있는 구조를 만드는 것이다.
