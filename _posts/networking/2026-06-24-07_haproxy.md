## 1. 이 글에서 다루는 범위

앞선 글에서 네트워크 기본기, HTTP/HTTPS, Kubernetes Network, Service, Ingress, Load Balancer를 정리했다면, 이 글에서는 그 지식을 바탕으로 HAProxy가 실제 운영 환경에서 어떻게 동작하고, 장애가 발생했을 때 어디를 확인해야 하는지까지 정리합니다.

HAProxy는 다음 역할을 수행할 수 있습니다.

- L4 TCP Proxy
- L7 HTTP Proxy
- Load Balancer
- Reverse Proxy
- TLS Termination
- SSL Passthrough
- SNI 기반 라우팅
- HTTP Host/Path/Header 기반 라우팅
- Health Check
- Sticky Session
- Rate Limiting
- Client IP 전달
- PROXY Protocol 전달
- Kubernetes Ingress 앞단 Load Balancer
- 온프레미스 환경의 VIP 진입점
- 서비스 장애 분석 지점

실무에서 HAProxy는 다음 서비스 앞단에 자주 위치합니다.

```text
Client
  ↓
HAProxy
  ↓
Kubernetes Ingress Controller
  ↓
Service
  ↓
Pod
```

또는 Kubernetes 없이 직접 Backend 서버로 보낼 수도 있습니다.

```text
Client
  ↓
HAProxy
  ├─ Harbor
  ├─ Keycloak
  ├─ Grafana
  ├─ ArgoCD
  └─ Ceph Dashboard
```

## 2. HAProxy란 무엇인가

HAProxy는 High Availability Proxy의 약자입니다. 이름 그대로 고가용성을 위한 Proxy입니다. HAProxy는 클라이언트와 서버 사이에 위치해서 요청을 받아 적절한 Backend로 전달합니다.

```text
Client
  ↓
HAProxy
  ↓
Backend Server
```

가장 기본적인 역할은 다음입니다.

```text
Client 요청을 받는다.
요청 조건을 확인한다.
정상 Backend 서버를 선택한다.
Backend로 요청을 전달한다.
Backend 응답을 Client에게 반환한다.
```

즉 HAProxy는 네트워크 경로 중간에서 트래픽을 제어하는 구성요소입니다.

### 2-1) HAProxy가 필요한 이유

서버가 한 대뿐이면 사용자는 그 서버로 직접 접속하면 됩니다.

```text
Client
  ↓
Server A
```

하지만 운영 환경에서는 서버 한 대만으로 안정적인 서비스를 제공하기 어렵습니다.

문제는 다음과 같습니다.

- 서버 한 대 장애 시 서비스 전체 장애
- 트래픽 증가 시 처리량 부족
- 배포 중 서비스 중단
- TLS 인증서 관리 분산
- 여러 도메인/서비스 라우팅 어려움
- Backend 서버 주소 노출
- Health Check와 장애 서버 제외 필요
- Client IP 전달과 로그 추적 필요

HAProxy를 두면 구조가 다음처럼 바뀝니다.

```text
Client
  ↓
HAProxy
  ├─ Server A
  ├─ Server B
  └─ Server C
```

이제 사용자는 HAProxy만 바라보고, HAProxy는 뒤쪽 서버 상태를 확인하면서 요청을 분산합니다.

### 2-2) HAProxy가 잘 맞는 영역

| 사용 영역 | 설명 |
|---|---|
| 웹 서비스 L7 라우팅 | Host, Path, Header 기반 분기 |
| TCP 서비스 로드밸런싱 | PostgreSQL, Redis, Kafka, Kubernetes API Server 등 |
| TLS Termination | 인증서를 HAProxy에서 처리 |
| SSL Passthrough | TLS를 해제하지 않고 Backend로 전달 |
| Kubernetes 앞단 LB | Ingress Controller 또는 NodePort 앞단 |
| 온프레미스 VIP 구성 | Keepalived와 함께 Active-Standby 구성 |
| 장애 서버 자동 제외 | Health Check 기반 |
| Rate Limit | Stick Table 기반 요청 제한 |
| 원본 IP 전달 | X-Forwarded-For, PROXY Protocol |

## 3. HAProxy Architecture

### 3-1) 기본 요청 흐름

HAProxy의 기본 요청 흐름은 다음과 같습니다.

```text
Client
  ↓
Frontend
  ↓
ACL 평가
  ↓
Backend 선택
  ↓
Load Balancing Algorithm
  ↓
Server 선택
  ↓
Backend Server
```

각 구성요소의 역할은 다음과 같습니다.

| 구성요소 | 역할 |
|---|---|
| global | HAProxy 프로세스 전체 설정 |
| defaults | frontend/backend/listen 공통 기본값 |
| frontend | 클라이언트 요청 수신 지점 |
| backend | 요청을 전달할 서버 그룹 |
| listen | frontend와 backend를 합친 형태 |
| acl | 요청 조건 검사 |
| use_backend | 조건에 따라 backend 선택 |
| server | 실제 Backend 서버 정의 |

### 3-2) HAProxy는 Ingress가 아니다

Kubernetes에서 Ingress와 HAProxy를 혼동하면 안 됩니다.

Ingress는 Kubernetes 리소스입니다.

```yaml
kind: Ingress
```

HAProxy는 실제 Proxy 소프트웨어입니다.

```text
Ingress:
Kubernetes 안에 저장되는 라우팅 규칙

Ingress Controller:
Ingress 규칙을 읽고 실제 Proxy 설정으로 반영하는 Controller

HAProxy:
실제 TCP/HTTP 트래픽을 처리하는 Proxy
```

HAProxy는 Kubernetes Ingress Controller로도 사용할 수 있고, Kubernetes 앞단 외부 Load Balancer로도 사용할 수 있습니다.

## 4. HAProxy 내부 동작 방식

### 4-1) Event-driven Architecture

HAProxy는 고성능 네트워크 처리를 위해 이벤트 기반 구조를 사용합니다.

전통적인 서버 모델에서는 연결 하나마다 스레드 하나를 붙이는 방식이 사용되기도 했습니다.

```text
Connection 1 -> Thread 1
Connection 2 -> Thread 2
Connection 3 -> Thread 3
```

이 방식은 연결 수가 많아지면 스레드 생성/전환 비용이 커집니다.

HAProxy는 이벤트 기반으로 많은 연결을 효율적으로 처리합니다.

```text
Event Loop
  ├─ Connection A readable
  ├─ Connection B writable
  ├─ Connection C timeout
  └─ Connection D closed
```

즉 HAProxy는 많은 연결 상태를 감시하고, 이벤트가 발생한 연결만 처리합니다.

### 4-2) Process, Thread, Worker

HAProxy는 버전과 설정에 따라 process와 thread를 사용할 수 있습니다. 현대 HAProxy에서는 multi-thread 구성이 일반적입니다.

```haproxy
global
    nbthread 8
```

의미:

```text
HAProxy worker가 8개 thread를 사용한다.
```

CPU 코어가 8개라면 `nbthread 8`을 고려할 수 있습니다. 단, 무조건 코어 수와 동일하게 설정한다고 항상 좋은 것은 아닙니다.

고려할 요소:

- CPU 코어 수
- TLS 처리량
- 동시 연결 수
- 네트워크 인터페이스 처리량
- NUMA 구조
- HAProxy 버전
- 커널 튜닝
- 로그 처리 방식

### 4-3) Connection, Session, Stream, Transaction

HAProxy 문서를 읽다 보면 다음 용어가 나옵니다.

```text
Connection
Session
Stream
Transaction
```

대략적으로는 다음처럼 이해할 수 있습니다.

| 용어 | 의미 |
|---|---|
| Connection | TCP 연결 자체 |
| Session | 클라이언트와 HAProxy 사이의 연결 흐름 |
| Stream | HAProxy 내부에서 요청/응답 흐름을 표현하는 단위 |
| Transaction | HTTP 요청 1개와 응답 1개 묶음 |

HTTP Keep-Alive를 사용하면 하나의 TCP Connection 안에서 여러 HTTP Transaction이 발생할 수 있습니다.

```text
TCP Connection 1개
  ├─ HTTP Request/Response 1
  ├─ HTTP Request/Response 2
  └─ HTTP Request/Response 3
```

이 개념을 모르면 keep-alive, timeout, connection reuse 장애를 이해하기 어렵습니다.

## 5. HAProxy 설정 파일 구조

HAProxy 설정 파일은 보통 `/etc/haproxy/haproxy.cfg`입니다.

기본 구조:

```haproxy
global
    log /dev/log local0
    maxconn 4096
    nbthread 4

defaults
    mode http
    log global
    option httplog
    option dontlognull
    timeout connect 5s
    timeout client  50s
    timeout server  50s

frontend fe_http
    bind *:80
    default_backend be_app

backend be_app
    balance roundrobin
    server app1 10.0.0.11:8080 check
    server app2 10.0.0.12:8080 check
```

### 5-1) global

`global`은 HAProxy 프로세스 전체 설정입니다.

주로 다음을 설정합니다.

- log
- maxconn
- nbthread
- daemon
- user/group
- chroot
- stats socket
- SSL 기본 옵션
- TLS cipher
- runtime API socket

예시:

```haproxy
global
    log /dev/log local0
    log /dev/log local1 notice
    maxconn 100000
    user haproxy
    group haproxy
    daemon
    nbthread 8
    stats socket /run/haproxy/admin.sock mode 660 level admin expose-fd listeners
```

중요한 옵션:

| 옵션 | 의미 |
|---|---|
| log | 로그 출력 위치 |
| maxconn | 전체 최대 동시 연결 수 |
| nbthread | HAProxy thread 수 |
| user/group | 실행 사용자 |
| daemon | 백그라운드 실행 |
| stats socket | Runtime API 소켓 |

### 5-2) defaults

`defaults`는 frontend, backend, listen에 공통으로 적용할 기본값입니다.

```haproxy
defaults
    mode http
    log global
    option httplog
    option dontlognull
    timeout connect 5s
    timeout client  60s
    timeout server  60s
    timeout http-request 10s
    timeout http-keep-alive 10s
```

`defaults`에 설정해두면 각 frontend/backend에서 반복해서 쓰지 않아도 됩니다.

### 5-3) frontend

`frontend`는 클라이언트 요청을 받는 영역입니다.

```haproxy
frontend fe_http
    bind *:80
    mode http
    acl host_api hdr(host) -i api.example.com
    use_backend be_api if host_api
    default_backend be_web
```

역할:

- 어떤 IP/Port에서 받을지 결정
- TCP/HTTP mode 지정
- TLS 인증서 적용
- ACL 평가
- backend 선택
- 요청 Header 조작
- access control
- redirect 처리

### 5-4) backend

`backend`는 실제 요청을 전달할 서버 그룹입니다.

```haproxy
backend be_api
    mode http
    balance roundrobin
    option httpchk GET /healthz
    http-check expect status 200
    server api1 10.0.0.11:8080 check
    server api2 10.0.0.12:8080 check
```

역할:

- Backend 서버 목록 정의
- Load Balancing Algorithm 적용
- Health Check 수행
- 서버별 weight/maxconn 설정
- Retry/timeout 설정
- 응답 Header 조작
- Sticky Session 처리

### 5-5) listen

`listen`은 frontend와 backend를 합친 형태입니다.

Stats 페이지처럼 간단한 설정에 자주 사용됩니다.

```haproxy
listen stats
    bind *:8404
    mode http
    stats enable
    stats uri /stats
    stats refresh 10s
    stats auth admin:password
```

## 6. Frontend 상세

### 6-1) Frontend는 진입점이다

Frontend는 HAProxy가 외부 요청을 받는 지점입니다.

```haproxy
frontend fe_web
    bind *:80
    mode http
    default_backend be_web
```

이 설정은 다음 의미입니다.

```text
모든 인터페이스의 80번 포트에서 요청을 받는다.
HTTP mode로 처리한다.
기본 backend는 be_web이다.
```

### 6-2) bind

`bind`는 HAProxy가 listen할 IP와 Port를 지정합니다.

```haproxy
bind *:80
bind 0.0.0.0:443
bind 192.168.10.10:443
```

TLS Termination을 할 때는 인증서를 함께 지정합니다.

```haproxy
bind *:443 ssl crt /etc/haproxy/certs/app.pem
```

여러 인증서를 디렉터리로 지정할 수도 있습니다.

```haproxy
bind *:443 ssl crt /etc/haproxy/certs/
```

### 6-3) Frontend에서 하는 일

Frontend에서 주로 처리하는 것은 다음입니다.

```text
1. 클라이언트 연결 수신
2. TLS Termination
3. SNI 확인
4. HTTP Header 확인
5. ACL 평가
6. 특정 backend로 분기
7. redirect 처리
8. 보안 정책 적용
9. 요청 Header 추가
```

예시:

```haproxy
frontend fe_https
    bind *:443 ssl crt /etc/haproxy/certs/
    mode http

    http-request set-header X-Forwarded-Proto https
    option forwardfor

    acl host_harbor hdr(host) -i harbor.example.com
    acl host_grafana hdr(host) -i grafana.example.com

    use_backend be_harbor if host_harbor
    use_backend be_grafana if host_grafana
    default_backend be_default
```

## 7. Backend 상세

### 7-1) Backend는 서버 그룹이다

Backend는 실제 트래픽을 받을 서버 목록입니다.

```haproxy
backend be_app
    balance roundrobin
    server app1 10.0.0.11:8080 check
    server app2 10.0.0.12:8080 check
```

이 설정은 다음 의미입니다.

```text
be_app backend에는 서버 2대가 있다.
roundrobin 방식으로 분산한다.
각 서버는 health check 대상이다.
```

### 7-2) server 옵션

`server` 라인은 실제 Backend 서버를 정의합니다.

```haproxy
server app1 10.0.0.11:8080 check weight 10 maxconn 100 inter 2s rise 2 fall 3
```

주요 옵션:

| 옵션 | 의미 |
|---|---|
| check | Health Check 활성화 |
| weight | 서버 가중치 |
| maxconn | 서버별 최대 연결 수 |
| backup | 기본 서버 장애 시 사용할 예비 서버 |
| inter | Health Check 간격 |
| rise | 정상으로 판단하기 위한 연속 성공 횟수 |
| fall | 장애로 판단하기 위한 연속 실패 횟수 |
| ssl | Backend로 HTTPS 연결 |
| verify none | Backend 인증서 검증 생략 |
| send-proxy | Backend로 PROXY Protocol 전달 |
| check-ssl | SSL Health Check 수행 |
| sni | Backend TLS 연결 시 SNI 지정 |

예시:

```haproxy
server app1 10.0.0.11:443 ssl verify none check
```

이 설정은 HAProxy가 Backend `10.0.0.11:443`으로 HTTPS 연결을 만든다는 의미입니다.

### 7-3) backup 서버

`backup` 옵션은 기본 서버가 모두 장애일 때만 사용하는 서버를 지정합니다.

```haproxy
backend be_app
    server app1 10.0.0.11:8080 check
    server app2 10.0.0.12:8080 check
    server app_backup 10.0.0.99:8080 check backup
```

사용 사례:

- DR 서버
- 임시 유지보수 서버
- 읽기 전용 fallback 페이지
- 장애 공지 서버

## 8. Balance Algorithm

### 8-1) roundrobin

```haproxy
balance roundrobin
```

서버에 순서대로 요청을 분산합니다.

```text
요청 1 -> app1
요청 2 -> app2
요청 3 -> app1
요청 4 -> app2
```

적합한 경우:

- 서버 성능이 비슷함
- 요청 처리 시간이 비슷함
- 단순 분산이 필요함

부적합한 경우:

- 요청마다 처리 시간이 크게 다름
- 장기 연결이 많음
- 서버 성능 차이가 큼

### 8-2) leastconn

```haproxy
balance leastconn
```

현재 연결 수가 가장 적은 서버로 보냅니다.

적합한 경우:

- WebSocket
- gRPC streaming
- 장기 TCP 연결
- 요청 처리 시간이 긴 서비스
- DB Proxy

주의할 점은 연결 수가 적다고 항상 CPU 부하가 낮은 것은 아니라는 점입니다.

### 8-3) source

```haproxy
balance source
```

클라이언트 source IP를 기준으로 Backend를 선택합니다.

같은 IP는 같은 Backend로 갈 가능성이 높습니다.

사용 사례:

- 간단한 Sticky Session
- 레거시 앱
- 서버 메모리 세션 의존 앱

주의:

- NAT 뒤 사용자들이 한 서버로 몰릴 수 있음
- 모바일 사용자의 IP가 바뀌면 sticky가 깨짐
- Proxy 뒤에서는 실제 Client IP가 아니라 Proxy IP 기준이 될 수 있음

### 8-4) uri

```haproxy
balance uri
```

URI를 기준으로 Backend를 선택합니다.

캐시 서버나 특정 경로를 같은 서버로 보내고 싶을 때 사용할 수 있습니다.

### 8-5) hdr

```haproxy
balance hdr(X-Tenant-ID)
```

특정 Header 값을 기준으로 Backend를 선택할 수 있습니다.

멀티테넌트 구조에서 tenant별로 같은 Backend에 보내고 싶은 경우 사용할 수 있습니다.

### 8-6) weighted load balancing

HAProxy에서는 server마다 weight를 줄 수 있습니다.

```haproxy
backend be_app
    balance roundrobin
    server app1 10.0.0.11:8080 check weight 10
    server app2 10.0.0.12:8080 check weight 5
    server app3 10.0.0.13:8080 check weight 1
```

의미:

```text
app1이 가장 많은 트래픽을 받는다.
app3는 가장 적은 트래픽을 받는다.
```

사용 사례:

- 서버 사양이 다름
- 신규 서버에 트래픽 일부만 보냄
- Canary 배포
- 점진적 트래픽 전환

## 9. HAProxy Mode

HAProxy에서 가장 중요한 개념 중 하나가 `mode`입니다.

```haproxy
mode tcp
mode http
```

이 설정에 따라 HAProxy가 트래픽을 해석하는 수준이 달라집니다.

### 9-1) TCP Mode

TCP Mode는 L4 Proxy입니다.

```haproxy
defaults
    mode tcp
```

또는

```haproxy
frontend fe_tcp
    mode tcp
```

TCP Mode에서 HAProxy는 TCP 연결을 다룹니다.

볼 수 있는 정보:

- Source IP
- Source Port
- Destination IP
- Destination Port
- TCP connection state
- TLS ClientHello 일부 정보, 예: SNI

볼 수 없는 정보:

- HTTP Host Header
- HTTP Path
- HTTP Method
- Cookie
- Authorization Header
- HTTP Status Code

사용 사례:

- SSH
- PostgreSQL
- MySQL
- Redis
- Kafka
- Kubernetes API Server
- TLS Passthrough
- raw TCP service

예시:

```haproxy
frontend fe_postgres
    bind *:5432
    mode tcp
    default_backend be_postgres

backend be_postgres
    mode tcp
    balance leastconn
    server pg1 10.0.0.11:5432 check
    server pg2 10.0.0.12:5432 check backup
```

### 9-2) HTTP Mode

HTTP Mode는 L7 Proxy입니다.

```haproxy
defaults
    mode http
```

HTTP Mode에서는 HAProxy가 HTTP 메시지를 해석할 수 있습니다.

볼 수 있는 정보:

- Host Header
- Path
- Method
- Header
- Cookie
- Query String
- HTTP Status Code
- Response Header
- Request/Response Body 일부

가능한 기능:

- Host 기반 라우팅
- Path 기반 라우팅
- Header 기반 라우팅
- Cookie 기반 Sticky Session
- HTTP Redirect
- Header Rewrite
- Rate Limit
- Access Control
- HTTP Health Check

예시:

```haproxy
frontend fe_http
    bind *:80
    mode http

    acl is_api path_beg /api
    use_backend be_api if is_api
    default_backend be_web
```

### 9-3) mode 선택 기준

| 상황 | 권장 mode |
|---|---|
| HTTP/HTTPS를 해제해서 Host/Path 라우팅 | http |
| TLS Passthrough | tcp |
| PostgreSQL, Redis, Kafka | tcp |
| HTTP Header 조작 필요 | http |
| X-Forwarded-For 추가 | http |
| PROXY Protocol 전달 | tcp/http 모두 가능 |
| SNI만 보고 TLS 라우팅 | tcp |
| HTTP Status 기반 health check | http |

## 10. Timeout

HAProxy 장애에서 timeout은 매우 중요합니다.

잘못된 timeout 설정은 다음 문제를 만듭니다.

- 408 Request Timeout
- 502 Bad Gateway
- 503 Service Unavailable
- 504 Gateway Timeout
- WebSocket 연결 끊김
- gRPC stream 끊김
- 대용량 업로드 실패
- 느린 API 중간 종료

기본 예시:

```haproxy
defaults
    timeout connect 5s
    timeout client  60s
    timeout server  60s
    timeout http-request 10s
    timeout http-keep-alive 10s
    timeout tunnel 1h
```

### 10-1) timeout connect

Backend 서버와 연결을 맺는 데 허용하는 시간입니다.

```haproxy
timeout connect 5s
```

Backend가 너무 느리게 연결되거나 방화벽에서 지연되면 connect timeout이 발생합니다.

가능 원인:

- Backend port 미오픈
- 방화벽 drop
- 라우팅 문제
- 서버 overload
- SYN backlog 문제

### 10-2) timeout client

클라이언트와 HAProxy 사이에서 데이터가 오지 않는 상태를 얼마나 기다릴지 결정합니다.

```haproxy
timeout client 60s
```

너무 짧으면 느린 클라이언트나 대용량 업로드가 끊길 수 있습니다.

### 10-3) timeout server

HAProxy와 Backend 사이에서 응답을 얼마나 기다릴지 결정합니다.

```haproxy
timeout server 60s
```

Backend API가 90초 걸리는데 `timeout server 60s`라면 60초 후 HAProxy가 연결을 종료할 수 있습니다.

### 10-4) timeout http-request

클라이언트가 HTTP request header 전체를 보내는 데 허용하는 시간입니다.

```haproxy
timeout http-request 10s
```

너무 길면 slowloris 공격에 취약하고, 너무 짧으면 느린 클라이언트가 실패할 수 있습니다.

### 10-5) timeout http-keep-alive

HTTP Keep-Alive 연결을 유휴 상태로 얼마나 유지할지 결정합니다.

```haproxy
timeout http-keep-alive 10s
```

Backend keepalive timeout과 맞지 않으면 연결 재사용 시 오류가 발생할 수 있습니다.

### 10-6) timeout tunnel

WebSocket, CONNECT tunnel, TCP tunnel처럼 장시간 유지되는 연결에 중요합니다.

```haproxy
timeout tunnel 1h
```

WebSocket이 일정 시간 후 끊긴다면 이 값을 확인해야 합니다.

## 11. Logging

### 11-1) 로그 설정

HAProxy 로그는 운영에서 매우 중요합니다.

확인 명령어:

```bash
journalctl -u haproxy -f

tail -f /var/log/syslog
```

### 11-2) HTTP 로그에서 봐야 할 것

HAProxy HTTP log에는 다음 정보가 포함될 수 있습니다.

- Client IP
- Frontend 이름
- Backend 이름
- Server 이름
- 요청 시간
- 응답 시간
- HTTP method
- Path
- Status Code
- Bytes
- Termination State
- Queue time
- Connect time
- Response time
- Active connection 수

장애 분석에서 특히 중요한 것은 다음입니다.

```text
status code
backend/server name
response time
termination state
queue time
connect time
```

## 12. TLS

### 12-1) TLS Termination

TLS Termination은 HAProxy가 HTTPS를 해제하고 Backend로 HTTP를 전달하는 방식입니다.

```text
Client --HTTPS--> HAProxy --HTTP--> Backend
```

설정 예시:

```haproxy
frontend fe_https
    bind *:443 ssl crt /etc/haproxy/certs/app.pem
    mode http
    http-request set-header X-Forwarded-Proto https
    option forwardfor
    default_backend be_app

backend be_app
    mode http
    server app1 10.0.0.11:8080 check
```

장점:

- HAProxy에서 Host/Path/Header 라우팅 가능
- 인증서 중앙 관리
- Backend는 HTTP만 처리하면 됨
- Access log 확보 용이

단점:

- HAProxy와 Backend 사이가 평문 HTTP일 수 있음
- Backend가 원래 요청이 HTTPS였는지 모름
- `X-Forwarded-Proto` 설정이 중요

### 12-2) SSL Offloading

SSL Offloading은 TLS 처리 비용을 HAProxy에서 담당하는 구조입니다.

```text
Backend는 TLS 암복호화 부담이 줄어든다.
```

TLS Handshake는 CPU를 사용합니다. 트래픽이 많으면 Backend마다 인증서를 관리하고 TLS를 처리하는 것이 부담이 됩니다.

### 12-3) SSL Bridging / Re-encrypt

SSL Bridging은 HAProxy에서 TLS를 해제한 뒤 Backend로 다시 HTTPS를 사용하는 구조입니다.

```text
Client --HTTPS--> HAProxy --HTTPS--> Backend
```

설정 예시:

```haproxy
backend be_secure
    mode http
    server app1 10.0.0.11:443 ssl verify none check
```

장점:

- HAProxy가 HTTP 내용을 보고 L7 라우팅 가능
- 내부 구간도 암호화 가능

단점:

- Backend 인증서 검증 설정 필요
- TLS 처리 비용 증가
- SNI와 CA chain 설정이 복잡할 수 있음

### 12-4) SSL Passthrough

SSL Passthrough는 HAProxy가 TLS를 해제하지 않고 Backend로 그대로 전달하는 방식입니다.

```text
Client --HTTPS--> HAProxy --HTTPS--> Backend
```

이때 HAProxy는 HTTP Header/Path를 볼 수 없습니다.

TCP mode를 사용합니다.

```haproxy
frontend fe_tls_passthrough
    bind *:443
    mode tcp
    default_backend be_tls_backend

backend be_tls_backend
    mode tcp
    server app1 10.0.0.11:443 check
```

장점:

- End-to-End TLS 유지
- Backend가 직접 인증서 처리
- HAProxy가 평문 HTTP를 보지 않음

단점:

- HTTP Host/Path 라우팅 불가
- X-Forwarded-For 같은 HTTP Header 추가 불가
- L7 기능 사용 제한
- Access log가 제한적

## 13. Certificate

### 13-1) PEM 파일

HAProxy에서 TLS 인증서를 지정할 때 자주 사용하는 형식은 PEM입니다.

HAProxy에서 `crt`로 지정하는 파일은 보통 다음 내용을 함께 포함해야 합니다.

```text
서버 인증서
중간 인증서 chain
개인키
```

예시:

```text
-----BEGIN CERTIFICATE-----
...
-----END CERTIFICATE-----
-----BEGIN CERTIFICATE-----
...
-----END CERTIFICATE-----
-----BEGIN PRIVATE KEY-----
...
-----END PRIVATE KEY-----
```

### 13-2) crt와 key 분리 파일

일부 환경에서는 `.crt`, `.key`가 분리되어 있습니다.

HAProxy에서 하나의 PEM으로 합칠 수 있습니다.

```bash
cat server.crt intermediate.crt server.key > app.pem
```

주의:

- private key 권한 보호
- 인증서 chain 순서
- 만료일 확인
- SAN 확인
- wildcard 인증서 범위 확인

### 13-3) 인증서 확인 명령어

```bash
openssl x509 -in app.pem -noout -text
openssl x509 -in app.pem -noout -dates
openssl x509 -in app.pem -noout -subject -issuer
```

원격 서버 확인:

```bash
openssl s_client -connect app.example.com:443 -servername app.example.com
```

확인할 항목:

- CN
- SAN
- 만료일
- 발급자
- chain
- SNI 응답 인증서
- TLS version

## 14. SNI

### 14-1) SNI란

SNI는 Server Name Indication입니다.

TLS Handshake 단계에서 클라이언트가 접속하려는 도메인을 서버에 알려주는 기능입니다.

왜 필요할까요?

하나의 IP에서 여러 HTTPS 사이트를 운영할 수 있기 때문입니다.

```text
203.0.113.10
  ├─ harbor.example.com
  ├─ grafana.example.com
  ├─ keycloak.example.com
  └─ argocd.example.com
```

TLS 인증서는 도메인별로 다를 수 있습니다. 서버는 ClientHello의 SNI를 보고 어떤 인증서를 반환할지 결정합니다.

### 14-2) HTTP Host와 SNI 차이

| 구분 | SNI | Host Header |
|---|---|---|
| 위치 | TLS Handshake | HTTP Request |
| 암호화 전/후 | TLS 초기에 확인 | TLS 해제 후 확인 |
| 사용 목적 | 인증서 선택, TLS 라우팅 | HTTP 가상호스트, L7 라우팅 |
| TCP mode에서 확인 | 가능 | 불가 |
| HTTP mode에서 확인 | TLS 종료 후 가능 | 가능 |

SSL Passthrough에서는 HAProxy가 HTTP Host를 볼 수 없습니다. 하지만 SNI는 볼 수 있으므로 SNI 기반으로 TCP backend를 선택할 수 있습니다.

### 14-3) SNI 기반 TCP 라우팅

```haproxy
frontend fe_tls
    bind *:443
    mode tcp
    tcp-request inspect-delay 5s
    tcp-request content accept if { req_ssl_hello_type 1 }

    acl sni_harbor req.ssl_sni -i harbor.example.com
    acl sni_grafana req.ssl_sni -i grafana.example.com

    use_backend be_harbor_tls if sni_harbor
    use_backend be_grafana_tls if sni_grafana
    default_backend be_default_tls

backend be_harbor_tls
    mode tcp
    server harbor 10.0.0.11:443 check

backend be_grafana_tls
    mode tcp
    server grafana 10.0.0.12:443 check
```

이 구조에서는 HAProxy가 TLS를 해제하지 않습니다. 따라서 Backend가 인증서를 가지고 있어야 합니다.

## 15. ACL

ACL은 Access Control List의 약자입니다. HAProxy에서는 요청 조건을 정의하는 데 사용합니다.

```haproxy
acl host_api hdr(host) -i api.example.com
acl path_admin path_beg /admin
```

ACL은 단독으로 끝나지 않고 `use_backend`, `http-request deny`, `redirect` 같은 액션과 함께 사용됩니다.

```haproxy
use_backend be_api if host_api
http-request deny if path_admin
```

### 15-1) Host 기반 ACL

```haproxy
acl host_api hdr(host) -i api.example.com
acl host_admin hdr(host) -i admin.example.com

use_backend be_api if host_api
use_backend be_admin if host_admin
```

### 15-2) Path 기반 ACL

```haproxy
acl path_api path_beg /api
acl path_static path_beg /static
acl path_health path /healthz
```

조건:

| 조건 | 의미 |
|---|---|
| path | 정확한 path |
| path_beg | prefix |
| path_end | suffix |
| path_sub | 포함 여부 |
| path_reg | 정규식 |

### 15-3) Method 기반 ACL

```haproxy
acl is_post method POST
acl is_get method GET

http-request deny if is_post path_admin
```

### 15-4) Header 기반 ACL

```haproxy
acl has_token req.hdr(Authorization) -m found
acl tenant_a req.hdr(X-Tenant-ID) -i tenant-a
```

### 15-5) Cookie 기반 ACL

```haproxy
acl beta_user req.cook(beta) -i true
use_backend be_beta if beta_user
```

### 15-6) Source IP 기반 ACL

```haproxy
acl internal_net src 10.0.0.0/8
acl office_ip src 203.0.113.10

http-request deny unless internal_net
```

### 15-7) ACL 조합

```haproxy
acl host_api hdr(host) -i api.example.com
acl path_v1 path_beg /v1
acl is_post method POST

use_backend be_api_v1_write if host_api path_v1 is_post
```

조건을 조합하면 세밀한 정책을 만들 수 있습니다. 다만 너무 많은 ACL을 무분별하게 사용하면 설정 가독성이 떨어지고 장애 분석이 어려워집니다.

## 16. Request Routing

HAProxy의 라우팅은 보통 다음 흐름으로 동작합니다.

```text
요청 수신
  ↓
Host 확인
  ↓
Path 확인
  ↓
Method/Header/Cookie 확인
  ↓
ACL 평가
  ↓
use_backend 선택
  ↓
default_backend fallback
```

예시:

```haproxy
frontend fe_web
    bind *:80
    mode http

    acl host_api hdr(host) -i api.example.com
    acl host_admin hdr(host) -i admin.example.com
    acl path_metrics path_beg /metrics

    use_backend be_metrics if path_metrics
    use_backend be_api if host_api
    use_backend be_admin if host_admin
    default_backend be_web
```

주의할 점:

- ACL 순서와 use_backend 순서가 중요할 수 있음
- default_backend가 없으면 요청 처리 실패 가능
- Host Header가 port를 포함할 수 있음
- path rewrite가 필요한 경우 Backend 앱 경로와 맞춰야 함

## 17. Header Manipulation

HAProxy는 HTTP Header를 추가, 수정, 삭제할 수 있습니다.

### 17-1) X-Forwarded-For

원본 Client IP 전달:

```haproxy
option forwardfor
```

또는 직접 설정:

```haproxy
http-request set-header X-Forwarded-For %[src]
```

### 17-2) X-Forwarded-Proto

TLS Termination 후 Backend가 원래 요청이 HTTPS였는지 알 수 있도록 설정합니다.

```haproxy
http-request set-header X-Forwarded-Proto https if { ssl_fc }
http-request set-header X-Forwarded-Proto http unless { ssl_fc }
```

이 설정은 Keycloak, Harbor, Grafana, ArgoCD에서 매우 중요합니다.

### 17-3) X-Forwarded-Host

```haproxy
http-request set-header X-Forwarded-Host %[req.hdr(Host)]
```

원래 Host를 Backend에 전달합니다.

### 17-4) X-Request-ID

요청 추적을 위해 Request ID를 추가할 수 있습니다.

```haproxy
http-request set-header X-Request-ID %[unique-id]
```

그리고 global/defaults에서 unique-id-format을 설정할 수 있습니다.

```haproxy
unique-id-format %{+X}o\ %ci:%cp_%fi:%fp_%Ts_%rt:%pid
```

X-Request-ID는 다음을 연결하는 데 사용됩니다.

```text
HAProxy Access Log
Ingress Log
Application Log
APM Trace
WhaTap / Grafana / Loki
```

### 17-5) Header 삭제

민감하거나 신뢰할 수 없는 Header를 삭제할 수 있습니다.

```haproxy
http-request del-header X-Forwarded-For
http-request del-header X-Real-IP
```

이후 HAProxy가 신뢰 가능한 값으로 다시 설정하는 방식이 안전합니다.

## 18. PROXY Protocol

### 18-1) PROXY Protocol이 필요한 이유

HTTP에서는 `X-Forwarded-For`로 Client IP를 전달할 수 있습니다. 하지만 TCP 서비스에는 HTTP Header가 없습니다.

예를 들어 다음 서비스들은 HTTP Header를 사용할 수 없습니다.

- PostgreSQL
- MySQL
- Redis
- Kafka
- raw TCP service
- TLS Passthrough 상태의 HTTPS

이때 원본 Client IP를 전달하려면 PROXY Protocol을 사용할 수 있습니다.

### 18-2) send-proxy

HAProxy가 Backend로 PROXY Protocol을 전달하는 설정입니다.

```haproxy
backend be_tcp
    mode tcp
    server app1 10.0.0.11:443 send-proxy check
```

Backend는 연결 시작 부분에서 PROXY Protocol header를 읽어야 합니다. Backend가 PROXY Protocol을 지원하지 않으면 오류가 발생합니다.

### 18-3) accept-proxy

HAProxy가 앞단 Load Balancer로부터 PROXY Protocol을 받는 설정입니다.

```haproxy
frontend fe_https
    bind *:443 accept-proxy
    mode tcp
    default_backend be_app
```

사용 예:

```text
AWS NLB
  ↓ PROXY Protocol
HAProxy
  ↓
Backend
```

### 18-4) X-Forwarded-For와 PROXY Protocol 비교

| 구분 | X-Forwarded-For | PROXY Protocol |
|---|---|---|
| 계층 | HTTP | TCP |
| 사용 가능 대상 | HTTP/HTTPS | TCP/HTTP 모두 |
| TLS Passthrough | 제한적 | 가능 |
| Backend 설정 필요 | HTTP 앱에서 Header 신뢰 | PROXY Protocol 파싱 필요 |
| 잘못 설정 시 | Client IP 오류 | 프로토콜 파싱 오류, 400, connection close |

## 19. Health Check

### 19-1) Health Check의 목적

HAProxy는 Backend 서버 상태를 확인하고, 비정상 서버를 트래픽 대상에서 제외할 수 있습니다.

```text
Healthy Server   -> 요청 전달
Unhealthy Server -> 요청 제외
```

### 19-2) TCP Check

TCP 연결 가능 여부만 확인합니다.

```haproxy
backend be_tcp
    mode tcp
    server app1 10.0.0.11:443 check
```

장점:

- 단순함
- TCP 서비스에 범용 사용 가능
- 빠름

단점:

- 애플리케이션 정상 여부는 모름
- 포트만 열려 있어도 정상으로 판단

### 19-3) HTTP Check

HTTP endpoint를 호출해서 상태를 확인합니다.

```haproxy
backend be_http
    mode http
    option httpchk GET /healthz
    http-check expect status 200
    server app1 10.0.0.11:8080 check
```

응답이 200이면 정상으로 판단합니다.

### 19-4) HTTPS Check

Backend가 HTTPS라면 SSL 옵션이 필요합니다.

```haproxy
backend be_https
    mode http
    option httpchk GET /healthz
    http-check expect status 200
    server app1 10.0.0.11:443 ssl verify none check
```

운영에서는 가능하면 `verify none` 대신 CA 검증을 구성하는 것이 좋습니다. 다만 사내 self-signed 인증서 환경에서는 임시로 `verify none`을 사용하는 경우도 있습니다.

### 19-5) rise, fall, inter

```haproxy
server app1 10.0.0.11:8080 check inter 2s rise 2 fall 3
```

의미:

| 옵션 | 의미 |
|---|---|
| inter 2s | 2초마다 check |
| rise 2 | 2번 연속 성공하면 UP |
| fall 3 | 3번 연속 실패하면 DOWN |

한 번 실패했다고 바로 DOWN 처리하면 일시적 네트워크 흔들림에도 장애 전환이 발생할 수 있습니다.

### 19-6) Health Check 설계 주의점

좋은 health check는 다음을 만족해야 합니다.

- 빠르게 응답
- 실제 트래픽 수신 가능 여부 반영
- 너무 많은 dependency 포함 금지
- 배포 직후 warm-up 고려
- DB 장애를 무조건 전체 앱 장애로 확대하지 않도록 주의

예를 들어 `/healthz`는 프로세스 생존 여부, `/readyz`는 실제 요청 수신 가능 여부로 나누는 것이 좋습니다.

## 20. Stick Table

### 20-1) Stick Table이란

Stick Table은 HAProxy가 특정 key에 대한 상태 정보를 저장하는 메모리 테이블입니다.

저장 가능한 key 예:

- Source IP
- Cookie
- Header
- URL parameter
- TLS fingerprint

저장 가능한 값 예:

- 요청 수
- 연결 수
- 에러 수
- byte 수
- rate
- session 정보

### 20-2) 사용 사례

Stick Table은 다음에 사용됩니다.

- Sticky Session
- Rate Limiting
- DDoS 방어
- 로그인 brute force 방어
- 특정 IP 요청 추적
- 비정상 요청 차단

### 20-3) 기본 예시

```haproxy
backend be_app
    stick-table type ip size 100k expire 10m store http_req_rate(10s)
    http-request track-sc0 src
    http-request deny if { sc_http_req_rate(0) gt 100 }
    server app1 10.0.0.11:8080 check
```

의미:

```text
source IP별 요청 rate를 10분 동안 추적한다.
10초 동안 요청 수가 100을 넘으면 차단한다.
```

## 21. Sticky Session

### 21-1) Source IP 기반 Sticky

```haproxy
backend be_app
    balance source
    server app1 10.0.0.11:8080 check
    server app2 10.0.0.12:8080 check
```

같은 source IP가 같은 서버로 갈 가능성이 높습니다.

단점:

- NAT 뒤 사용자 몰림
- 모바일 IP 변경 시 세션 깨짐
- Proxy IP 기준으로 묶일 수 있음

### 21-2) Cookie 기반 Sticky

```haproxy
backend be_app
    balance roundrobin
    cookie SERVERID insert indirect nocache
    server app1 10.0.0.11:8080 check cookie app1
    server app2 10.0.0.12:8080 check cookie app2
```

HAProxy가 `SERVERID` cookie를 삽입하고, 다음 요청부터 같은 서버로 보냅니다.

주의:

- 서버 장애 시 sticky가 깨질 수 있음
- Cookie 보안 속성 고려 필요
- 가능하면 Redis/DB 기반 외부 세션 저장소 권장

## 22. Rate Limiting

Rate Limit은 과도한 요청을 제한하는 기능입니다.

```haproxy
frontend fe_http
    bind *:80
    mode http

    stick-table type ip size 100k expire 10m store http_req_rate(10s)
    http-request track-sc0 src
    http-request deny if { sc_http_req_rate(0) gt 100 }

    default_backend be_app
```

의미:

```text
source IP 기준으로 10초 요청 수가 100개를 넘으면 차단
```

사용 사례:

- brute force 방어
- 과도한 API 호출 제한
- 임시 DDoS 완화
- 로그인 endpoint 보호
- 내부 관리 페이지 보호

주의:

- NAT 환경에서는 여러 사용자가 하나의 IP로 묶일 수 있음
- 정상 사용자도 차단될 수 있음
- 인증 사용자 기준 rate limit은 애플리케이션/API Gateway에서 처리하는 것이 더 정확할 수 있음

## 23. Runtime API

### 23-1) Runtime API란

Runtime API는 HAProxy를 실행 중에 제어하고 상태를 조회하는 인터페이스입니다.

설정:

```haproxy
global
    stats socket /run/haproxy/admin.sock mode 660 level admin expose-fd listeners
```

접속:

```bash
echo "show info" | socat - /run/haproxy/admin.sock
echo "show stat" | socat - /run/haproxy/admin.sock
```

### 23-2) 자주 쓰는 명령

```bash
echo "show info" | socat - /run/haproxy/admin.sock
echo "show stat" | socat - /run/haproxy/admin.sock
echo "show servers state" | socat - /run/haproxy/admin.sock
echo "show table" | socat - /run/haproxy/admin.sock
```

특정 서버를 drain할 수도 있습니다.

```bash
echo "set server be_app/app1 state drain" | socat - /run/haproxy/admin.sock
```

다시 ready로 바꾸기:

```bash
echo "set server be_app/app1 state ready" | socat - /run/haproxy/admin.sock
```

### 23-3) Runtime API 사용 시 주의

Runtime API는 강력합니다.

운영에서는 다음을 지켜야 합니다.

- socket 권한 제한
- root 또는 haproxy group만 접근
- 외부 네트워크 노출 금지
- 변경 명령 기록
- 자동화 스크립트 오작동 방지

## 24. Zero-Downtime Reload

HAProxy는 설정 변경 후 reload를 통해 무중단에 가깝게 설정을 반영할 수 있습니다.

설정 문법 검사:

```bash
haproxy -c -f /etc/haproxy/haproxy.cfg
```

reload:

```bash
systemctl reload haproxy
```

Zero-downtime reload의 개념:

```text
기존 worker:
기존 연결 처리 유지

새 worker:
새 설정으로 신규 연결 처리
```

주의할 점:

- 설정 문법 검사를 먼저 해야 함
- 오래 유지되는 WebSocket/gRPC 연결 고려
- reload가 너무 잦으면 운영 부담
- cert 파일 권한/경로 확인
- Runtime state 유지 설정 고려

## 25. Backend State

HAProxy Stats나 Runtime API에서 Backend server 상태를 확인할 수 있습니다.

대표 상태:

| 상태 | 의미 |
|---|---|
| UP | 정상, 트래픽 수신 가능 |
| DOWN | 장애, 트래픽 제외 |
| MAINT | 유지보수 모드 |
| DRAIN | 새 세션은 받지 않고 기존 세션 처리 |
| NOLB | Load balancing 대상에서 제외 |
| BACKUP | 기본 서버 장애 시 사용되는 예비 서버 |

### 25-1) DRAIN

DRAIN은 배포나 유지보수 때 중요합니다.

```text
새 요청은 보내지 않는다.
기존 연결은 처리하게 둔다.
```

Runtime API 예시:

```bash
echo "set server be_app/app1 state drain" | socat - /run/haproxy/admin.sock
```

### 25-2) MAINT

MAINT는 서버를 유지보수 상태로 둡니다.

```bash
echo "set server be_app/app1 state maint" | socat - /run/haproxy/admin.sock
```

이 상태에서는 서버가 트래픽을 받지 않습니다.


접속:

```text
http://<haproxy-ip>:8404/stats
```

권장:

- 사내망에서만 접근
- Basic Auth 사용
- 방화벽 제한
- 별도 관리망 사용
- 인터넷 직접 노출 금지

### 26-2) Stats에서 봐야 할 지표

| 항목 | 의미 |
|---|---|
| Status | UP/DOWN/MAINT/DRAIN |
| Sessions | 현재 세션 수 |
| Rate | 초당 요청/연결 비율 |
| Bytes | 송수신 바이트 |
| Denied | 거부된 요청 |

장애 시 우선 볼 것:

```text
1. Backend server가 UP인가?
2. 특정 server만 세션이 몰리는가?
3. error 수가 증가하는가?
4. check가 실패 중인가?

6. queue가 쌓이는가?
```

## 27. HAProxy High Availability

HAProxy 자체도 장애 날 수 있습니다. 따라서 운영 환경에서는 HAProxy를 이중화해야 합니다.

### 27-1) Active-Standby

```text
Client
  ↓
VIP
  ↓
HAProxy Active
HAProxy Standby
```

Active가 장애 나면 Standby가 VIP를 가져갑니다.

온프레미스에서는 보통 Keepalived와 VRRP를 사용합니다.

### 27-2) Active-Active

여러 HAProxy가 동시에 트래픽을 처리합니다.

구현 방법:

- DNS Round Robin
- Anycast
- 상위 L4 Load Balancer
- ECMP
- 클라우드 LB

장점:

- 자원 활용도 높음
- 처리량 증가

단점:

- 상태 동기화 복잡
- stick table 동기화 고려
- 장애 분석 복잡

## 28. HAProxy + Keepalived

Keepalived는 VRRP를 사용해 VIP를 Active 서버에 붙입니다.

구조:

```text
VIP: 192.168.10.200

HAProxy-1: 192.168.10.11
HAProxy-2: 192.168.10.12
```

평상시:

```text
VIP -> HAProxy-1
```

HAProxy-1 장애:

```text
VIP -> HAProxy-2
```

확인 명령어:

```bash
ip addr
systemctl status keepalived
journalctl -u keepalived -f
```

Keepalived는 HAProxy 프로세스 상태를 감시하도록 구성할 수 있습니다.

```text
HAProxy 죽음
  ↓
Keepalived priority 감소
  ↓
VIP가 Standby로 이동
```

## 29. HAProxy + Kubernetes

### 29-1) Ingress Controller 앞단 HAProxy

온프레미스 Kubernetes에서 자주 쓰는 구조:

```text
Client
  ↓
HAProxy
  ↓
Ingress Controller NodePort 또는 LoadBalancer IP
  ↓
Ingress Controller Pod
  ↓
Service
  ↓
Pod
```

HAProxy는 외부 진입점 역할을 하고, Ingress Controller는 Kubernetes 내부 L7 라우팅을 담당합니다.

### 29-2) NodePort 연동

```haproxy
backend be_ingress_http
    mode tcp
    balance roundrobin
    server node1 192.168.10.83:30080 check
    server node2 192.168.10.84:30080 check
    server node3 192.168.10.85:30080 check
```

HTTPS:

```haproxy
backend be_ingress_https
    mode tcp
    balance roundrobin
    server node1 192.168.10.83:30443 check
    server node2 192.168.10.84:30443 check
    server node3 192.168.10.85:30443 check
```

### 29-3) ExternalTrafficPolicy

Kubernetes Service에서 `externalTrafficPolicy: Local`을 사용하면 원본 Client IP 보존에 유리합니다.

하지만 HAProxy가 어느 Node로 보낼지와 해당 Node에 Ingress Pod가 있는지가 중요합니다.

```text
HAProxy -> Node A
Node A에 Ingress Pod 없음
  ↓
트래픽 처리 실패 가능
```

이 경우 Ingress Controller를 DaemonSet으로 배포하거나, HAProxy health check가 local endpoint가 있는 node만 healthy로 보도록 설계해야 합니다.

### 29-4) MetalLB 연동

MetalLB가 LoadBalancer IP를 제공하고, HAProxy는 그 앞단 또는 대체 구성으로 들어갈 수 있습니다.

구조 예:

```text
Client
  ↓
HAProxy VIP
  ↓
MetalLB External IP
  ↓
Ingress Controller
```

또는:

```text
Client
  ↓
MetalLB External IP
  ↓
Ingress Controller
```

둘 중 무엇이 맞는지는 네트워크 설계, VIP 관리, 인증서 위치, 운영 정책에 따라 다릅니다.

## 30. Harbor 구조 예제

Harbor를 HAProxy 뒤에 둘 때 중요한 것은 다음입니다.

- HTTPS 인식
- external URL
- X-Forwarded-Proto
- X-Forwarded-Host
- 대용량 이미지 push/pull
- timeout
- body size
- Docker client와 registry API 동작

구조:

```text
Docker Client
  ↓
HAProxy
  ↓
Ingress Controller
  ↓
Harbor Service
  ↓
Harbor Pod
```

TLS Termination이 HAProxy에서 발생한다면 Backend는 요청을 HTTP로 받을 수 있습니다.

이때 Harbor가 원래 요청을 HTTPS로 인식해야 합니다.

```haproxy
http-request set-header X-Forwarded-Proto https if { ssl_fc }
http-request set-header X-Forwarded-Host %[req.hdr(Host)]
option forwardfor
```

문제 예:

```text
docker login 실패
redirect URL이 http로 생성
token service URL 불일치
image push 중 timeout
```

확인할 것:

```text
Harbor external_url
Ingress TLS 설정
HAProxy X-Forwarded-* Header
timeout client/server
request body 제한
registry service endpoint
```

## 34. Ceph Dashboard 구조 예제

```text

  ↓
HAProxy
  ↓
Ceph Manager Dashboard
```

```text
400 Bad Request
```

확인할 것:

```text
현재 active mgr는 어디인가?

Health Check가 dashboard endpoint와 맞는가?
```

Ceph Dashboard가 특정 Host/Scheme을 기대하는 경우, HAProxy에서 Header 전달이 맞지 않으면 400 계열 오류가 발생할 수 있습니다.

## 35. 실전 운영 예제

### 35-1) Host 기반 HTTP 라우팅

```haproxy
frontend fe_https
    bind *:443 ssl crt /etc/haproxy/certs/
    mode http

    option forwardfor
    http-request set-header X-Forwarded-Proto https if { ssl_fc }
    http-request set-header X-Forwarded-Host %[req.hdr(Host)]

    acl host_harbor hdr(host) -i harbor.example.com
    acl host_grafana hdr(host) -i grafana.example.com
    acl host_keycloak hdr(host) -i keycloak.example.com

    use_backend be_harbor if host_harbor
    use_backend be_grafana if host_grafana
    use_backend be_keycloak if host_keycloak
    default_backend be_default

backend be_harbor
    mode http
    balance roundrobin
    option httpchk GET /api/v2.0/ping
    server harbor1 10.0.0.11:80 check

backend be_grafana
    mode http
    option httpchk GET /api/health
    server grafana1 10.0.0.12:3000 check

backend be_keycloak
    mode http
    option httpchk GET /health/ready
    server keycloak1 10.0.0.13:8080 check
```

### 35-2) Path 기반 라우팅

```haproxy
frontend fe_web
    bind *:80
    mode http

    acl path_api path_beg /api
    acl path_admin path_beg /admin

    use_backend be_api if path_api
    use_backend be_admin if path_admin
    default_backend be_frontend
```

### 35-3) TCP Proxy 예제

```haproxy
frontend fe_k8s_api
    bind *:6443
    mode tcp
    default_backend be_k8s_api

backend be_k8s_api
    mode tcp
    balance roundrobin
    server cp1 192.168.10.83:6443 check
    server cp2 192.168.10.84:6443 check
    server cp3 192.168.10.85:6443 check
```

Kubernetes API Server처럼 HTTPS 기반이지만 HAProxy에서 내용을 해석할 필요가 없다면 TCP mode가 적합할 수 있습니다.

### 35-4) TLS Passthrough + SNI 예제

```haproxy
frontend fe_tls
    bind *:443
    mode tcp
    tcp-request inspect-delay 5s
    tcp-request content accept if { req_ssl_hello_type 1 }

    acl sni_harbor req.ssl_sni -i harbor.example.com
    acl sni_grafana req.ssl_sni -i grafana.example.com

    use_backend be_harbor_tls if sni_harbor
    use_backend be_grafana_tls if sni_grafana
    default_backend be_default_tls
```

## 38. 장애 유형별 원인

### 38-1) 502 Bad Gateway

가능 원인:

- Backend port 미오픈
- Backend connection refused
- Backend가 TLS인데 HAProxy가 HTTP로 접속
- Backend가 HTTP인데 HAProxy가 HTTPS로 접속
- PROXY Protocol mismatch
- Backend가 응답 전 연결 종료
- timeout server

확인:

```bash
curl -v http://backend:port
ss -lntp
journalctl -u haproxy -f
```

### 38-2) 503 Service Unavailable

가능 원인:

- 모든 backend server DOWN
- Health Check 실패
- ACL은 맞지만 backend에 usable server 없음
- 서버가 MAINT/DRAIN 상태
- DNS resolver 실패

확인:

```bash
echo "show stat" | socat - /run/haproxy/admin.sock
```

### 38-3) 504 Gateway Timeout

가능 원인:

- Backend 응답 지연
- timeout server 짧음
- DB 지연
- 애플리케이션 thread pool 고갈
- 네트워크 지연

확인:

```bash
curl -v --max-time 10 https://app.example.com/slow
```

### 38-4) TLS Error

가능 원인:

- 인증서 파일 경로 오류
- PEM에 private key 없음
- 인증서 만료
- SAN 불일치
- chain 누락
- cipher mismatch

확인:

```bash
haproxy -c -f /etc/haproxy/haproxy.cfg
openssl s_client -connect app.example.com:443 -servername app.example.com
```

### 38-5) SNI Error

가능 원인:

- 클라이언트가 SNI를 보내지 않음
- ACL의 req.ssl_sni 조건 오류
- inspect-delay 부족
- 잘못된 backend 선택
- wildcard 인증서 범위 불일치

### 38-6) PROXY Protocol Error

가능 원인:

- HAProxy가 send-proxy를 보냈는데 backend가 지원하지 않음
- 앞단 LB가 PROXY Protocol을 보내는데 HAProxy bind에 accept-proxy 없음
- v1/v2 protocol mismatch

증상:

- 400 Bad Request
- connection reset
- backend 로그에 이상한 문자열
- TLS handshake failure

### 38-7) Infinite Redirect

가능 원인:

- X-Forwarded-Proto 누락
- Backend가 HTTPS를 HTTP로 인식
- 애플리케이션 base URL 오류
- OAuth redirect URI 불일치
- Cookie Secure/SameSite 설정 오류

관련 서비스:

- Keycloak
- Harbor
- Grafana
- ArgoCD

## 41. 요약

HAProxy는 단순한 로드밸런서가 아닙니다.

운영 환경에서는 다음을 모두 이해해야 합니다.

```text
Frontend
Backend
Bind
Server
ACL
TCP Mode
HTTP Mode
TLS Termination
SSL Passthrough
SNI
X-Forwarded-For
X-Forwarded-Proto
PROXY Protocol
Health Check
Stick Table
Sticky Session
Rate Limit
Runtime API
Stats
Zero-Downtime Reload
Keepalived
Kubernetes 연동
Harbor/Keycloak/Grafana/ArgoCD/Ceph Dashboard Proxy 설정
```

특히 Kubernetes 환경에서는 HAProxy가 다음 흐름 중 하나에 위치합니다.

```text
Client
  ↓
HAProxy
  ↓
Ingress Controller
  ↓
Service
  ↓
Pod
```

따라서 장애가 발생했을 때는 HAProxy만 보거나 Kubernetes만 보면 안 됩니다.

반드시 다음 순서로 함께 봐야 합니다.

```text
DNS
VIP
HAProxy frontend
ACL
backend
Health Check
Ingress
Service
EndpointSlice
Pod readiness
Application log
```

이 관점이 잡히면 Harbor 로그인 실패, Keycloak redirect loop, Grafana root_url 문제, ArgoCD gRPC 문제, Ceph Dashboard 400 Bad Request 같은 문제를 훨씬 체계적으로 분석할 수 있습니다.
