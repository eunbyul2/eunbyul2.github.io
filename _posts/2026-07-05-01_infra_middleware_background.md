## 1. 이 문서의 목적

RabbitMQ, Kafka, Redis는 모두 백엔드 시스템에서 자주 등장하지만, 세 제품은 같은 문제를 해결하기 위해 만들어진 도구가 아니다. RabbitMQ는 메시지를 안정적으로 전달하고 작업을 분산 처리하기 위한 메시지 브로커에 가깝고, Kafka는 대량 이벤트를 로그처럼 저장하고 여러 소비자가 독립적으로 읽게 하는 이벤트 스트리밍 플랫폼에 가깝다. Redis는 기본적으로 인메모리 데이터 저장소이며 캐시, 세션, 분산락, 간단한 메시징, 스트림 처리 등에 활용된다.

- 왜 API 서버와 DB만으로는 부족해지는가?
- 동기 처리와 비동기 처리의 차이는 무엇인가?
- 메시지 브로커는 왜 필요한가?
- 이벤트 브로커는 메시지 브로커와 무엇이 다른가?
- 캐시는 왜 DB 앞에 위치하는가?
- Kubernetes 환경에서 이런 미들웨어를 운영할 때 무엇을 봐야 하는가?

## 2. 기본 백엔드 구조와 한계

가장 단순한 서비스 구조는 다음과 같다.

```text
Client
  ↓
Frontend
  ↓
Backend API
  ↓
Database
```

작은 서비스에서는 이 구조만으로도 충분하다. 사용자가 요청을 보내면 API 서버가 비즈니스 로직을 처리하고, DB에 데이터를 저장하거나 조회한 뒤 응답을 반환한다. 문제는 서비스 규모가 커질수록 API 서버가 처리해야 하는 일이 점점 많아진다는 점이다.

예를 들어 쇼핑몰에서 주문 하나가 들어왔다고 가정해보자.

```text
주문 요청
  ↓
결제 승인
  ↓
주문 DB 저장
  ↓
재고 차감
  ↓
이메일 발송
  ↓
SMS/알림 발송
  ↓
포인트 적립
  ↓
로그 저장
  ↓
추천 시스템 이벤트 전달
  ↓
통계 시스템 반영
```

이 작업을 모두 하나의 API 요청 안에서 동기적으로 처리하면 사용자는 모든 작업이 끝날 때까지 기다려야 한다. 결제가 2초, 재고 차감이 1초, 이메일 발송이 3초, 배송 관련 처리가 2초 걸린다고 하면 사용자는 총 8초 가까이 기다리게 된다. 이메일 서버가 느려지면 전체 응답 시간은 30초 이상으로 늘어날 수도 있다.

더 큰 문제는 장애 전파다. 이메일 발송 서버가 느리거나 죽었는데 주문 API 전체가 실패하면 안 된다. 로그 시스템이 잠깐 장애가 났다고 해서 사용자의 주문 자체가 실패하면 안 된다. 즉, 모든 작업을 하나의 동기 흐름에 묶어두면 한 구성 요소의 장애가 전체 서비스 장애로 확산된다.

## 3. 동기 처리와 비동기 처리

### 3-1) 동기 처리

동기 처리는 요청을 보낸 쪽이 결과를 받을 때까지 기다리는 방식이다.

```text
API Server → Email Server
API Server ← Email Result
```

이 방식은 직관적이고 구현이 쉽다. 요청을 보내고 응답을 받으면 그 결과에 따라 다음 코드를 실행하면 된다. 하지만 다음과 같은 문제가 있다.

- 응답 시간이 길어진다. API 서버는 외부 시스템이 응답할 때까지 대기해야 한다. 
- 외부 시스템 장애가 API 서버에 직접 영향을 준다. 
- 여러 시스템에 동시에 데이터를 전달해야 할 경우 송신자가 모든 수신자의 상태와 실패 처리를 고려해야 한다. 
- 수신 대상이 늘어날수록 API 서버의 코드가 복잡해진다.

마이크로서비스 환경에서는 이 문제가 더 커진다. 예를 들어 하나의 게시글이 작성될 때 피드 서비스, 알림 서비스, 통계 서비스, 로그 서비스, 검색 색인 서비스가 모두 해당 사실을 알아야 한다면, 게시글 서비스가 모든 서비스에 직접 HTTP 요청을 보내는 구조는 관리하기 어렵다.

### 3-2) 비동기 처리

비동기 처리는 요청을 받은 즉시 핵심 작업만 처리하고, 오래 걸리거나 나중에 해도 되는 작업은 별도 시스템에 맡기는 방식이다.

```text
API Server
  ↓
Broker / Queue
  ↓
Worker
```

예를 들어 사용자가 회원가입을 하면 API 서버는 회원 정보를 DB에 저장한 뒤 즉시 응답한다. 환영 이메일 발송, SMS 발송, 이벤트 로그 저장은 큐에 넣어두고 워커가 나중에 처리한다.

```text
회원가입 요청
  ↓
API Server
  ├─ DB 저장
  └─ Email Task를 Queue에 등록
       ↓
     Worker가 이메일 발송
```

이 구조의 장점은 API 서버와 워커가 분리된다는 것이다. API 서버는 사용자의 요청을 빠르게 처리하는 데 집중하고, 워커는 시간이 걸리는 백그라운드 작업을 처리한다. 워커가 잠시 죽어도 큐에 작업이 남아 있으면 나중에 다시 처리할 수 있다.

## 4. 인메모리 백그라운드 작업의 한계

비동기 처리를 처음 배울 때는 애플리케이션 내부에서 백그라운드 작업을 실행하는 방식부터 떠올리기 쉽다. 예를 들어 Python의 `asyncio`, FastAPI의 background task, Node.js의 setTimeout, Go의 goroutine처럼 서버 프로세스 내부에서 별도 작업을 실행할 수 있다.

단일 서버에서 간단한 작업을 처리할 때는 이 방식도 충분하다. 그러나 운영 환경에서는 한계가 명확하다.

- 서버가 재시작되면 메모리에 있던 작업이 유실된다. 작업이 프로세스 메모리에만 존재하기 때문이다.
- 서버를 여러 대로 늘리면 각 서버가 자기 메모리의 작업만 처리한다. 특정 서버가 죽으면 그 서버 안의 작업만 사라질 수 있다.
- 작업이 폭증하면 웹 서버 메모리에 작업이 쌓여 API 서버 자체가 불안정해질 수 있다.
- 웹 서버와 워커를 독립적으로 확장하기 어렵다.

그래서 분산 시스템에서는 작업을 애플리케이션 프로세스 안에만 두지 않고, 별도의 메시지 브로커에 저장한다.

```text
Producer(API Server)
  ↓
Message Broker
  ↓
Consumer / Worker
```

브로커는 작업을 받아 저장하고, 워커가 처리할 수 있을 때 가져가도록 만든다. 워커가 죽으면 다른 워커가 이어서 처리할 수 있고, 워커 수를 늘려 작업 처리량을 확장할 수 있다.

## 5. 인프라 미들웨어란 무엇인가

인프라 미들웨어는 애플리케이션과 데이터베이스 사이, 또는 서비스와 서비스 사이에서 공통 기능을 제공하는 소프트웨어 계층이다. 직접적인 비즈니스 로직을 담당한다기보다, 서비스가 안정적이고 확장 가능하게 동작하도록 도와준다.

대표적인 인프라 미들웨어는 다음과 같다.

| 분류 | 대표 기술 | 주요 역할 |
|---|---|---|
| 메시지 브로커 | RabbitMQ, ActiveMQ, NATS | 작업 큐, 메시지 라우팅, 비동기 처리 |
| 이벤트 스트리밍 | Kafka, Pulsar, Redpanda | 이벤트 저장, 대량 스트림 처리, 데이터 파이프라인 |
| 인메모리 저장소 | Redis, Memcached | 캐시, 세션, 분산락, 빠른 임시 저장 |
| 검색/로그 분석 | Elasticsearch, OpenSearch, Loki | 로그 검색, 전문 검색, 장애 분석 |
| 모니터링 | Prometheus, Grafana, VictoriaMetrics | 메트릭 수집, 시각화, 알림 |
| 오브젝트 스토리지 | S3, MinIO, Ceph RGW | 파일, 이미지, 백업, 모델 아티팩트 저장 |
| 프록시/API Gateway | Nginx, HAProxy, Envoy, Kong | 라우팅, TLS 종료, 인증, 트래픽 제어 |

RabbitMQ, Kafka, Redis는 이 중에서도 백엔드 트래픽 흐름과 데이터 전달 구조에 직접 영향을 주는 핵심 미들웨어다.

## 6. 메시지, 이벤트, 작업의 차이

### 6-1) 메시지(Message)

메시지는 시스템 사이에서 전달되는 데이터 단위다. 예를 들어 이메일 발송 요청은 다음과 같은 메시지로 표현될 수 있다.

```json
{
  "type": "SEND_EMAIL",
  "user_id": 1001,
  "email": "user@example.com",
  "template": "welcome"
}
```

메시지는 보통 "이 일을 처리해줘"라는 명령에 가깝다. RabbitMQ 같은 메시지 큐에서는 메시지를 워커가 가져가서 처리하고, 처리가 끝나면 큐에서 제거하는 모델이 일반적이다.

### 6-2) 이벤트(Event)

이벤트는 시스템에서 이미 발생한 사실이다.

```json
{
  "event_type": "OrderCreated",
  "order_id": "ORD-1001",
  "user_id": 1001,
  "created_at": "2026-07-05T15:00:00+09:00"
}
```

이벤트는 "주문이 생성되었다"처럼 과거에 발생한 사실을 표현한다. Kafka 같은 이벤트 스트리밍 플랫폼에서는 이벤트를 로그처럼 저장해두고, 여러 소비자가 각자 필요한 시점에 읽는다.

### 6-3) 작업(Task)

작업은 실제로 수행해야 하는 단위다. 이메일 발송, 이미지 리사이징, 영상 인코딩, 정산 배치, 외부 API 호출 등이 작업에 해당한다. 작업 큐에서는 보통 하나의 작업을 하나의 워커가 가져가 처리한다.

```text
Task Queue
  ├─ send_email(user_id=1001)
  ├─ resize_image(file_id=88)
  └─ generate_invoice(order_id=ORD-1001)
```

## 7. 메시지 브로커와 이벤트 브로커

메시징 플랫폼은 크게 메시지 브로커와 이벤트 브로커로 나눠서 이해할 수 있다.

### 7-1) 메시지 브로커

메시지 브로커는 생산자와 소비자 사이에서 메시지를 받아 보관하고 전달한다. 메시지는 보통 큐에 들어가고, 소비자가 처리하면 삭제된다. 핵심은 안정적인 전달과 작업 분산이다.

```text
Producer
  ↓
Message Broker
  ↓
Consumer / Worker
```

RabbitMQ가 대표적인 메시지 브로커다. 메시지 브로커는 이메일 발송, 이미지 처리, 주문 후속 처리, 결제 후처리처럼 "작업을 안전하게 전달하고 처리해야 하는 경우"에 적합하다.

### 7-2) 이벤트 브로커

이벤트 브로커는 이벤트를 로그처럼 저장하고, 여러 소비자가 독립적으로 읽을 수 있게 한다. 메시지를 소비했다고 바로 삭제하지 않고, 설정된 보존 기간이나 용량 기준에 따라 유지한다.

```text
Producer
  ↓
Event Broker / Event Log
  ├─ Consumer A
  ├─ Consumer B
  └─ Consumer C
```

Kafka가 대표적인 이벤트 브로커 또는 이벤트 스트리밍 플랫폼이다. 이벤트 브로커는 로그 수집, 실시간 분석, 데이터 파이프라인, 여러 팀이 같은 이벤트를 독립적으로 소비해야 하는 구조에 적합하다.

### 7-3) 핵심 차이

메시지 브로커는 보통 메시지를 전달하고 처리되면 제거하는 구조다. 반면 이벤트 브로커는 이벤트를 저장해두고, 소비자가 어디까지 읽었는지를 오프셋으로 관리한다. 그래서 Kafka에서는 장애가 발생했을 때 특정 지점부터 다시 읽거나, 새로운 소비자가 과거 이벤트를 재처리하는 것이 가능하다.

## 8. Redis가 함께 비교되는 이유

Redis는 RabbitMQ나 Kafka와 출발점이 다르다. Redis는 원래 인메모리 데이터 구조 저장소다. 캐시, 세션 저장소, TTL 기반 임시 데이터, 분산락, 랭킹, 카운터 등에 많이 사용된다.

그런데 Redis는 List, Pub/Sub, Stream 같은 기능을 제공하기 때문에 간단한 큐나 메시징에도 사용할 수 있다. 그래서 실무에서는 다음과 같은 질문이 자주 나온다.

- 이미 Redis를 캐시로 쓰고 있는데, 간단한 작업 큐도 Redis로 처리해도 되는가?
- Redis Pub/Sub와 Kafka는 무엇이 다른가?
- Redis Streams는 Kafka를 대체할 수 있는가?
- Celery 브로커로 Redis와 RabbitMQ 중 무엇을 선택해야 하는가?

답은 상황에 따라 다르다. 단순하고 빠른 작업 큐나 캐시 중심 구조라면 Redis가 충분할 수 있다. 하지만 메시지 유실이 치명적이고 복잡한 라우팅, ACK, DLQ, 장기적인 운영 안정성이 필요하면 RabbitMQ가 더 적합하다. 대량 이벤트를 장기간 저장하고 여러 소비자가 독립적으로 재처리해야 하면 Kafka가 더 적합하다.

## 9. 인프라 및 플랫폼 엔지니어 관점에서 봐야 할 것

애플리케이션 개발자는 보통 "어떤 API를 호출하면 되는가"에 집중한다. 반면 인프라 및 플랫폼 엔지니어는 다음을 봐야 한다.

첫째, 배포 구조다. Kubernetes에서 Deployment로 올릴지 StatefulSet으로 올릴지, Pod 이름과 네트워크 ID가 유지되어야 하는지, Headless Service가 필요한지 확인해야 한다.

둘째, 저장소다. RabbitMQ와 Kafka는 메시지 또는 로그를 디스크에 저장할 수 있으므로 PVC, StorageClass, IOPS, latency가 중요하다. Redis는 메모리 기반이지만 RDB/AOF 같은 영속화 옵션을 사용할 경우 디스크 성능과 복구 시간이 중요해진다.

셋째, 고가용성이다. 브로커 노드가 죽었을 때 메시지가 유실되는지, 다른 노드가 승격되는지, 클러스터 재구성이 자동으로 되는지 확인해야 한다.

넷째, 장애 복구다. Consumer가 죽으면 메시지가 재전달되는지, Kafka consumer offset은 어디에 저장되는지, Redis master 장애 시 Sentinel 또는 Cluster가 어떻게 failover 하는지 알아야 한다.

다섯째, 관측성이다. 큐 적체량, consumer lag, broker disk usage, memory usage, connection count, rejected message, retry, DLQ 등을 모니터링해야 한다.

## 10. 정리

RabbitMQ, Kafka, Redis를 비교할 때 가장 먼저 잡아야 하는 기준은 제품명이 아니라 해결하려는 문제다.

RabbitMQ는 "작업을 안정적으로 전달하고 처리해야 하는 문제"를 해결한다. Kafka는 "대량 이벤트를 저장하고 여러 시스템이 독립적으로 읽어야 하는 문제"를 해결한다. Redis는 "데이터를 매우 빠르게 조회하거나 임시 상태를 저장해야 하는 문제"를 해결한다.
