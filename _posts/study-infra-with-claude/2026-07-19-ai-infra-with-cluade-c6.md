## 6장. 엔터프라이즈 환경으로 한 단계 더

5장에서는 Gateway API와 Argo Rollouts를 이용해 Blue/Green 배포까지 구성했다.

처음에는 여기까지 구현하고 나니 "이 정도면 실제 서비스와 크게 다르지 않겠는데?"라는 생각이 들었다. 하지만 6장을 읽기 시작하면서 아직 운영 환경에서는 고려해야 할 부분이 많이 남아 있다는 것을 알게 되었다.

서비스가 재시작되면 데이터는 어떻게 유지할까?

비밀번호는 Git 저장소에 넣어도 괜찮을까?

Blue/Green보다 더 안전한 배포 방법은 없을까?

그리고 지금까지 만든 시스템을 다른 사람이 한눈에 이해하려면 어떻게 문서화해야 할까?

이번 장은 새로운 기능을 만드는 것이 목적이 아니라, 지금까지 만든 서비스를 실제 운영 환경에 더 가깝게 다듬는 과정이었다.

특히 이번에도 바로 구현하기보다 Claude에게 먼저 질문을 던져 보기로 했다.

"어떤 선택지가 있는지."

"왜 그 방법을 추천하는지."

"다른 방법은 어떤 단점이 있는지."

이번 장은 기술을 배우는 것보다 **기술을 선택하는 과정**이 훨씬 인상 깊었다.

### 6.1 Pod가 다시 생성되면 Counter는 어디로 사라질까?

책에서는 가장 먼저 Counter 기능에서 발생하는 문제를 이야기한다.

현재 API는 요청이 들어올 때마다 ID를 하나씩 증가시키고 있었다.

예를 들면

```

1
2
3
4
5

```

처럼 증가하는 간단한 Counter이다.

처음에는 아무 문제 없어 보였다.

하지만 Kubernetes에서는 Pod가 영원히 살아있지 않는다.

Rolling Update가 발생해도 새로운 Pod가 만들어지고,

노드 장애가 발생해도 새로운 Pod가 생성되고,

HPA가 동작해도 Pod 개수가 계속 바뀐다.

그러면 메모리에 저장되어 있던 Counter는 모두 사라진다.

즉

```

Pod A
Counter = 154

↓

Pod 재생성

↓

Pod B
Counter = 0

```

이 되어 버린다.

처음에는 "그냥 메모리에 두면 되는 것 아닌가?"라고 생각했는데, Kubernetes에서는 오히려 메모리에 저장하는 것이 가장 위험한 방법이라는 것을 다시 느꼈다.

Pod는 언제든 없어질 수 있기 때문이다.-

### Redis를 쓰면 되는 거 아닌가?

Counter 문제를 해결하려면 가장 먼저 떠오르는 것이 Redis이다.

나도 처음에는 당연히 Redis를 사용할 것이라고 생각했다.

그래서 Claude에게 먼저 물어봤다.

> Pod 간 상태를 공유하려고 하는데 Redis를 쓰면 될까?

Claude는 가능하지만 지금이라면 **Valkey를 추천한다**고 답했다.

처음에는 Valkey라는 이름도 생소했다.

왜 Redis가 아니라 Valkey일까?

### Redis와 Valkey는 무엇이 다를까?

Valkey는 Redis를 기반으로 만들어진 오픈소스 프로젝트이다.

2024년 Redis의 라이선스가 BSD에서 RSAL(Redis Source Available License)로 변경되면서 많은 기업들이 오픈소스 대안을 찾기 시작했다.

그 결과 Linux Foundation 산하에서 Valkey 프로젝트가 시작되었다.

즉, Redis를 대체하기 위해 새롭게 만든 데이터베이스가 아니라,

기존 Redis 7.2를 기반으로 커뮤니티가 계속 발전시키는 프로젝트라고 이해하면 된다.

그래서 사용하는 명령어도 거의 동일하다.

```
SET
GET
INCR
EXPIRE
DEL
```

모두 그대로 사용할 수 있다.

애플리케이션 코드도 거의 수정하지 않는다.

이 부분이 꽤 흥미로웠다.

단순히 새로운 기술을 배우는 것이 아니라, 라이선스 정책 변화 하나가 오픈소스 생태계를 얼마나 크게 바꿀 수 있는지를 실제 사례로 볼 수 있었기 때문이다.

### 왜 데이터베이스가 아니라 Cache를 사용할까?

처음에는 Counter를 데이터베이스에 저장하면 되는 것 아닌가 하는 생각도 들었다.

하지만 Counter는 읽고 쓰는 횟수가 매우 많다.

요청이 들어올 때마다

```
SELECT

UPDATE

COMMIT
```

을 반복하게 된다.

이런 작업을 관계형 데이터베이스가 모두 처리하기에는 부담이 크다.

반면 Valkey는 메모리 기반으로 동작하기 때문에

```
INCR counter
```

한 줄만으로 원자적으로 값을 증가시킬 수 있다.

여러 Pod가 동시에 요청을 보내더라도 중복 없이 안전하게 증가한다.

실제로 책에서도 `INCR` 명령 하나로 ID를 생성하는 구조를 사용한다.

여기서 다시 느낀 점은,

"모든 데이터를 데이터베이스에 저장하는 것이 정답은 아니다."

용도에 따라 가장 적합한 저장소가 따로 존재한다는 것이다.

### 왜 Deployment가 아니라 StatefulSet일까?

Valkey를 구성하면서 또 하나 궁금했던 점이 있었다.

왜 StatefulSet을 사용할까?

보통 애플리케이션은 Deployment를 사용한다.

하지만 Valkey는 상태(State)를 가지고 있는 서비스이다.

즉,

- 데이터가 유지되어야 하고
- Pod 이름이 바뀌면 안 되고
- PersistentVolume도 계속 유지되어야 한다.

Deployment는 Pod가 다시 만들어질 때

```
valkey-6fd8f4

↓

valkey-8b9d31
```

처럼 이름이 계속 변경된다.

반면 StatefulSet은

```
valkey-0

valkey-1

valkey-2
```

처럼 고정된 이름을 유지한다.

PVC도 함께 유지되기 때문에 Pod가 재생성되어도 같은 데이터를 그대로 사용할 수 있다.

이번 장을 보면서 Deployment와 StatefulSet의 차이를 다시 한번 명확하게 이해할 수 있었다.

둘 다 Pod를 관리하는 리소스이지만,

Deployment는 **Stateless 서비스**를,

StatefulSet은 **Stateful 서비스**를 위한 리소스라는 점이 핵심이다.

### 여기까지 구현하고 나니

Valkey를 추가했다고 해서 눈에 보이는 기능이 크게 늘어난 것은 아니다.

사용자는 여전히 같은 API를 호출한다.

하지만 내부 구조는 완전히 달라졌다.

이전에는 Pod 하나가 모든 상태를 들고 있었다면,

이제는 상태를 Pod 밖으로 분리했다.

애플리케이션은 Stateless하게 유지하고,

상태는 Valkey가 책임진다.

처음에는 단순히 Counter 하나를 저장하기 위해 Valkey를 사용하는 것이 과한 것처럼 느껴졌다.

하지만 이후 Secret Manager, Canary 배포까지 이어지는 내용을 보니 6장은 "기능 추가"가 아니라 "운영 환경을 만드는 과정"이라는 것을 조금씩 이해하게 되었다.

다음으로는 Git 저장소에 Secret을 보관하지 않기 위해 GCP Secret Manager와 Workload Identity를 적용해 보려고 한다.

### 6.2 Kubernetes Secret이면 충분한 줄 알았다

Valkey를 적용하면서 Pod가 여러 개로 늘어나도 상태를 안전하게 공유할 수 있게 되었다.

하지만 곧바로 또 다른 고민이 생겼다.

애플리케이션에서 사용하는 API Key나 비밀번호는 어디에 저장해야 할까?

처음에는 별생각 없이 Kubernetes Secret을 사용하면 되는 줄 알았다.

실제로 Kubernetes를 처음 공부할 때도 Secret 리소스를 만들어 환경변수로 주입하는 예제를 가장 많이 봤기 때문이다.

그런데 Claude에게 운영 환경에서는 어떤 방식을 많이 사용하는지 물어보니 생각보다 선택지가 정말 많았다.

- Kubernetes Secret
- Sealed Secrets
- External Secrets Operator
- Google Secret Manager + CSI Driver

처음에는 이름도 생소했고 다 비슷해 보였다.

그래서 하나씩 비교해 보기 시작했다.

### Kubernetes Secret은 정말 안전할까?

처음에는 이름이 Secret이니까 당연히 암호화되어 저장되는 줄 알았다.

그런데 찾아보니 Kubernetes Secret은 기본적으로 Base64 인코딩만 적용된다.

즉,

```yaml
data:
  password: YWRtaW4xMjM=
```

이렇게 저장되는데 Base64는 암호화가 아니라 단순한 인코딩이다.

디코딩하면 원래 값을 바로 확인할 수 있다.

물론 etcd 암호화나 RBAC 같은 보안 기능을 추가할 수는 있지만, Secret 자체가 안전한 저장소라는 의미는 아니었다.

이 부분은 조금 의외였다.

"Secret"이라는 이름 때문에 막연히 안전하다고 생각했던 것 같다.

### 그러면 Sealed Secrets가 정답일까?

다음으로 비교한 것은 Sealed Secrets였다.

GitOps 환경에서는 이미 많이 사용하는 방식이라 처음에는 이게 가장 좋은 선택처럼 보였다.

Secret을 암호화한 상태로 Git 저장소에 올릴 수 있기 때문에 민감한 정보를 평문으로 관리하지 않아도 된다.

하지만 GCP 환경을 기준으로 생각해 보니 조금 다른 문제가 있었다.

Secret이 변경될 때마다 다시 암호화해서 Git에 반영해야 하고, 결국 Secret도 Git을 통해 관리하게 된다.

운영 환경에서 Secret을 자주 변경해야 한다면 생각보다 번거로운 작업이 될 수도 있겠다는 생각이 들었다.

### 결국 선택한 것은 Google Secret Manager

Claude는 GKE 환경이라면 Google Secret Manager를 사용하는 것이 가장 자연스럽다고 추천했다.

처음에는 "Secret을 Kubernetes 밖에서 관리한다고?"라는 생각이 들었는데, 오히려 그게 가장 큰 장점이었다.

애플리케이션은 Secret의 실제 값을 알 필요가 없다.

필요한 순간에만 Secret Manager에서 읽어 오면 된다.

즉, Git에도 없고, YAML에도 없고, Container Image에도 없다.

Secret은 Google Secret Manager 한 곳에서만 관리된다.

실제로 운영 환경이라면 이런 방식이 훨씬 안전하겠다는 생각이 들었다.

### Workload Identity는 왜 필요할까?

Secret Manager를 사용하려면 애플리케이션이 Google Cloud에 접근할 수 있어야 한다.

처음에는 Service Account Key(JSON)를 Pod 안에 넣어야 하는 줄 알았다.

예전 예제들도 대부분 그런 방식이었다.

그런데 책에서는 Workload Identity를 사용한다.

처음에는 조금 생소했는데 구조를 이해하고 나니 왜 최근에는 이 방식을 많이 사용하는지 알 수 있었다.

Pod 안에 Key 파일을 넣는 대신,

Kubernetes ServiceAccount와 Google ServiceAccount를 연결해 주는 방식이다.

그러면 Pod는 별도의 인증 키를 가지고 있지 않아도 된다.

필요할 때 Google이 자동으로 인증을 처리해 준다.

민감한 Key 파일 자체가 없어지는 셈이다.

보안 측면에서도 훨씬 깔끔한 구조라는 생각이 들었다.

### CSI Driver가 하는 일

Workload Identity까지 연결한 뒤에는 Secret Store CSI Driver를 설치했다.

처음에는 이름만 보고 스토리지 관련 기능인 줄 알았다.

그런데 실제 역할은 Secret을 Pod 안으로 전달하는 것이었다.

애플리케이션 입장에서는 Secret Manager를 직접 호출하지 않는다.

Pod가 실행되면 CSI Driver가 Secret Manager에서 값을 읽어 파일 형태로 마운트해 준다.

즉,

```
Google Secret Manager

↓

CSI Driver

↓

Pod 내부 파일

↓

애플리케이션
```

이런 흐름으로 동작한다.

덕분에 애플리케이션 코드도 크게 수정하지 않았다.

파일만 읽으면 되기 때문이다.

실습하면서 CSI Driver가 단순히 디스크를 연결하는 기능만 있는 것이 아니라 다양한 외부 시스템과 연결하는 역할도 한다는 것을 처음 알게 되었다.

### 직접 적용해 보니

SecretProviderClass를 생성하고 Rollout에도 볼륨을 추가한 뒤 애플리케이션에서 파일을 읽도록 수정했다.

생각보다 YAML 수정은 많지 않았다.

하지만 구조는 꽤 크게 달라졌다.

기존에는 Secret을 Kubernetes 안에서 관리했다면,

이제는 Secret Manager가 진짜 Secret 저장소가 되고 Kubernetes는 필요한 순간에만 가져다 사용하는 구조가 되었다.

처음에는 단순히 Secret 저장 위치만 바뀌는 줄 알았는데, 실제로는 운영 방식 자체가 달라지는 경험이었다.

특히 Git 저장소 어디에도 Secret 값이 남지 않는다는 점이 가장 마음에 들었다.

이 부분은 앞으로 GitOps 환경을 운영하게 된다면 꼭 다시 참고하고 싶은 내용이었다.

### 6.4 AI도 프로젝트를 이해할 수 있어야 한다

Canary까지 적용하면서 문득 이런 생각이 들었다.

지금까지는 기능을 하나씩 추가하는 데 집중했다.

Valkey를 붙이고, Secret Manager를 적용하고, Canary 배포도 구성했다.

그런데 시간이 지나서 다시 이 프로젝트를 보게 된다면?

혹은 다른 개발자가 이 저장소를 처음 열어본다면?

그리고 사람이 아니라 Claude가 이 프로젝트를 다시 분석해야 한다면?

과연 지금 구조를 바로 이해할 수 있을까?

이번 절은 지금까지의 내용과 조금 다른 이야기였다.

새로운 기술을 설치하는 것이 아니라 **AI와 사람이 같은 아키텍처를 바라볼 수 있도록 만드는 과정**이었다.

### Claude도 프로젝트를 처음 보면 아무것도 모른다

처음에는 Claude가 저장소만 보면 구조를 금방 이해할 줄 알았다.

하지만 조금만 프로젝트가 커져도 생각보다 쉽지 않았다.

예를 들어 현재 프로젝트에는

- Kubernetes Manifest
- Argo Rollouts
- Gateway API
- Secret Manager
- Valkey
- GitOps

같은 기술들이 함께 존재한다.

파일도 수십 개가 된다.

사람도 처음 보는 프로젝트를 바로 이해하기 어려운데 AI도 마찬가지다.

파일을 하나씩 읽으면서 추론은 할 수 있지만,

프로젝트 전체 구조를 항상 정확하게 이해하는 것은 또 다른 문제였다.

그래서 Claude에게도 프로젝트를 설명하는 문서가 필요했다.

### CLAUDE.md와 architecture.md는 무엇이 다를까?

처음에는 둘 다 비슷한 문서라고 생각했다.

하지만 목적이 완전히 달랐다.

### CLAUDE.md

Claude에게 작업 규칙을 알려주는 문서이다.

예를 들면

- 어떤 명령을 사용해야 하는지
- 코드를 어떻게 작성해야 하는지
- 어떤 방식으로 답변해야 하는지
- 프로젝트에서 지켜야 하는 규칙

처럼 **행동 방식**을 정의한다.

즉,

"어떻게 일할 것인가"

를 설명하는 문서이다.

---

반면

### architecture.md

이 문서는

현재 시스템이

어떻게 구성되어 있는지를 설명한다.

예를 들면

```
User

↓

Gateway API

↓

Argo Rollouts

↓

Application

↓

Valkey

↓

Google Secret Manager
```

같은 구조를 설명한다.

즉,

"현재 시스템이 어떻게 생겼는가"

를 설명하는 문서이다.

처음에는 둘 다 프로젝트 설명 문서인 줄 알았는데,

실제로는 역할이 완전히 달랐다.

### 왜 architecture.md가 필요할까?

이 부분은 개인적으로 이번 장에서 가장 인상 깊었다.

보통 프로젝트를 하다 보면

README 정도만 작성하는 경우가 많다.

하지만 README는 설치 방법이나 실행 방법을 설명하는 문서인 경우가 대부분이다.

반면 architecture.md는

왜 이런 구조를 선택했는지,

컴포넌트가 어떻게 연결되는지,

데이터가 어떤 흐름으로 이동하는지를 설명한다.

즉,

프로젝트의 설계도를 문서로 남기는 것이다.

사람도 새로운 프로젝트를 맡으면

먼저 아키텍처 다이어그램부터 보는 경우가 많다.

Claude도 마찬가지라는 점이 흥미로웠다.

### AI도 Context가 중요하다

Claude Code를 사용하면서 가장 많이 느낀 것은

AI의 성능은 Context에 크게 영향을 받는다는 점이었다.

예를 들어

"Secret Manager를 추가해줘."

라고 요청하는 것보다

```
현재 프로젝트는

GitOps를 사용하고

ArgoCD가 자동 동기화를 수행하며

Gateway API를 사용하고

Secret은 CSI Driver를 통해 주입한다.
```

라는 Context를 먼저 제공하면

답변 품질이 훨씬 좋아진다.

이번 장에서 architecture.md를 만드는 이유도 결국 이것이었다.

AI가 프로젝트의 현재 상태를 이해한 뒤 작업하도록 만드는 것이다.

단순히 문서를 하나 더 만드는 것이 아니라,

앞으로 AI와 협업하기 위한 기반을 만드는 과정이라는 점이 인상 깊었다.

### 결국 문서도 하나의 코드다

예전에는 문서를 개발이 끝난 뒤 작성하는 부가 작업이라고 생각했다.

하지만 이번 장을 공부하면서 생각이 조금 바뀌었다.

문서가 최신 상태가 아니면

사람도 잘못 이해하고,

AI도 잘못 이해한다.

결국 잘못된 이해는 잘못된 코드로 이어질 수 있다.

그래서 architecture.md는

문서를 위한 문서가 아니라,

프로젝트를 올바르게 이해하기 위한 코드의 일부라는 생각이 들었다.

앞으로 프로젝트를 진행하게 된다면

README만 관리하는 것이 아니라

아키텍처 문서도 함께 관리하는 습관을 들여야겠다고 느꼈다.

---

### 6.5 AI에게도 실수를 줄이는 장치가 필요하다

이번 장의 마지막은 조금 의외였다.

새로운 기능을 추가하는 내용이 아니라,

Claude가 실수하지 않도록 Guardrail을 만드는 내용이었다.

처음에는 "AI가 실수하면 내가 수정하면 되는 것 아닌가?" 라고 생각했다.

하지만 프로젝트 규모가 커질수록 같은 실수를 반복해서 수정하는 것이 더 큰 비용이라는 것을 알게 되었다.

### Guardrail은 AI를 제한하는 것이 아니다

Guardrail이라는 단어를 처음 들었을 때는

AI의 기능을 막는 장치라고 생각했다.

하지만 실제 의미는 조금 달랐다.

AI가 항상 프로젝트의 규칙을 지키도록 만드는 안전장치였다.

예를 들어

- GKE Managed CSI Driver를 사용해야 한다.
- Git Push가 끝나기 전에는 Rollout을 삭제하면 안 된다.
- 프로젝트에서는 Gateway API를 사용한다.

같은 규칙을 미리 알려 두는 것이다.

그러면 Claude는 매번 같은 실수를 반복하지 않는다.

### 이번 장을 마치며

6장은 지금까지 실습했던 내용 중 가장 운영 환경에 가까운 장이었다.

Valkey를 추가하면서 상태를 Pod 밖으로 분리했고,

Secret Manager를 적용하면서 Secret 관리 방식도 개선했다.

Canary를 통해 점진적 배포를 경험했고,

architecture.md를 작성하면서 AI와 사람 모두 프로젝트를 이해할 수 있는 구조를 만들었다.

마지막으로 Guardrail을 통해 반복되는 실수를 줄이는 방법까지 살펴봤다.

처음에는 각각 독립적인 기술처럼 보였다.

하지만 끝까지 공부하고 나니 모두 같은 방향을 바라보고 있었다.

**"서비스를 만드는 것"이 아니라 "서비스를 안정적으로 운영하는 것."**

이번 장을 통해 가장 크게 느낀 점도 바로 이것이었다.

기능 하나를 추가하는 것보다,

그 기능을 오래 운영할 수 있는 구조를 만드는 것이 훨씬 어렵고 중요하다는 사실을 다시 한번 느낄 수 있었다.

다음 장에서는 이러한 운영 환경을 바탕으로 더 실제 서비스에 가까운 GitOps 운영 방식을 계속 살펴볼 예정이다.
