# Amazon EBS 전체 정리

이 문서는 Amazon EBS 공식 문서 전체 흐름을 기준으로, EBS를 처음 공부하는 사람이 개념, 구조, 운영, 백업, 보안, 성능, 모니터링까지 한 번에 이해할 수 있도록 정리한 문서이다.

EBS는 단순히 EC2에 붙이는 디스크가 아니라, AWS에서 제공하는 **네트워크 기반 블록 스토리지 서비스**다. EC2 인스턴스에 연결하면 운영체제 입장에서는 로컬 디스크처럼 보이지만, 실제로는 AWS의 스토리지 인프라에 존재하며 EC2와 네트워크 경로를 통해 I/O를 처리한다.

---

## 1. Amazon EBS 개요

### 1-1) Amazon EBS란?

Amazon EBS는 `Amazon Elastic Block Store`의 약자다.

Amazon EC2 인스턴스와 함께 사용할 수 있는 확장 가능한 고성능 블록 스토리지 서비스다. 쉽게 말하면 EC2에 붙여서 사용하는 디스크다.

하지만 일반 PC의 내부 디스크와는 다르다. EBS는 EC2 인스턴스와 별도로 존재하는 네트워크 기반 스토리지이기 때문에 EC2 인스턴스를 중지하거나 종료하더라도 설정에 따라 데이터가 유지될 수 있다.

EBS에서 핵심적으로 다루는 리소스는 크게 두 가지다.

- EBS Volume
- EBS Snapshot

EBS Volume은 EC2 인스턴스에 연결해서 사용하는 블록 스토리지다. 볼륨을 인스턴스에 연결하면 운영체제에서는 하드디스크나 SSD처럼 인식한다.

EBS Snapshot은 특정 시점의 EBS 볼륨 백업이다. 스냅샷을 이용하면 장애 복구, 백업, 다른 가용 영역 또는 리전으로의 데이터 이동이 가능하다.

### 1-2) 블록 스토리지란?

스토리지는 크게 다음과 같이 나눌 수 있다.

- 블록 스토리지
- 파일 스토리지
- 오브젝트 스토리지

EBS는 블록 스토리지다.

블록 스토리지는 데이터를 일정한 크기의 블록 단위로 저장한다. 운영체제는 이 블록 디바이스 위에 파일시스템을 만들고, 애플리케이션은 파일시스템을 통해 데이터를 읽고 쓴다.

예를 들면 Linux에서 다음과 같은 흐름으로 사용한다.

```text
EBS Volume
↓
/dev/nvme1n1 또는 /dev/xvdf
↓
파일시스템 생성: XFS, EXT4 등
↓
마운트: /data, /var/lib/mysql 등
↓
애플리케이션에서 사용
```

블록 스토리지는 데이터베이스, 파일시스템, 부트 디스크처럼 낮은 지연 시간과 직접적인 디스크 접근이 필요한 경우에 적합하다.

### 1-3) EBS와 EC2의 관계

EBS는 EC2와 함께 사용하는 스토리지다.

EC2 인스턴스에는 보통 루트 볼륨이 필요하다. 루트 볼륨은 운영체제가 설치된 디스크다. 대부분의 EC2 인스턴스는 EBS 기반 AMI를 사용하고, 이 경우 루트 볼륨도 EBS로 생성된다.

EBS는 다음과 같은 방식으로 사용된다.

- EC2 루트 볼륨
- 애플리케이션 데이터 볼륨
- 데이터베이스 저장소
- 로그 저장소
- 백업 전 임시 저장소
- Kubernetes PVC의 실제 스토리지

중요한 점은 EBS 볼륨과 EC2 인스턴스는 **같은 가용 영역(AZ)** 안에 있어야 연결할 수 있다는 것이다.

예를 들어 EC2가 `ap-northeast-2a`에 있으면 EBS 볼륨도 `ap-northeast-2a`에 있어야 한다. 다른 AZ의 EBS 볼륨은 직접 연결할 수 없다.

### 1-4) EBS와 인스턴스 스토어의 차이

EC2에서 사용할 수 있는 디스크에는 EBS와 인스턴스 스토어가 있다.

EBS는 EC2와 분리된 영구 블록 스토리지다. 인스턴스를 중지하거나 재시작해도 데이터가 유지될 수 있다.

인스턴스 스토어는 EC2 호스트 서버에 물리적으로 붙어 있는 임시 스토리지다. 성능이 빠를 수 있지만 인스턴스가 중지되거나 종료되면 데이터가 사라질 수 있다.

| 구분 | EBS | Instance Store |
|---|---|---|
| 성격 | 네트워크 기반 블록 스토리지 | EC2 호스트의 로컬 임시 스토리지 |
| 데이터 지속성 | 인스턴스 수명과 독립적 | 인스턴스 수명에 종속적 |
| 스냅샷 | 가능 | 직접 EBS Snapshot 불가 |
| 연결/분리 | 가능 | 인스턴스에 고정 |
| 주요 용도 | OS, DB, 일반 데이터 | 캐시, 임시 파일, 고성능 임시 처리 |

### 1-5) EBS의 주요 특징

EBS의 핵심 특징은 다음과 같다.

- EC2에 연결해서 사용하는 블록 스토리지
- 인스턴스와 독립적인 데이터 지속성 제공
- 다양한 볼륨 타입 제공
- 볼륨 크기와 성능 조정 가능
- 스냅샷 기반 백업 및 복구 가능
- KMS 기반 암호화 지원
- CloudWatch 기반 모니터링 지원
- Data Lifecycle Manager로 백업 자동화 가능
- Nitro 기반 인스턴스에서 NVMe 디바이스로 노출
- 일부 io1/io2 볼륨에서 Multi-Attach 지원

## 2. Amazon EBS 설정과 기본 보안 준비

### 2-1) AWS 계정과 루트 사용자

EBS를 사용하려면 AWS 계정이 필요하다. 계정을 생성하면 루트 사용자가 만들어진다.

루트 사용자는 계정의 모든 권한을 가지므로 일상 작업에 사용하면 안 된다. 루트 사용자는 결제, 계정 설정, 일부 보안 작업처럼 꼭 필요한 경우에만 사용해야 한다.

기본 보안 원칙은 다음과 같다.

- 루트 사용자 MFA 활성화
- 일상 작업은 IAM Identity Center 또는 IAM 사용자/역할로 수행
- 최소 권한 원칙 적용
- 관리자 권한은 필요한 사용자에게만 부여
- 액세스 키 장기 사용 지양

### 2-2) 관리자 사용자 생성

EBS를 관리하려면 EC2, EBS, KMS, CloudWatch, IAM 관련 권한이 필요할 수 있다.

운영 환경에서는 사용자 개인에게 무제한 권한을 주는 것보다 역할 기반 권한을 구성하는 것이 좋다.

예를 들어 다음 역할을 분리할 수 있다.

- 인프라 관리자
- 백업 관리자
- 보안 관리자
- 모니터링 담당자
- 개발자

### 2-3) EBS 암호화용 KMS 키 준비

EBS 암호화는 AWS KMS 키를 사용한다.

기본적으로 AWS 관리형 키 `aws/ebs`를 사용할 수 있다. 하지만 조직에서 키 정책, 키 회전, 감사, 접근 제어를 직접 관리해야 한다면 고객 관리형 KMS 키를 사용하는 것이 좋다.

고객 관리형 키를 사용하면 다음을 직접 제어할 수 있다.

- 키 정책
- IAM 권한
- 키 회전
- 키 비활성화
- CloudTrail 감사
- 특정 계정 또는 역할에 대한 사용 권한

### 2-4) 스냅샷 퍼블릭 액세스 차단

EBS 스냅샷은 특정 계정과 공유하거나 공개 공유할 수 있다.

스냅샷에는 실제 데이터가 들어 있으므로 공개 공유는 매우 위험하다. 실수로 공개 공유되면 민감한 데이터가 외부에 노출될 수 있다.

이를 방지하기 위해 EBS 스냅샷에 대한 퍼블릭 액세스 차단을 활성화할 수 있다.

주요 모드는 다음과 같다.

- 모든 퍼블릭 액세스 차단
- 새 퍼블릭 공유 차단

운영 환경에서는 특별한 이유가 없다면 퍼블릭 스냅샷 공유를 차단하는 것이 안전하다.

## 3. Amazon EBS 볼륨

### 3-1) EBS 볼륨이란?

EBS 볼륨은 EC2 인스턴스에 연결할 수 있는 내구성 있는 블록 스토리지 디바이스다.

운영체제에서는 일반 디스크처럼 인식되며, 파일시스템을 만들고 마운트해서 사용할 수 있다.

EBS 볼륨은 다음 용도로 자주 사용된다.

- EC2 루트 디스크
- 애플리케이션 데이터 저장소
- 데이터베이스 저장소
- 로그 저장소
- 백업 전 임시 스토리지
- 컨테이너 워크로드의 Persistent Volume

### 3-2) EBS 볼륨의 데이터 가용성

EBS 볼륨은 생성된 가용 영역 내부에서 자동으로 복제된다. 이는 단일 하드웨어 구성 요소 장애로 인한 데이터 손실을 줄이기 위한 구조다.

다만 EBS 볼륨은 특정 AZ에 속한다. 다른 AZ의 EC2 인스턴스에 직접 연결할 수 없다.

다른 AZ에서 사용하려면 보통 다음 절차를 거친다.

```text
기존 EBS 볼륨
↓
스냅샷 생성
↓
다른 AZ에 새 볼륨 생성
↓
새 EC2에 연결
```

### 3-3) EBS 볼륨의 데이터 지속성

EBS 볼륨은 EC2 인스턴스 수명과 독립적으로 유지될 수 있다.

인스턴스를 종료해도 EBS 볼륨은 삭제되지 않도록 설정할 수 있다. 단, 루트 볼륨은 기본적으로 인스턴스 종료 시 삭제되도록 설정되는 경우가 많다.

중요 설정은 `DeleteOnTermination`이다.

- true: 인스턴스 종료 시 볼륨 삭제
- false: 인스턴스 종료 후에도 볼륨 유지

데이터 볼륨은 보통 종료 시 삭제하지 않도록 설정한다. 루트 볼륨도 복구 목적이 있다면 삭제되지 않도록 조정할 수 있다.

### 3-4) EBS 볼륨의 암호화

EBS 볼륨은 암호화할 수 있다.

EBS 암호화는 다음 데이터를 보호한다.

- 볼륨 내부 데이터
- 볼륨과 인스턴스 사이에서 이동하는 데이터
- 볼륨에서 생성한 스냅샷
- 암호화된 스냅샷에서 생성한 새 볼륨

암호화는 AWS KMS 키를 사용한다.

### 3-5) EBS 볼륨의 스냅샷

EBS 볼륨은 스냅샷으로 백업할 수 있다.

스냅샷은 특정 시점의 볼륨 상태를 저장한다. 이후 이 스냅샷에서 새 볼륨을 만들 수 있다.

스냅샷은 증분 방식이므로 매번 전체 데이터를 저장하지 않고, 이전 스냅샷 이후 변경된 블록을 저장한다.

### 3-6) EBS 볼륨의 유연성

EBS는 Elastic Volumes 기능을 통해 운영 중에도 다음 변경을 지원한다.

- 볼륨 크기 증가
- 볼륨 타입 변경
- IOPS 조정
- Throughput 조정

단, 볼륨 크기는 줄일 수 없다. 줄여야 하는 경우 새 작은 볼륨을 만들고 파일 수준 복사를 해야 한다.

## 4. EBS 볼륨 유형

### 4-1) 볼륨 유형을 왜 구분해야 하는가?

EBS 볼륨 유형은 성능 특성과 비용이 다르다.

워크로드에 맞지 않는 볼륨 유형을 선택하면 비용을 낭비하거나 성능 문제가 발생한다.

큰 분류는 다음과 같다.

- SSD 기반 볼륨
- HDD 기반 볼륨
- 이전 세대 Magnetic 볼륨

SSD 기반 볼륨은 작은 랜덤 I/O와 트랜잭션 워크로드에 적합하다.

HDD 기반 볼륨은 대용량 순차 I/O와 처리량 중심 워크로드에 적합하다.

### 4-2) gp3

`gp3`는 범용 SSD 볼륨이다.

현재 대부분의 일반 워크로드에서 기본 선택지로 볼 수 있다.

특징은 다음과 같다.

- 기본 3,000 IOPS 제공
- 기본 125MiB/s 처리량 제공
- 볼륨 크기와 성능을 독립적으로 설정 가능
- gp2보다 예측 가능한 성능 제공
- BurstBalance 개념 없음
- 개발/테스트, 부트 볼륨, 일반 애플리케이션, 중간 규모 DB에 적합

예를 들어 작은 용량의 볼륨이라도 IOPS와 Throughput을 별도로 늘릴 수 있다. gp2는 크기를 늘려야 성능이 같이 올라가는 구조였기 때문에 gp3가 더 유연하다.

### 4-3) gp2

`gp2`는 이전 세대 범용 SSD 볼륨이다.

특징은 다음과 같다.

- 볼륨 크기에 따라 기준 IOPS 결정
- GiB당 3 IOPS
- 작은 볼륨은 기준 IOPS가 낮음
- 일정 조건에서 3,000 IOPS까지 버스트 가능
- BurstBalance 고갈 시 기준 성능으로 제한

예를 들어 100GiB gp2 볼륨은 기준 성능이 약 300 IOPS다. 버스트 크레딧이 있을 때는 더 높은 성능을 낼 수 있지만, 크레딧이 고갈되면 성능이 떨어진다.

운영 환경에서는 gp2를 gp3로 마이그레이션하는 경우가 많다.

### 4-4) io1

`io1`은 프로비저닝된 IOPS SSD 볼륨이다.

특징은 다음과 같다.

- 높은 IOPS를 직접 지정 가능
- 성능 일관성이 중요한 워크로드에 적합
- I/O 집약적 데이터베이스에 적합
- Multi-Attach 지원 가능

io1은 gp 계열보다 비용이 높지만, IOPS 성능을 안정적으로 확보해야 하는 경우 사용한다.

### 4-5) io2 및 io2 Block Express

`io2`는 프로비저닝된 IOPS SSD 볼륨이며, io1보다 내구성과 성능 특성이 더 강화된 유형이다.

io2 Block Express는 고성능 스토리지 아키텍처를 기반으로 하며, Nitro 기반 인스턴스에서 매우 높은 IOPS와 낮은 지연 시간이 필요한 워크로드에 적합하다.

적합한 워크로드는 다음과 같다.

- Oracle
- SAP HANA
- Microsoft SQL Server
- 대규모 PostgreSQL/MySQL
- 고성능 분석 워크로드
- 미션 크리티컬 DB

### 4-6) st1

`st1`은 처리량 최적화 HDD 볼륨이다.

특징은 다음과 같다.

- IOPS보다 Throughput 중심
- 대용량 순차 I/O에 적합
- 빅데이터, 로그 처리, 데이터 웨어하우스, EMR, ETL에 적합
- 부트 볼륨으로 사용 불가
- 작은 랜덤 I/O에는 부적합

st1은 많은 양의 데이터를 순차적으로 읽고 쓰는 워크로드에 적합하다. 데이터베이스 트랜잭션처럼 작은 랜덤 I/O가 많은 워크로드에는 맞지 않는다.

### 4-7) sc1

`sc1`은 콜드 HDD 볼륨이다.

특징은 다음과 같다.

- 저비용 HDD 볼륨
- 자주 접근하지 않는 데이터에 적합
- 대용량 순차 콜드 데이터에 적합
- 부트 볼륨으로 사용 불가
- 성능보다 비용 절감이 중요한 경우 사용

### 4-8) Magnetic standard

Magnetic 또는 standard 볼륨은 이전 세대 볼륨이다.

특징은 다음과 같다.

- 낮은 성능
- 낮은 일관성
- 자주 접근하지 않는 소규모 데이터에만 적합
- 신규 설계에서는 일반적으로 현재 세대 볼륨 사용 권장

### 4-9) 볼륨 유형 선택 기준

| 워크로드 | 권장 볼륨 |
|---|---|
| 일반 서버, 부트 볼륨, 개발/테스트 | gp3 |
| 일반 DB, 중간 규모 애플리케이션 | gp3, 필요 시 io2 |
| 고성능 DB | io2 |
| 미션 크리티컬 DB | io2 Block Express |
| 빅데이터 순차 처리 | st1 |
| 자주 접근하지 않는 대용량 데이터 | sc1 |
| 신규 권장 안 함 | standard |

## 5. EBS 볼륨 제약 조건

### 5-1) 볼륨 크기 제한

EBS 볼륨 크기는 볼륨 유형, 파티셔닝 방식, 파일시스템에 따라 제한을 받는다.

일반적으로 대용량 볼륨을 사용할 때는 다음을 확인해야 한다.

- EBS 볼륨 유형의 최대 크기
- MBR 또는 GPT 파티션 방식
- OS가 인식 가능한 크기
- 파일시스템 최대 크기

### 5-2) MBR 제한

MBR 파티션 방식은 2TiB 제한이 있다.

2TiB를 초과하는 볼륨을 제대로 사용하려면 GPT를 사용해야 한다.

특히 Windows 비부트 볼륨이나 Linux 데이터 볼륨에서 2TiB를 넘는 경우 GPT 여부를 반드시 확인해야 한다.

### 5-3) GPT

GPT는 대용량 볼륨을 지원하는 파티셔닝 방식이다.

2TiB를 초과하는 EBS 볼륨은 GPT를 사용하는 것이 일반적이다.

부트 볼륨에서 GPT를 사용하려면 OS와 부트 모드가 이를 지원해야 한다.

### 5-4) 파일시스템 제한

EBS가 특정 크기를 지원하더라도 파일시스템이 해당 크기를 모두 지원하지 못할 수 있다.

예를 들어 EXT4, XFS, NTFS는 각각 구현상 지원 가능한 최대 크기가 다르다.

따라서 대용량 볼륨을 설계할 때는 EBS 제한뿐 아니라 파일시스템 제한도 확인해야 한다.

## 6. EBS 볼륨과 NVMe

### 6-1) Nitro 기반 인스턴스와 NVMe

Nitro 기반 EC2 인스턴스에서는 EBS 볼륨이 NVMe 디바이스로 표시된다.

예를 들어 콘솔에서 `/dev/sdf`로 연결했더라도 OS에서는 다음처럼 보일 수 있다.

```bash
/dev/nvme1n1
```

이는 정상 동작이다.

### 6-2) 디바이스 이름이 바뀔 수 있는 이유

NVMe 디바이스 이름은 부팅 중 디바이스가 응답하는 순서에 따라 달라질 수 있다.

따라서 `/dev/nvme1n1` 같은 이름을 고정적으로 신뢰하면 위험하다.

권장 방식은 다음과 같다.

- EBS Volume ID 확인
- UUID 사용
- 파일시스템 UUID 기반으로 fstab 설정

### 6-3) 볼륨 ID와 NVMe 디바이스 매핑

Linux에서 확인할 수 있는 명령어는 다음과 같다.

```bash
lsblk -o NAME,SIZE,FSTYPE,MOUNTPOINT,SERIAL
```

Amazon Linux 계열에서는 다음 명령도 사용할 수 있다.

```bash
sudo /sbin/ebsnvme-id /dev/nvme1n1
```

일반 Linux 배포판에서는 다음 명령을 사용할 수 있다.

```bash
sudo nvme id-ctrl -V /dev/nvme1n1
```

### 6-4) NVMe I/O Timeout

Nitro 기반 인스턴스에서 EBS는 NVMe 드라이버를 통해 동작한다.

Linux에서는 NVMe I/O timeout 설정이 존재한다.

확인 명령은 다음과 같다.

```bash
cat /sys/module/nvme_core/parameters/io_timeout
```

I/O 지연이 timeout보다 길어지면 Linux NVMe 드라이버가 I/O 실패를 반환할 수 있다. 경우에 따라 파일시스템이 read-only로 다시 마운트될 수 있다.

### 6-5) NVMe Abort 명령

Abort 명령은 timeout을 초과한 NVMe I/O 명령을 종료하기 위해 사용되는 NVMe 관리 명령이다.

EBS에서는 지원되는 인스턴스와 디바이스 조건에 따라 Abort 명령이 처리된다.

운영 중 `dmesg`에서 NVMe timeout, abort, reset 관련 메시지가 보이면 EBS 지연 또는 I/O path 문제를 의심해야 한다.

## 7. EBS 볼륨 수명 주기와 기본 작업

### 7-1) 볼륨 상태

EBS 볼륨은 수명 주기 동안 여러 상태를 가진다.

주요 상태는 다음과 같다.

- creating
- available
- in-use
- deleting
- deleted
- error

`available`은 인스턴스에 연결되지 않은 상태다. `in-use`는 인스턴스에 연결된 상태다.

### 7-2) 볼륨 생성

볼륨은 다음 두 방식으로 생성할 수 있다.

- 빈 볼륨 생성
- 스냅샷에서 볼륨 생성

빈 볼륨은 생성 후 바로 최대 성능을 제공한다.

스냅샷에서 생성한 볼륨은 초기화가 필요할 수 있다. 처음 접근하는 블록은 S3에서 다운로드되어야 하므로 지연이 증가할 수 있다.

### 7-3) 볼륨 연결

EBS 볼륨은 같은 AZ의 EC2 인스턴스에 연결해야 한다.

CLI 예시는 다음과 같다.

```bash
aws ec2 attach-volume \
  --volume-id vol-xxxxxxxxxxxxxxxxx \
  --instance-id i-xxxxxxxxxxxxxxxxx \
  --device /dev/sdf
```

Nitro 인스턴스에서는 OS 내부 디바이스명이 `/dev/nvme1n1`처럼 보일 수 있다.

### 7-4) 볼륨 포맷과 마운트

Linux에서 새 빈 볼륨을 사용하려면 파일시스템을 만들고 마운트해야 한다.

디바이스 확인:

```bash
lsblk
```

파일시스템 확인:

```bash
sudo file -s /dev/nvme1n1
```

XFS 생성:

```bash
sudo mkfs -t xfs /dev/nvme1n1
```

마운트 디렉터리 생성:

```bash
sudo mkdir /data
```

마운트:

```bash
sudo mount /dev/nvme1n1 /data
```

### 7-5) 재부팅 후 자동 마운트

재부팅 후 자동 마운트를 하려면 `/etc/fstab`에 등록해야 한다.

권장 방식은 디바이스명 대신 UUID를 사용하는 것이다.

UUID 확인:

```bash
sudo blkid
```

fstab 예시:

```text
UUID=<UUID> /data xfs defaults,nofail 0 2
```

`nofail` 옵션을 넣으면 해당 볼륨이 없어도 인스턴스 부팅이 실패하지 않는다.

### 7-6) 볼륨 정보 보기

볼륨 정보는 콘솔 또는 CLI로 확인한다.

```bash
aws ec2 describe-volumes \
  --volume-ids vol-xxxxxxxxxxxxxxxxx
```

OS 내부에서는 다음을 확인한다.

```bash
lsblk
sudo lsblk -f
df -hT
```

### 7-7) 볼륨 수정

Elastic Volumes를 사용하면 볼륨을 분리하지 않고도 크기, 유형, IOPS, Throughput을 변경할 수 있다.

예시:

```bash
aws ec2 modify-volume \
  --volume-id vol-xxxxxxxxxxxxxxxxx \
  --volume-type gp3 \
  --iops 6000 \
  --throughput 250
```

수정 상태 확인:

```bash
aws ec2 describe-volumes-modifications \
  --volume-ids vol-xxxxxxxxxxxxxxxxx
```

### 7-8) 볼륨 크기 확장 후 파일시스템 확장

EBS 볼륨 크기를 키운 뒤에는 OS에서 파티션과 파일시스템도 확장해야 한다.

파티션 확장:

```bash
sudo growpart /dev/nvme0n1 1
```

XFS 확장:

```bash
sudo xfs_growfs -d /
```

EXT4 확장:

```bash
sudo resize2fs /dev/nvme0n1p1
```

### 7-9) 볼륨 분리

볼륨을 분리하기 전에는 먼저 파일시스템을 마운트 해제해야 한다.

```bash
sudo umount /data
```

CLI 분리:

```bash
aws ec2 detach-volume \
  --volume-id vol-xxxxxxxxxxxxxxxxx
```

마운트된 상태에서 분리하면 데이터 불일치나 파일시스템 손상이 발생할 수 있다.

### 7-10) 볼륨 삭제

볼륨은 `available` 상태일 때 삭제할 수 있다.

```bash
aws ec2 delete-volume \
  --volume-id vol-xxxxxxxxxxxxxxxxx
```

삭제하면 데이터가 사라진다. 필요한 데이터가 있다면 먼저 스냅샷을 생성해야 한다.

### 7-11) 스냅샷으로 볼륨 교체

문제가 생긴 볼륨은 기존 스냅샷에서 새 볼륨을 생성해 교체할 수 있다.

절차는 다음과 같다.

```text
스냅샷에서 새 볼륨 생성
↓
기존 볼륨 마운트 해제
↓
기존 볼륨 분리
↓
새 볼륨 연결
↓
마운트
↓
애플리케이션 확인
```

## 8. EBS Multi-Attach

### 8-1) Multi-Attach란?

Multi-Attach는 하나의 EBS 볼륨을 여러 EC2 인스턴스에 동시에 연결하는 기능이다.

모든 볼륨에서 가능한 것은 아니며, 프로비저닝된 IOPS SSD 계열인 io1 또는 io2에서 지원된다.

### 8-2) Multi-Attach 제한

주요 제한은 다음과 같다.

- 같은 AZ의 인스턴스에만 연결 가능
- Nitro 기반 인스턴스 필요
- 최대 16개 인스턴스에 연결 가능
- 표준 파일시스템 XFS, EXT4를 여러 인스턴스에서 동시에 쓰는 것은 위험
- 클러스터 파일시스템 또는 애플리케이션 수준 동시성 제어 필요
- 부트 볼륨으로 사용 불가

### 8-3) Multi-Attach 성능

각 인스턴스가 볼륨에 I/O를 보낼 수 있지만, 전체 합산 성능은 볼륨의 프로비저닝된 성능을 초과할 수 없다.

예를 들어 io2 볼륨이 80,000 IOPS라면 연결된 모든 인스턴스의 합산 IOPS가 80,000을 넘을 수 없다.

### 8-4) NVMe 예약

Multi-Attach io2 볼륨은 NVMe 예약을 지원한다.

NVMe 예약은 여러 인스턴스가 공유 볼륨에 접근할 때 쓰기 접근을 제어하고 데이터 일관성을 유지하기 위한 스토리지 펜싱 기능이다.

공유 스토리지를 안정적으로 운영하려면 애플리케이션이 동시 쓰기를 제어해야 한다.

## 9. EBS 볼륨 상태 확인과 장애 대응

### 9-1) 볼륨 상태 확인

EBS는 볼륨 상태 확인을 통해 데이터 불일치 가능성이나 I/O 상태를 알려준다.

주요 상태는 다음과 같다.

- ok
- warning
- impaired
- insufficient-data

`ok`는 상태 확인을 통과한 상태다.

`warning`은 성능이 예상보다 낮을 수 있는 상태다.

`impaired`는 볼륨 손상 또는 I/O 비활성화 가능성을 의미한다.

### 9-2) I/O 비활성화

EBS가 볼륨 데이터의 잠재적 불일치를 감지하면 데이터 손상을 방지하기 위해 I/O를 비활성화할 수 있다.

이 경우 이벤트에 `Awaiting Action: Enable IO`가 나타날 수 있다.

### 9-3) 볼륨 이벤트

주요 이벤트는 다음과 같다.

- Awaiting Action: Enable IO
- IO Enabled
- IO Auto-Enabled
- Normal
- Degraded
- Severely Degraded
- Stalled

### 9-4) 손상된 볼륨 대응

대응 방법은 크게 세 가지다.

- 연결된 인스턴스에서 I/O 활성화 후 일관성 검사
- 다른 인스턴스에 연결해서 일관성 검사
- 필요 없으면 볼륨 삭제 또는 스냅샷에서 복구

I/O 활성화:

```bash
aws ec2 enable-volume-io \
  --volume-id vol-xxxxxxxxxxxxxxxxx
```

Linux 파일시스템 검사는 마운트 해제 상태에서 수행해야 한다.

EXT 계열:

```bash
sudo fsck -f /dev/nvme1n1
```

XFS:

```bash
sudo xfs_repair /dev/nvme1n1
```

### 9-5) Auto-Enable IO

Auto-Enable IO를 활성화하면 EBS가 잠재적 불일치를 감지해도 I/O를 자동으로 다시 활성화할 수 있다.

하지만 데이터 일관성 확인 없이 계속 사용할 수 있으므로 주의가 필요하다.

## 10. EBS에서 오류 테스트

### 10-1) AWS FIS

AWS Fault Injection Service는 장애 주입 실험 서비스다.

EBS에서는 다음 실험이 가능하다.

- I/O Pause
- Latency Injection

### 10-2) I/O Pause

I/O Pause는 EBS 볼륨과 EC2 인스턴스 사이의 I/O를 일정 시간 중단한다.

이를 통해 다음을 확인할 수 있다.

- 애플리케이션 timeout
- DB 장애 반응
- CloudWatch Alarm 동작
- 장애 조치 절차
- VolumeStalledIOCheck 반응

### 10-3) Latency Injection

Latency Injection은 일부 읽기/쓰기 I/O에 인위적으로 지연 시간을 주입한다.

이를 통해 다음을 확인할 수 있다.

- 지연 증가 시 서비스가 버티는지
- p95/p99 latency가 어떻게 변하는지
- 장애 조치 임계값이 적절한지
- 모니터링과 알람이 동작하는지

### 10-4) 운영 환경 주의사항

운영에서 FIS를 사용할 때는 반드시 다음을 준비해야 한다.

- 스냅샷
- 승인 절차
- 점검 시간
- 중지 조건
- CloudWatch 알람
- 롤백 계획
- 담당자 대기

## 11. EBS 스냅샷

### 11-1) EBS 스냅샷이란?

EBS 스냅샷은 EBS 볼륨의 특정 시점 백업이다.

스냅샷을 만들면 해당 시점의 볼륨 데이터를 기반으로 나중에 새 EBS 볼륨을 복원할 수 있다.

AWS는 EBS 볼륨 데이터를 자동으로 백업하지 않는다. 스냅샷 생성 또는 백업 자동화는 사용자 책임이다.

### 11-2) 스냅샷 저장 위치

EBS 스냅샷은 Amazon S3에 저장되지만, 사용자가 S3 콘솔에서 직접 접근하는 일반 버킷에 저장되는 것은 아니다.

스냅샷 관리는 EC2 콘솔, EC2 API, AWS CLI를 통해 수행한다.

### 11-3) 증분 백업

EBS 스냅샷은 증분 백업이다.

첫 번째 스냅샷은 사용 중인 블록을 저장하고, 이후 스냅샷은 이전 스냅샷 이후 변경된 블록만 저장한다.

이 구조로 인해 백업 시간과 저장 비용을 줄일 수 있다.

다만 스냅샷을 삭제한다고 해서 항상 그만큼 비용이 즉시 줄어드는 것은 아니다. 다른 스냅샷이 참조하는 블록은 계속 유지된다.

### 11-4) 스냅샷에서 볼륨 생성

스냅샷에서 새 EBS 볼륨을 생성할 수 있다.

새 볼륨은 스냅샷을 만든 시점의 원본 볼륨과 동일한 데이터를 가진다.

다른 AZ에 볼륨을 생성할 수 있으므로 AZ 간 데이터 이동에도 사용할 수 있다.

### 11-5) 스냅샷 생성 시 주의사항

실행 중인 인스턴스에 연결된 볼륨도 스냅샷을 만들 수 있다.

하지만 애플리케이션 일관성이 필요한 경우에는 다음이 필요할 수 있다.

- 애플리케이션 쓰기 중지
- 파일시스템 flush
- DB freeze 또는 backup mode
- 다중 볼륨 스냅샷 사용
- Data Lifecycle Manager 사전/사후 스크립트 사용

### 11-6) 다중 볼륨 스냅샷

여러 EBS 볼륨이 하나의 애플리케이션을 구성하는 경우, 각각의 볼륨을 따로 스냅샷하면 시점이 맞지 않을 수 있다.

다중 볼륨 스냅샷을 사용하면 인스턴스에 연결된 여러 EBS 볼륨에 대해 특정 시점의 충돌 일관성 있는 스냅샷을 만들 수 있다.

RAID 구성이나 DB 데이터/로그 분리 구성에서 중요하다.

### 11-7) 스냅샷 복사

스냅샷은 다른 리전으로 복사할 수 있다.

주요 목적은 다음과 같다.

- 재해 복구
- 리전 간 마이그레이션
- 데이터 복제
- 규정 준수 목적의 원격 보관

암호화된 스냅샷을 복사할 때는 KMS 키 권한과 대상 리전의 키를 고려해야 한다.

### 11-8) 스냅샷 공유

스냅샷은 다른 AWS 계정과 공유할 수 있다.

공유 방식은 크게 두 가지다.

- 특정 계정과 프라이빗 공유
- 퍼블릭 공유

운영 환경에서는 퍼블릭 공유는 매우 위험하다. 민감한 데이터 유출 위험이 있으므로 퍼블릭 액세스 차단을 사용하는 것이 좋다.

### 11-9) 암호화된 스냅샷 공유

암호화된 스냅샷을 공유하려면 스냅샷뿐 아니라 KMS 키 사용 권한도 공유해야 한다.

KMS 권한이 없으면 상대 계정이 스냅샷을 사용할 수 없다.

### 11-10) 스냅샷 아카이브

스냅샷 아카이브는 자주 접근하지 않는 장기 보관용 스토리지 계층이다.

일반 스냅샷보다 비용을 줄일 수 있지만, 복원에 시간이 걸린다.

적합한 경우는 다음과 같다.

- 장기 규정 준수 보관
- 거의 접근하지 않는 백업
- 비용 최적화 목적

### 11-11) Fast Snapshot Restore

Fast Snapshot Restore는 스냅샷에서 생성한 볼륨이 즉시 완전한 성능을 제공하도록 하는 기능이다.

일반적으로 스냅샷 기반 볼륨은 처음 접근하는 블록에서 지연이 발생할 수 있다. FSR을 활성화하면 이 초기화 지연을 줄일 수 있다.

적합한 경우는 다음과 같다.

- 빠른 복구가 필요한 운영 볼륨
- Auto Scaling에서 빠르게 생성되는 인스턴스의 루트 볼륨
- 대규모 장애 복구
- 성능 저하 없이 즉시 사용해야 하는 DB 볼륨

### 11-12) 스냅샷 잠금

스냅샷 잠금은 스냅샷이 실수 또는 악의적 작업으로 삭제되는 것을 방지하는 기능이다.

규정 준수, 랜섬웨어 대응, 장기 백업 보호에 사용할 수 있다.

### 11-13) 스냅샷 퍼블릭 액세스 차단

스냅샷 퍼블릭 액세스 차단은 스냅샷 공개 공유를 방지하는 계정/리전 단위 보호 기능이다.

운영 환경에서는 활성화하는 것이 안전하다.

## 12. EBS 암호화

### 12-1) EBS 암호화 개념

EBS 암호화는 EBS 볼륨과 스냅샷을 보호하는 암호화 기능이다.

AWS KMS 키를 사용하며, 사용자가 별도 암호화 시스템을 직접 구축하지 않아도 된다.

### 12-2) 암호화되는 데이터

EBS 암호화가 활성화되면 다음 데이터가 암호화된다.

- 볼륨 내부 데이터
- EC2 인스턴스와 EBS 볼륨 사이에서 이동하는 데이터
- 볼륨에서 생성된 스냅샷
- 암호화된 스냅샷에서 생성한 볼륨

### 12-3) KMS 키

EBS 암호화에는 KMS 키가 사용된다.

사용 가능한 키 유형은 다음과 같다.

- AWS 관리형 키: aws/ebs
- 고객 관리형 키

고객 관리형 키를 사용하면 키 정책, 키 회전, 접근 권한, 감사 등을 더 세밀하게 제어할 수 있다.

### 12-4) 기본 암호화

계정/리전 단위로 EBS 기본 암호화를 활성화할 수 있다.

기본 암호화가 활성화되면 새로 생성되는 EBS 볼륨과 스냅샷 기반 볼륨이 자동으로 암호화된다.

### 12-5) 암호화 결과

암호화 상태는 다음에 따라 달라진다.

- 기본 암호화 활성화 여부
- 소스 스냅샷 암호화 여부
- KMS 키 지정 여부
- 콘솔/CLI/API 사용 방식

암호화된 스냅샷에서 생성된 볼륨은 암호화된다. 암호화된 볼륨에서 생성한 스냅샷도 암호화된다.

### 12-6) KMS 권한 문제

암호화된 EBS 리소스를 사용할 때 KMS 권한이 없으면 볼륨 생성, 연결, 복원 작업이 실패할 수 있다.

특히 다른 계정과 암호화된 스냅샷을 공유할 때는 다음이 모두 필요하다.

- 스냅샷 공유 권한
- KMS 키 사용 권한
- 대상 계정의 적절한 IAM 권한

## 13. EBS 볼륨 성능

### 13-1) 성능에 영향을 주는 요소

EBS 성능은 다음 요소에 의해 결정된다.

- 볼륨 유형
- 볼륨 크기
- 프로비저닝된 IOPS
- 프로비저닝된 Throughput
- I/O 크기
- Queue Depth
- 워크로드의 랜덤/순차 패턴
- EC2 인스턴스의 EBS 대역폭
- EBS 최적화 여부
- 스냅샷 복원 초기화 여부

### 13-2) EBS 최적화 인스턴스

EBS 최적화 인스턴스는 EC2와 EBS 사이의 I/O를 위한 전용 대역폭을 제공한다.

EBS 최적화가 없으면 일반 네트워크 트래픽과 EBS 트래픽이 경합할 수 있다.

### 13-3) 볼륨 초기화

스냅샷에서 생성한 EBS 볼륨은 모든 블록이 즉시 준비된 상태가 아닐 수 있다.

처음 접근하는 블록은 S3에서 EBS로 다운로드되어야 하므로 지연이 발생할 수 있다.

해결 방법은 다음과 같다.

- Fast Snapshot Restore
- 볼륨 초기화 속도 지정
- 수동 초기화

수동 초기화 예시:

```bash
sudo fio --filename=/dev/xvdf --rw=read --bs=1M --iodepth=32 --ioengine=libaio --direct=1 --name=volume-initialize
```

### 13-4) IOPS

IOPS는 초당 I/O 작업 수다.

작은 랜덤 I/O가 많은 DB 워크로드에서 중요하다.

### 13-5) Throughput

Throughput은 초당 처리 가능한 데이터 양이다.

대용량 순차 읽기/쓰기, 백업, 로그 처리, ETL에서 중요하다.

### 13-6) I/O 크기

SSD 볼륨은 최대 256KiB I/O 단위로 계산된다.

예를 들어 1,024KiB I/O는 256KiB I/O 4개로 분할되어 4 IOPS로 계산될 수 있다.

HDD 볼륨은 1,024KiB 순차 I/O에 최적화되어 있다.

### 13-7) Queue Depth와 Latency

Queue Depth는 대기 중인 I/O 요청 수다.

Queue가 너무 낮으면 볼륨 성능을 충분히 활용하지 못할 수 있고, 너무 높으면 Latency가 증가한다.

일반적으로 SSD 계열에서는 프로비저닝된 1,000 IOPS당 Queue Depth 1 정도를 기준으로 볼 수 있다.

### 13-8) RAID 0

단일 EBS 볼륨보다 더 높은 IOPS 또는 Throughput이 필요할 때 여러 EBS 볼륨을 RAID 0으로 묶을 수 있다.

장점은 성능 증가다.

단점은 볼륨 하나가 손상되면 전체 어레이 데이터가 손상될 수 있다는 점이다.

따라서 RAID 0은 반드시 백업/스냅샷 전략과 함께 사용해야 한다.

### 13-9) 벤치마킹

EBS 성능은 fio 또는 DiskSpd 같은 도구로 테스트할 수 있다.

운영 볼륨에서 직접 실행하면 위험하므로 반드시 테스트 볼륨에서 수행해야 한다.

fio 설치:

```bash
sudo yum install -y fio
```

랜덤 읽기 예시:

```bash
sudo fio --directory=/data --name=test --direct=1 --rw=randread --bs=16k --size=1G --numjobs=16 --runtime=180 --time_based
```

## 14. Amazon Data Lifecycle Manager

### 14-1) Data Lifecycle Manager란?

Amazon Data Lifecycle Manager는 EBS 스냅샷과 EBS-backed AMI의 생성, 보존, 삭제를 자동화하는 기능이다.

주요 목적은 다음과 같다.

- 정기 백업 자동화
- 오래된 백업 삭제
- 비용 절감
- 규정 준수 보관
- 재해 복구용 리전/계정 복사
- 표준 AMI 생성 자동화

### 14-2) DLM이 관리할 수 있는 리소스

DLM은 DLM 정책으로 생성된 스냅샷과 AMI를 관리한다.

다른 방식으로 생성한 스냅샷이나 AMI는 DLM이 자동으로 관리하지 못한다.

### 14-3) 기본 정책과 사용자 지정 정책

DLM 정책은 크게 다음으로 볼 수 있다.

- 기본 정책
- 사용자 지정 정책

기본 정책은 계정/리전 단위의 기본 백업 정책을 빠르게 만들 때 유용하다.

사용자 지정 정책은 태그 기반 대상 지정, 일정, 보존, 복사, 스크립트 실행 등 세밀한 구성이 필요할 때 사용한다.

### 14-4) 스냅샷 정책

스냅샷 정책은 EBS 볼륨을 대상으로 주기적으로 스냅샷을 생성한다.

보통 태그를 기준으로 백업 대상을 선택한다.

예를 들어 `Backup=true` 태그가 있는 볼륨만 백업하도록 설정할 수 있다.

### 14-5) AMI 정책

EBS-backed AMI 정책은 EC2 인스턴스를 대상으로 AMI 생성을 자동화한다.

표준화된 서버 이미지를 주기적으로 생성해야 할 때 유용하다.

### 14-6) 사전/사후 스크립트

애플리케이션 일관성 있는 스냅샷을 위해 DLM은 사전/사후 스크립트를 사용할 수 있다.

예를 들어 스냅샷 생성 전에 DB를 freeze하고, 생성 후 다시 thaw하는 방식이다.

### 14-7) 교차 계정/교차 리전 복사

DLM은 백업을 다른 계정이나 다른 리전으로 복사하는 정책을 구성할 수 있다.

이는 재해 복구와 랜섬웨어 대응에 중요하다.

### 14-8) DLM 모니터링

DLM은 CloudWatch, EventBridge, CloudTrail과 연동해 정책 실행 상태를 모니터링할 수 있다.

## 15. EBS Direct API

### 15-1) EBS Direct API란?

EBS Direct API는 EBS 스냅샷의 블록 데이터에 직접 접근할 수 있는 API다.

일반적으로 사용자가 직접 자주 쓰는 기능은 아니며, 백업 솔루션, 재해 복구 솔루션, 마이그레이션 도구를 만드는 경우 중요하다.

### 15-2) 가능한 작업

EBS Direct API로 할 수 있는 작업은 다음과 같다.

- 스냅샷 블록 목록 조회
- 두 스냅샷 간 변경 블록 비교
- 스냅샷 블록 읽기
- 새 스냅샷 생성 시작
- 스냅샷에 블록 쓰기
- 체크섬으로 데이터 검증

### 15-3) 주요 개념

EBS Direct API는 스냅샷을 블록 단위로 다룬다.

주요 API 개념은 다음과 같다.

- ListSnapshotBlocks
- ListChangedBlocks
- GetSnapshotBlock
- StartSnapshot
- PutSnapshotBlock
- CompleteSnapshot

### 15-4) 활용 사례

주요 활용 사례는 다음과 같다.

- 증분 백업 솔루션
- 온프레미스에서 AWS로 재해 복구 데이터 적재
- 스냅샷 간 변경 블록 분석
- EC2 인스턴스 없이 스냅샷 데이터 접근

### 15-5) 보안과 권한

EBS Direct API 사용에는 IAM 권한이 필요하다.

암호화된 스냅샷을 다룰 경우 KMS 권한도 필요하다.

### 15-6) 성능 최적화

스냅샷 블록 작업은 병렬 요청으로 성능을 높일 수 있다.

다만 API 할당량, 재시도 정책, 체크섬 검증, 멱등성 처리를 고려해야 한다.

## 16. 휴지통

### 16-1) 휴지통이란?

휴지통은 삭제된 EBS 볼륨, EBS 스냅샷, EBS-backed AMI를 일정 기간 보존하여 실수로 삭제된 리소스를 복구할 수 있게 해주는 기능이다.

### 16-2) 동작 방식

보존 규칙에 일치하는 리소스가 삭제되면 즉시 영구 삭제되지 않고 휴지통에 보관된다.

보존 기간이 끝나기 전에 복구할 수 있다.

보존 기간이 만료되면 영구 삭제되며 복구할 수 없다.

### 16-3) 지원 리소스

휴지통은 다음 리소스를 지원한다.

- EBS 볼륨
- EBS 스냅샷
- EBS-backed AMI

### 16-4) 보존 규칙

휴지통은 보존 규칙을 기준으로 동작한다.

보존 규칙은 리소스 유형, 태그 조건, 보존 기간 등을 정의한다.

### 16-5) 규칙 잠금

휴지통 보존 규칙을 잠가서 삭제나 변경을 방지할 수 있다.

이는 랜섬웨어나 내부 실수로부터 백업 리소스를 보호하는 데 유용하다.

### 16-6) 복구

휴지통에 있는 리소스는 보존 기간 내 복구할 수 있다.

복구하면 해당 리소스가 휴지통에서 제거되고 일반 리소스처럼 다시 사용할 수 있다.

## 17. Amazon EBS 보안

### 17-1) 공동 책임 모델

AWS는 클라우드 자체의 보안을 책임지고, 사용자는 클라우드 안에서 구성하는 리소스와 데이터 보안을 책임진다.

EBS에서 사용자가 책임져야 하는 대표 항목은 다음과 같다.

- IAM 권한 관리
- KMS 키 관리
- 스냅샷 공유 설정
- 퍼블릭 액세스 차단
- 백업 정책
- 보안 모니터링
- 데이터 분류와 보존 정책

### 17-2) 데이터 보호

EBS 데이터 보호를 위해 다음을 사용한다.

- EBS 암호화
- KMS 키 정책
- 스냅샷 암호화
- 스냅샷 잠금
- 휴지통
- DLM 백업 자동화
- 퍼블릭 액세스 차단

### 17-3) IAM 액세스 제어

EBS는 EC2 API의 일부로 관리되므로 IAM 정책을 통해 접근을 제어한다.

제어 가능한 작업 예시는 다음과 같다.

- CreateVolume
- AttachVolume
- DetachVolume
- DeleteVolume
- CreateSnapshot
- DeleteSnapshot
- ModifyVolume
- EnableVolumeIO

운영 환경에서는 최소 권한 원칙을 적용해야 한다.

### 17-4) 스냅샷 공유 보안

스냅샷은 민감한 데이터를 포함할 수 있다.

따라서 다음을 준수해야 한다.

- 퍼블릭 공유 금지
- 계정 간 공유 시 최소 대상 계정만 허용
- 암호화된 스냅샷은 KMS 권한 검토
- 스냅샷 퍼블릭 액세스 차단 사용
- CloudTrail로 공유 변경 감시

### 17-5) 데이터 복원력

EBS 볼륨은 AZ 내부에서 복제된다.

하지만 사용자의 실수, 파일시스템 손상, 애플리케이션 오류, 랜섬웨어까지 자동으로 막아주는 것은 아니다.

따라서 스냅샷, DLM, 휴지통, 스냅샷 잠금, 교차 리전 복사를 함께 사용해야 한다.

## 18. Amazon EBS 모니터링

### 18-1) 모니터링 도구

EBS 모니터링에는 다음 도구를 사용할 수 있다.

- Amazon CloudWatch
- AWS CloudTrail
- Amazon EventBridge
- EBS 세부 성능 통계
- Amazon GuardDuty
- 결제 및 사용량 보고서

### 18-2) CloudWatch 지표

CloudWatch는 EBS 볼륨의 성능과 상태를 모니터링한다.

대표 지표는 다음과 같다.

- VolumeReadOps
- VolumeWriteOps
- VolumeReadBytes
- VolumeWriteBytes
- VolumeQueueLength
- VolumeIdleTime
- VolumeTotalReadTime
- VolumeTotalWriteTime
- BurstBalance
- VolumeThroughputPercentage
- VolumeIOPSPercentage
- VolumeAvgReadLatency
- VolumeAvgWriteLatency
- VolumeStalledIOCheck
- VolumeIOPSExceededCheck
- VolumeThroughputExceededCheck

### 18-3) EventBridge

EventBridge는 EBS 이벤트에 자동 대응할 수 있게 한다.

사용 예시는 다음과 같다.

- 스냅샷 완료 이벤트 감지
- 볼륨 상태 이상 이벤트 감지
- DLM 정책 실행 이벤트 감지
- 휴지통 이벤트 감지
- Lambda 또는 SNS와 연동

### 18-4) CloudTrail

CloudTrail은 EBS 관련 API 호출 기록을 남긴다.

확인 가능한 정보는 다음과 같다.

- 누가 호출했는가
- 언제 호출했는가
- 어떤 API를 호출했는가
- 어떤 IP에서 호출했는가
- 성공했는가 실패했는가

예를 들어 스냅샷 삭제, 볼륨 삭제, 공유 권한 변경 같은 작업을 추적할 수 있다.

### 18-5) EBS 세부 성능 통계

Nitro 기반 인스턴스에서는 EBS 세부 성능 통계를 통해 더 자세한 I/O 성능 정보를 볼 수 있다.

이를 통해 평균 지연 시간, IOPS, Throughput, 지연 시간 분포 등을 더 정밀하게 분석할 수 있다.

### 18-6) GuardDuty Malware Protection

GuardDuty는 EC2 인스턴스에 연결된 EBS 볼륨을 스캔하여 잠재적 악성 활동을 탐지할 수 있다.

보안 사고 대응이나 악성코드 탐지에 활용된다.

## 19. Amazon EBS 할당량

### 19-1) 할당량이란?

AWS 계정에는 EBS 관련 기본 할당량이 있다.

할당량은 리전별로 적용되는 경우가 많으며, 일부는 증가 요청이 가능하고 일부는 고정되어 있다.

### 19-2) 주요 할당량 유형

EBS에서 고려해야 할 할당량은 다음과 같다.

- 총 EBS 볼륨 스토리지 용량
- 볼륨 유형별 총 용량
- 프로비저닝 가능한 IOPS
- 스냅샷 개수
- 동시 스냅샷 복사 수
- 볼륨당 동시 스냅샷 수
- EBS Direct API 요청 제한
- Fast Snapshot Restore 관련 제한
- DLM 정책 수

### 19-3) 할당량 확인

Service Quotas 콘솔에서 확인할 수 있다.

CLI 또는 콘솔을 통해 현재 리전의 제한과 증가 가능 여부를 확인하고, 필요한 경우 증가 요청을 해야 한다.

### 19-4) 할당량 관련 장애 사례

운영 중 다음 문제가 발생할 수 있다.

- 새 볼륨 생성 실패
- 스냅샷 복사 실패
- IOPS 증설 실패
- 동시 작업 제한 초과
- DLM 정책 생성 실패
- EBS Direct API 요청 제한

이런 경우에는 애플리케이션 문제가 아니라 AWS 서비스 할당량 문제일 수 있다.

## 21. EKS와 EBS CSI Driver

### 21-1) Kubernetes에서 EBS 사용

EKS에서 EBS는 Persistent Volume으로 자주 사용된다.

Kubernetes에서는 사용자가 PVC를 생성하면 StorageClass와 EBS CSI Driver를 통해 실제 EBS 볼륨이 생성되고 Pod에 연결된다.

### 21-2) 기본 흐름

```text
StorageClass
↓
PersistentVolumeClaim
↓
EBS CSI Driver
↓
EBS Volume 생성
↓
Node에 Attach
↓
Pod에 Mount
```

### 21-3) 주의할 점

EBS는 AZ 단위 리소스다.

따라서 EBS가 생성된 AZ와 Pod가 실행되는 노드의 AZ가 맞아야 한다.

또한 일반 EBS 볼륨은 보통 `ReadWriteOnce` 형태로 사용된다. 여러 노드에서 동시에 읽고 쓰려면 EFS 같은 파일 스토리지나 Multi-Attach/클러스터 파일시스템을 별도로 검토해야 한다.

### 21-4) EKS에서 확인할 항목

- StorageClass의 volume type
- gp2인지 gp3인지
- allowVolumeExpansion 여부
- reclaimPolicy
- volumeBindingMode
- PVC 용량
- Pod와 Node의 AZ
- EBS CSI Driver 상태

## 22. 전체 요약

Amazon EBS는 EC2에서 사용하는 네트워크 기반 블록 스토리지다.

EBS를 제대로 이해하려면 단순히 볼륨 생성 방법만 보면 안 된다. 볼륨 유형, 성능 모델, 스냅샷, 암호화, 모니터링, 장애 대응, 백업 자동화, 보안까지 함께 이해해야 한다.

핵심 요약은 다음과 같다.

- EBS Volume은 EC2에 붙는 블록 스토리지다.
- EBS Snapshot은 특정 시점 백업이며 증분 방식이다.
- EBS는 AZ 단위 리소스이므로 EC2와 같은 AZ에 있어야 한다.
- gp3는 일반적인 기본 선택지다.
- gp2는 BurstBalance 고갈 문제를 주의해야 한다.
- io2는 고성능 DB에 적합하다.
- st1/sc1은 대용량 순차 I/O에 적합하다.
- EBS 암호화는 KMS를 사용한다.
- 스냅샷은 백업, 복구, 리전 이동, 계정 공유에 사용된다.
- DLM으로 스냅샷과 AMI 수명 주기를 자동화할 수 있다.
- 휴지통은 실수로 삭제된 볼륨, 스냅샷, AMI를 복구하는 보호 장치다.
- CloudWatch, EventBridge, CloudTrail로 모니터링과 자동화를 구성한다.
- EBS 성능은 볼륨뿐 아니라 EC2 인스턴스의 EBS 대역폭에도 영향을 받는다.
- EKS에서는 EBS CSI Driver를 통해 PVC의 실제 스토리지로 사용된다.
