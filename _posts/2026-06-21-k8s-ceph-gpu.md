## 1. 이 문서에서 다루는 범위

GPU 인프라를 Kubernetes와 Ceph 환경에서 본다는 것은 단순히 GPU 서버에 NVIDIA Driver를 설치하는 수준이 아닙니다. 실무에서는 GPU가 장착된 노드를 Kubernetes 클러스터에 붙이고, Pod가 GPU를 안정적으로 사용할 수 있게 하며, 학습 데이터와 모델 체크포인트를 Ceph 기반 스토리지에 저장하고, 멀티 GPU 또는 멀티 노드 학습 시 네트워크 병목을 줄이며, 모니터링과 장애 대응까지 설계해야 합니다.

이 문서의 핵심 범위는 다음과 같습니다.

| 구분 | 알아야 하는 내용 |
|---|---|
| Kubernetes GPU | GPU Operator, Device Plugin, NVIDIA Container Toolkit, GPU Feature Discovery, DCGM Exporter, MIG, Time-Slicing, MPS, CDI |
| Kubernetes Network | CNI, Multus, SR-IOV, RDMA, RoCE, HostDevice, Macvlan, NetworkAttachmentDefinition |
| Ceph + GPU | RBD, CephFS, RGW, ceph-csi, PVC, StorageClass, checkpoint, dataset staging, metadata 병목 |
| GPU Passthrough | VM 또는 Pod에 GPU/NIC 같은 PCIe 장치를 직접 할당하는 방식과 장단점 |
| OpenStack 연계 | Nova GPU Passthrough, vGPU, SR-IOV, Cinder/RBD, Glance/RBD, Manila/CephFS |
| 운영 | 모니터링, 장애 대응, 성능 병목, 보안, 멀티테넌시, 용량 산정 |

NVIDIA GPU Operator는 Kubernetes에서 GPU 사용에 필요한 NVIDIA Driver, Kubernetes Device Plugin, NVIDIA Container Toolkit, GPU Feature Discovery, DCGM 기반 모니터링 등을 자동화하는 도구입니다. Kubernetes는 GPU 같은 특수 하드웨어를 기본 리소스로 자동 인식하지 않기 때문에 Device Plugin 같은 확장 구조가 필요합니다. Ceph는 object, block, file 인터페이스를 제공하는 분산 스토리지이며 Kubernetes에서는 ceph-csi를 통해 RBD와 CephFS를 PVC로 사용할 수 있습니다.

참고 기준 문서: NVIDIA GPU Operator 공식 문서, Kubernetes Device Plugin 공식 문서, Kubernetes GPU Scheduling 문서, NVIDIA Container Toolkit 문서, NVIDIA Network Operator 문서, SR-IOV Network Device Plugin 문서, Ceph 공식 문서, ceph-csi 문서.

## 2. Kubernetes에서 GPU를 사용하는 전체 구조

### 2-1) Kubernetes는 GPU를 기본 리소스로 자동 관리하지 않는다

Kubernetes는 CPU와 memory는 기본 리소스로 관리합니다. 하지만 GPU, NIC, FPGA, RDMA 장치 같은 특수 하드웨어는 기본 리소스가 아닙니다. 이런 장치를 Kubernetes가 인식하고 Pod에 할당하려면 Device Plugin 또는 Dynamic Resource Allocation 같은 확장 메커니즘이 필요합니다.

NVIDIA GPU를 Kubernetes에서 사용하려면 일반적으로 다음 흐름이 필요합니다.

```text
GPU Node
→ NVIDIA Driver
→ NVIDIA Container Toolkit
→ NVIDIA Kubernetes Device Plugin
→ kubelet이 nvidia.com/gpu 리소스 인식
→ Pod가 nvidia.com/gpu 리소스 요청
→ 컨테이너 내부에서 CUDA/PyTorch/TensorFlow가 GPU 사용
```

GPU Operator를 사용하면 위 구성의 상당 부분을 Operator가 자동으로 배포하고 관리합니다.

### 2-2) Kubernetes GPU 구성요소 전체 그림

| 구성요소 | 역할 |
|---|---|
| NVIDIA Driver | Linux OS가 GPU 하드웨어를 제어하게 하는 드라이버 |
| NVIDIA Container Toolkit | Docker/containerd 컨테이너가 Host GPU를 사용할 수 있게 연결 |
| NVIDIA Device Plugin | Kubernetes에 `nvidia.com/gpu` 같은 GPU 리소스를 광고 |
| GPU Feature Discovery | GPU 모델, MIG 지원 여부 등 GPU feature를 node label로 노출 |
| Node Feature Discovery | 노드의 CPU, PCI, GPU 등 하드웨어 feature를 label로 노출 |
| DCGM Exporter | GPU metric을 Prometheus 형식으로 노출 |
| MIG Manager | MIG 지원 GPU에서 MIG partition 구성 관리 |
| GPU Operator | 위 구성요소를 Kubernetes에서 일관되게 설치·관리하는 Operator |
| CDI | 컨테이너 런타임에 GPU 같은 장치 주입을 표준화하는 방식 |

### 2-3) Pod가 GPU를 사용하는 방식

Pod는 GPU를 일반 CPU처럼 요청하지 않습니다. NVIDIA Device Plugin이 노드의 GPU를 `nvidia.com/gpu`라는 extended resource로 광고하면, Pod는 이 리소스를 limits에 지정해서 요청합니다.

예시 개념은 다음과 같습니다.

```yaml
resources:
  limits:
    nvidia.com/gpu: 1
```

중요한 점은 GPU 리소스는 기본적으로 CPU처럼 millicore 단위로 쪼개 쓰는 방식이 아니라는 점입니다. `nvidia.com/gpu: 1`은 GPU 1개를 요청한다는 의미입니다. GPU 공유가 필요하면 MIG, Time-Slicing, MPS 같은 별도 기능을 검토해야 합니다.

### 2-4) GPU Pod 배치 흐름

GPU Pod가 실행되는 흐름은 다음과 같습니다.

```text
1. GPU Node에 NVIDIA Driver와 GPU stack이 준비됨
2. NVIDIA Device Plugin이 kubelet에 GPU 리소스를 등록함
3. Node의 capacity/allocatable에 nvidia.com/gpu가 표시됨
4. 사용자가 GPU resource limit이 포함된 Pod를 생성함
5. Scheduler가 GPU 리소스가 남아 있는 Node를 선택함
6. kubelet이 컨테이너 런타임을 통해 GPU 장치를 컨테이너에 주입함
7. 컨테이너 내부 애플리케이션이 CUDA를 통해 GPU를 사용함
```

장애를 볼 때도 이 흐름을 역으로 추적하면 됩니다.

## 3. GPU Operator

### 3-1) GPU Operator란 무엇인가

GPU Operator는 Kubernetes에서 NVIDIA GPU를 사용하기 위해 필요한 소프트웨어 스택을 자동으로 배포하고 관리하는 Operator입니다. 수동 방식으로는 각 GPU 노드에 NVIDIA Driver를 설치하고, container runtime을 설정하고, Device Plugin을 배포하고, DCGM Exporter를 배포하고, node labeling을 구성해야 합니다. GPU Operator는 이 과정을 선언적으로 관리합니다.

GPU Operator가 관리하는 대표 구성요소는 다음과 같습니다.

| 구성요소 | 설명 |
|---|---|
| NVIDIA Driver | GPU 하드웨어 제어용 드라이버 |
| NVIDIA Container Toolkit | 컨테이너에서 GPU를 사용할 수 있게 하는 runtime hook/toolkit |
| NVIDIA Device Plugin | GPU를 Kubernetes extended resource로 노출 |
| GPU Feature Discovery | GPU 관련 정보를 node label로 노출 |
| Node Feature Discovery | 하드웨어 feature discovery |
| DCGM Exporter | GPU metric 수집 및 Prometheus 연동 |
| MIG Manager | MIG 설정 관리 |
| NVIDIA Validator | GPU stack 정상 동작 검증 |

### 3-2) GPU Operator를 쓰는 이유

GPU Operator를 사용하는 가장 큰 이유는 GPU 노드 구성을 표준화하고, Kubernetes 방식으로 관리하기 위해서입니다. GPU 노드가 1~2대면 수동 설치도 가능합니다. 하지만 GPU 노드가 여러 대가 되면 driver 버전, toolkit 설정, device plugin 버전, monitoring agent 버전이 노드마다 달라지는 문제가 생깁니다.

GPU Operator는 이런 문제를 줄여줍니다.

| 수동 구성 | GPU Operator 구성 |
|---|---|
| 노드마다 driver 직접 설치 | Operator가 driver daemonset 또는 pre-installed driver 방식 관리 |
| container runtime 직접 설정 | Toolkit 관련 컴포넌트 자동 구성 가능 |
| Device Plugin 직접 배포 | Operator가 Device Plugin 배포 |
| DCGM Exporter 별도 구성 | Operator monitoring 옵션으로 구성 가능 |
| MIG 설정 수동 관리 | MIG Manager로 구성 가능 |
| 버전 drift 발생 가능 | Operator chart/component matrix로 일관성 관리 |

### 3-3) GPU Operator 설치 방식 선택

GPU Operator 운영에서 중요한 선택지는 Driver를 누가 관리할 것인가입니다.

#### 방식 1: Operator가 Driver까지 설치

GPU 노드에 NVIDIA Driver가 설치되어 있지 않은 상태에서 GPU Operator가 Driver를 설치합니다. 장점은 Kubernetes에서 GPU stack을 한 번에 관리할 수 있다는 것입니다. 단점은 커널 버전, OS 이미지, 보안 정책, Secure Boot, 사내 표준 driver 관리 방식과 충돌할 수 있다는 점입니다.

#### 방식 2: Host에 Driver를 미리 설치하고 Operator는 나머지만 관리

GPU 노드 OS 이미지에 NVIDIA Driver와 Container Toolkit을 미리 설치하고, GPU Operator는 Device Plugin, GFD, DCGM Exporter 등만 관리합니다. NVIDIA 문서에서도 pre-installed driver가 있는 경우 `driver.enabled=false`와 같은 방식으로 Operator가 driver를 설치하지 않게 구성할 수 있습니다.

#### 실무 판단 기준

| 기준 | 권장 방향 |
|---|---|
| 노드 수가 적고 테스트 환경 | Operator가 driver까지 설치해도 가능 |
| 운영 표준 OS 이미지가 있음 | Host pre-installed driver 방식 선호 |
| Secure Boot/보안 정책 엄격 | 사전 검증된 driver image 또는 host driver 권장 |
| DGX/Base OS 사용 | pre-installed driver 구조 검토 |
| Kubernetes에서 모든 GPU stack을 선언형으로 관리하고 싶음 | GPU Operator driver 관리 검토 |

### 3-4) GPU Operator와 Device Plugin의 차이

GPU Operator와 Device Plugin은 같은 것이 아닙니다.

| 구분 | GPU Operator | NVIDIA Device Plugin |
|---|---|---|
| 성격 | 여러 NVIDIA GPU 구성요소를 관리하는 Operator | GPU 리소스를 kubelet에 광고하는 단일 구성요소 |
| 범위 | Driver, Toolkit, Device Plugin, GFD, DCGM, MIG 등 | `nvidia.com/gpu` 리소스 노출 |
| 목적 | GPU 노드 운영 자동화 | Kubernetes scheduling 가능하게 함 |
| 단독 사용 | GPU Operator 없이 Device Plugin만 배포 가능 | Device Plugin만으로는 driver/toolkit/monitoring 관리 안 됨 |

정리하면 Device Plugin은 GPU를 Kubernetes 리소스로 보이게 하는 핵심 부품이고, GPU Operator는 그 부품을 포함한 전체 NVIDIA GPU 운영 스택을 관리하는 도구입니다.

### 3-5) GPU Operator 장애 확인 포인트

GPU Operator 환경에서 Pod가 GPU를 못 쓰는 경우 다음 순서로 봅니다.

| 증상 | 확인 위치 |
|---|---|
| Node에 `nvidia.com/gpu`가 안 보임 | Device Plugin Pod, driver 상태, kubelet 로그 |
| GPU Operator Pod CrashLoop | operator logs, daemonset logs, 권한/RBAC |
| driver container 실패 | kernel header, Secure Boot, OS/kernel 호환성 |
| GPU는 host에서 보이는데 Pod에서 안 보임 | Container Toolkit, runtimeClass, CDI, device plugin |
| DCGM metric이 없음 | dcgm-exporter, ServiceMonitor, Prometheus scrape 설정 |
| MIG 설정이 반영 안 됨 | MIG Manager, GPU model, MIG mode, node label |

## 4. NVIDIA Device Plugin, CDI, Container Toolkit

### 4-1) NVIDIA Device Plugin

NVIDIA Device Plugin은 Kubernetes Device Plugin Framework를 이용해 GPU를 kubelet에 등록합니다. 이 플러그인이 정상 동작하면 노드의 capacity와 allocatable에 `nvidia.com/gpu`가 표시됩니다.

Device Plugin은 다음 역할을 합니다.

1. 노드의 GPU 장치 탐지
2. GPU 리소스를 kubelet에 광고
3. Pod가 GPU를 요청하면 해당 GPU를 컨테이너에 할당
4. MIG 또는 GPU sharing 정책에 따라 리소스 형태를 노출

Device Plugin이 없으면 Kubernetes는 GPU 장치를 스케줄링 리소스로 인식하지 못합니다.

### 4-2) NVIDIA Container Toolkit

NVIDIA Container Toolkit은 컨테이너가 host의 NVIDIA GPU와 driver library를 사용할 수 있도록 해주는 구성요소입니다. 컨테이너 내부에는 CUDA runtime이나 PyTorch가 있을 수 있지만, 실제 GPU 장치를 제어하는 Driver는 host에 있습니다. Container Toolkit은 이 둘을 연결해줍니다.

구조는 다음과 같습니다.

```text
Host OS
→ NVIDIA Driver
→ NVIDIA Container Toolkit
→ Docker/containerd
→ GPU-enabled Container
→ CUDA/PyTorch/TensorFlow
```

중요한 점은 컨테이너 안에 CUDA가 있어도 host driver가 없거나 Container Toolkit 설정이 잘못되면 GPU를 사용할 수 없다는 것입니다.

### 4-3) CDI(Container Device Interface)

CDI는 컨테이너 런타임에 GPU, NIC 같은 장치를 주입하는 방식을 표준화하기 위한 인터페이스입니다. 기존에는 런타임별로 장치 주입 방식이 달랐고, NVIDIA runtime 설정에 강하게 의존하는 경우가 많았습니다. CDI는 장치 정보를 spec 형태로 관리하여 컨테이너 런타임이 표준화된 방식으로 장치를 주입할 수 있게 합니다.

GPU 운영에서 CDI를 알아야 하는 이유는 다음과 같습니다.

| 이유 | 설명 |
|---|---|
| 런타임 표준화 | Docker, containerd, CRI-O 환경 차이를 줄임 |
| 장치 주입 명확화 | 어떤 GPU 장치를 컨테이너에 넣는지 spec으로 표현 |
| Kubernetes 연계 | Device Plugin과 함께 사용될 수 있음 |
| GPU 외 장치 확장성 | GPU뿐 아니라 NIC, DPU 같은 장치에도 적용 가능 |

### 4-4) Docker와 containerd에서의 차이

단일 서버에서는 Docker 기반으로 GPU 컨테이너를 테스트하는 경우가 많습니다. 하지만 Kubernetes는 일반적으로 containerd를 사용합니다. 따라서 `docker run --gpus all`이 된다고 해서 Kubernetes Pod에서 GPU가 반드시 되는 것은 아닙니다.

구분하면 다음과 같습니다.

| 환경 | 확인해야 하는 것 |
|---|---|
| Docker 단일 서버 | NVIDIA Container Toolkit, Docker runtime 설정 |
| Kubernetes/containerd | containerd config, NVIDIA runtime/CDI, kubelet, Device Plugin |
| GPU Operator | Operator가 toolkit/runtime 설정을 어떻게 관리하는지 |

## 5. GPU Passthrough, vGPU, MIG, Time-Slicing, MPS 차이

### 5-1) GPU Passthrough란 무엇인가

GPU Passthrough는 물리 GPU 또는 GPU의 특정 기능을 VM이나 컨테이너에 직접 할당하는 방식입니다. OpenStack, KVM, VMware 같은 가상화 환경에서는 PCI Passthrough를 통해 VM이 GPU를 직접 사용하는 구조를 만들 수 있습니다.

GPU Passthrough의 핵심은 hypervisor가 GPU를 에뮬레이션하는 것이 아니라, PCIe 장치를 VM에 직접 넘겨 VM 내부 OS가 GPU Driver를 설치하고 사용하는 방식입니다.

### 5-2) GPU Passthrough 장단점

| 항목 | 장점 | 단점 |
|---|---|---|
| 성능 | native에 가까운 성능 | GPU 단위로 점유되어 공유 효율 낮음 |
| 호환성 | VM 내부에서 일반 GPU 서버처럼 사용 가능 | live migration 제약 큼 |
| 격리 | VM 단위 격리 명확 | 한 GPU를 여러 VM이 유연하게 나누기 어려움 |
| 운영 | 구조가 직관적 | IOMMU, BIOS, driver, hypervisor 설정 필요 |

### 5-3) GPU Passthrough와 Kubernetes GPU 할당의 차이

| 구분 | GPU Passthrough | Kubernetes GPU Device Plugin |
|---|---|---|
| 대상 | VM 또는 특정 bare-metal workload | Pod/Container |
| 계층 | Hypervisor/PCIe passthrough | Kubernetes/kubelet/container runtime |
| 사용 예 | OpenStack Nova VM에 GPU 할당 | AI 학습 Pod에 GPU 할당 |
| 격리 | VM 격리 | 컨테이너 격리 |
| 공유 | 보통 GPU 단위 독점 | MIG/Time-Slicing/MPS로 공유 가능 |

### 5-4) vGPU

vGPU는 물리 GPU를 여러 가상 GPU로 나누어 여러 VM에 제공하는 기술입니다. NVIDIA vGPU는 별도 라이선스와 지원 GPU/Hypervisor 조건이 필요할 수 있습니다. GPU Passthrough가 GPU를 VM에 통째로 넘기는 방식에 가깝다면, vGPU는 GPU를 가상화하여 여러 VM이 나눠 쓰게 하는 방식입니다.

실무에서 vGPU는 VDI, 그래픽 워크스테이션, 일부 AI inference/개발 환경에 사용될 수 있습니다. 다만 대규모 학습에서는 vGPU보다 bare-metal GPU 또는 Kubernetes GPU Pod, Slurm job이 더 일반적입니다.

### 5-5) MIG

MIG는 Multi-Instance GPU입니다. 지원되는 NVIDIA GPU에서 하나의 물리 GPU를 여러 개의 하드웨어 격리된 GPU instance로 나누는 기능입니다. 대표적으로 A100, H100 계열에서 중요합니다.

MIG는 다음 상황에 유리합니다.

| 상황 | 이유 |
|---|---|
| 작은 추론 workload 여러 개 | 큰 GPU 하나를 여러 서비스가 나누어 사용 |
| 개발/테스트 환경 | 사용자별 작은 GPU slice 제공 |
| 멀티테넌시 | 하드웨어 수준 격리 제공 |
| GPU 활용률 개선 | 큰 GPU가 작은 작업 하나에 낭비되는 상황 방지 |

MIG를 사용할 때는 profile 개념을 알아야 합니다. 예를 들어 하나의 GPU를 여러 slice로 나누면 각 slice는 별도의 compute/memory capacity를 갖습니다. 정확한 profile 이름과 용량은 GPU 모델과 driver/MIG 지원 상태에 따라 다릅니다.

### 5-6) Time-Slicing

Time-Slicing은 여러 Pod가 같은 GPU를 시간 분할 방식으로 공유하게 하는 방식입니다. GPU Operator는 NVIDIA Device Plugin의 GPU sharing 옵션을 통해 oversubscription을 지원할 수 있습니다.

Time-Slicing의 장점은 설정이 비교적 단순하고, 작은 추론/개발 workload가 GPU를 공유할 수 있다는 것입니다. 단점은 MIG처럼 강한 하드웨어 격리를 제공하지 않고, 성능 예측성이 낮을 수 있다는 것입니다.

### 5-7) MPS

MPS는 CUDA Multi-Process Service입니다. 여러 CUDA 프로세스가 하나의 GPU를 더 효율적으로 공유하도록 돕는 NVIDIA 기능입니다. MPS는 작은 CUDA kernel을 여러 프로세스가 실행할 때 GPU 활용률을 높이는 데 도움이 될 수 있습니다.

### 5-8) GPU 공유 방식 비교

| 방식 | 대상 | 격리 수준 | 성능 예측성 | 대표 용도 |
|---|---|---|---|---|
| GPU Passthrough | VM | 높음 | 높음 | OpenStack VM GPU 제공 |
| vGPU | VM | 중간~높음 | 중간 | VDI, VM 기반 GPU 공유 |
| Kubernetes Device Plugin 기본 | Pod | GPU 단위 할당 | 높음 | 학습/추론 Pod |
| MIG | Pod/Container | 하드웨어 격리 높음 | 높음 | 추론, 개발, 멀티테넌시 |
| Time-Slicing | Pod/Container | 낮음 | 낮음~중간 | 개발/소형 추론 공유 |
| MPS | CUDA Process | 중간 | 중간 | CUDA 프로세스 병렬 실행 최적화 |

## 6. SR-IOV와 GPU 인프라

### 6-1) SR-IOV란 무엇인가

SR-IOV는 Single Root I/O Virtualization입니다. 하나의 물리 PCIe 장치, 보통 NIC를 여러 개의 가상 기능으로 나누어 VM이나 Pod가 직접 사용할 수 있게 하는 기술입니다.

SR-IOV에서 중요한 용어는 다음과 같습니다.

| 용어 | 의미 |
|---|---|
| PF | Physical Function. 실제 물리 장치 기능 |
| VF | Virtual Function. PF에서 생성된 가상 기능 |
| PCI Passthrough | PCIe 장치를 VM/Pod에 직접 할당하는 방식 |
| IOMMU | 장치의 DMA 접근을 격리·매핑하는 메커니즘 |
| VFIO | Linux에서 userspace/VM에 장치를 안전하게 전달하는 framework |

### 6-2) GPU와 SR-IOV의 관계

SR-IOV는 주로 NIC에서 많이 사용됩니다. GPU 자체를 SR-IOV로 나누어 쓰는 것과 일반적으로 말하는 GPU Passthrough, vGPU, MIG는 구분해야 합니다.

GPU 인프라에서 SR-IOV가 중요한 이유는 GPU 자체보다 **GPU 노드의 고성능 네트워크** 때문입니다. AI 학습이나 추론에서 네트워크 성능이 중요한 경우, Pod나 VM에 SR-IOV VF를 직접 붙여 overlay 네트워크 오버헤드를 줄일 수 있습니다.

예를 들어 다음과 같은 상황에서 SR-IOV가 등장합니다.

```text
GPU Pod
→ 기본 Pod Network는 Calico/Cilium 사용
→ 데이터 통신 또는 RDMA 통신은 SR-IOV VF를 추가 NIC로 사용
→ NCCL/RDMA/RoCE 통신 성능 확보
```

### 6-3) Kubernetes에서 SR-IOV 구성요소

Kubernetes에서 SR-IOV를 사용하려면 보통 다음 구성요소가 필요합니다.

| 구성요소 | 역할 |
|---|---|
| SR-IOV Network Device Plugin | VF/PF/SF 같은 네트워크 장치를 Kubernetes resource로 광고 |
| SR-IOV CNI | Pod에 SR-IOV VF 네트워크 인터페이스를 붙임 |
| Multus CNI | Pod에 여러 네트워크 인터페이스를 붙일 수 있게 함 |
| NetworkAttachmentDefinition | 추가 네트워크 정의 |
| SR-IOV Network Operator | SR-IOV 장치와 네트워크 구성을 Operator 방식으로 관리 |
| Node Feature Discovery | SR-IOV 가능한 장치 feature 감지 및 labeling |

### 6-4) Multus와 SR-IOV 관계

Kubernetes Pod는 기본적으로 하나의 Pod network interface를 가집니다. 하지만 고성능 네트워크나 분리된 데이터망이 필요할 때는 Pod에 추가 NIC를 붙여야 합니다. 이때 Multus가 사용됩니다.

구조는 다음과 같습니다.

```text
Pod
├─ eth0: 기본 Kubernetes CNI 네트워크
└─ net1: SR-IOV VF 기반 고성능 네트워크
```

GPU workload에서는 eth0로 Kubernetes 제어/서비스 트래픽을 처리하고, net1으로 NCCL, RDMA, 데이터 전송 같은 고성능 트래픽을 처리하는 구조를 설계할 수 있습니다.

### 6-5) SR-IOV와 RDMA

SR-IOV VF가 RDMA를 지원하는 NIC에서 생성되면 Pod나 VM이 RDMA-capable interface를 사용할 수 있습니다. 단, 단순히 VF를 Pod에 붙였다고 RDMA가 자동으로 잘 되는 것은 아닙니다. NIC, firmware, driver, kernel module, RDMA device plugin, CNI, network fabric, MTU, RoCE 설정까지 맞아야 합니다.

### 6-6) SR-IOV와 GPU Passthrough 차이

| 구분 | SR-IOV | GPU Passthrough |
|---|---|---|
| 주 대상 | NIC 같은 PCIe I/O 장치 | GPU 같은 PCIe 장치 |
| 목적 | 고성능 네트워크 VF 제공 | VM/Pod에 GPU 직접 제공 |
| 공유 방식 | 하나의 PF에서 여러 VF 생성 | GPU 통째 할당 또는 vGPU/MIG 별도 사용 |
| Kubernetes 구성 | SR-IOV CNI + Device Plugin + Multus | NVIDIA Device Plugin/GPU Operator |
| OpenStack 구성 | Neutron SR-IOV port + Nova PCI passthrough | Nova PCI passthrough/vGPU |

### 6-7) SR-IOV를 언제 써야 하는가

SR-IOV는 항상 필요한 기술이 아닙니다. 다음 조건이면 검토합니다.

| 상황 | SR-IOV 필요성 |
|---|---|
| 일반 웹/API Pod | 낮음 |
| 일반 AI 추론 API | 보통 낮음 |
| 대규모 멀티 노드 학습 | 높음 가능성 |
| NFV/Telco workload | 높음 |
| RDMA/RoCE 사용 | 높음 |
| Overlay 네트워크 오버헤드가 문제 | 검토 필요 |
| VM에 고성능 NIC 제공 | 높음 |

## 7. RDMA, RoCE, InfiniBand, GPUDirect RDMA

### 7-1) GPU 클러스터에서 네트워크가 중요한 이유

GPU가 한 장이거나 한 서버 안에서만 동작하면 네트워크보다 CPU, RAM, PCIe, VRAM, 스토리지가 더 중요할 수 있습니다. 하지만 여러 GPU 서버가 하나의 학습 작업에 참여하면 네트워크가 핵심 병목이 됩니다.

분산 학습에서는 각 GPU가 계산한 gradient를 서로 교환합니다. 이 과정이 느리면 GPU는 계산을 끝내고 다른 GPU의 결과를 기다립니다. 결국 비싼 GPU가 idle 상태가 됩니다.

### 7-2) RDMA

RDMA는 Remote Direct Memory Access입니다. 일반 TCP/IP 통신은 kernel network stack, socket buffer, system call, memory copy를 거칩니다. RDMA는 이 경로를 줄여 원격 메모리에 더 직접 접근함으로써 낮은 latency와 낮은 CPU overhead를 제공합니다.

GPU 인프라에서 RDMA가 중요한 이유는 다음과 같습니다.

| 효과 | 의미 |
|---|---|
| 낮은 latency | GPU 간 동기화 대기 감소 |
| 낮은 CPU overhead | CPU가 네트워크 처리에 덜 묶임 |
| 높은 throughput | 대량 gradient/parameter 전송에 유리 |
| GPU idle time 감소 | 분산 학습 효율 향상 |

### 7-3) RoCE

RoCE는 RDMA over Converged Ethernet입니다. Ethernet 위에서 RDMA를 구현하는 방식입니다. InfiniBand 전용망을 사용하지 않고 기존 Ethernet 기반 네트워크 위에서 RDMA를 사용할 수 있다는 장점이 있습니다.

하지만 RoCE는 아무 Ethernet 환경에서나 잘 되는 기술이 아닙니다. lossless network 구성이 중요하며, PFC, ECN, QoS, MTU, switch buffer, NIC firmware, driver 설정이 맞아야 합니다.

### 7-4) InfiniBand

InfiniBand는 HPC와 대규모 GPU 클러스터에서 사용하는 고성능 네트워크 fabric입니다. 낮은 latency, 높은 bandwidth, RDMA 지원이 핵심입니다.

InfiniBand에서 자주 나오는 용어는 다음과 같습니다.

| 용어 | 의미 |
|---|---|
| HCA | Host Channel Adapter. InfiniBand/RDMA NIC |
| Subnet Manager | InfiniBand fabric 관리, LID 할당, 라우팅 계산 |
| LID | Local Identifier. IB subnet 내부 주소 |
| Fabric | IB switch와 host가 구성하는 전체 네트워크 |
| Queue Pair | RDMA 송수신 큐 구조 |
| Completion Queue | RDMA 작업 완료 이벤트 큐 |

### 7-5) GPUDirect RDMA

GPUDirect RDMA는 NIC 같은 peer device가 GPU memory에 직접 접근할 수 있도록 하는 NVIDIA 기술입니다. 일반적으로는 GPU VRAM의 데이터를 CPU RAM으로 복사한 뒤 NIC를 통해 전송하지만, GPUDirect RDMA는 이 복사 경로를 줄여 CPU와 system memory 개입을 줄이는 방향입니다.

구조적으로 보면 다음 차이가 있습니다.

```text
일반 경로:
GPU VRAM → CPU RAM → NIC → Network

GPUDirect RDMA 경로:
GPU VRAM ↔ NIC → Network
```

이 기술은 대규모 분산 학습, 고성능 데이터 전송, RDMA 기반 GPU cluster에서 중요합니다.

### 7-6) RDMA와 SR-IOV 관계

SR-IOV는 장치를 Pod/VM에 직접 가깝게 할당하는 기술이고, RDMA는 네트워크 통신에서 kernel/CPU overhead를 줄이는 기술입니다. 서로 다른 개념이지만 GPU 클러스터에서는 함께 쓰일 수 있습니다.

예를 들어 RDMA-capable NIC에서 SR-IOV VF를 만들고, 그 VF를 Pod에 붙여 RDMA 통신을 하도록 구성할 수 있습니다. 이 경우 Multus, SR-IOV CNI, SR-IOV Device Plugin, RDMA device plugin, NIC driver, RoCE/IB fabric 설정이 모두 중요해집니다.

## 8. Ceph와 GPU 워크로드

### 8-1) Ceph가 GPU 인프라에서 왜 중요한가

GPU 서버는 연산 자원이고 Ceph는 데이터 자원입니다. AI 학습과 추론은 모델, 데이터셋, 체크포인트, 로그, artifact를 저장하고 읽어야 합니다. GPU가 아무리 빨라도 데이터 공급이 느리면 GPU는 놀게 됩니다.

Ceph는 object, block, file 인터페이스를 모두 제공하므로 GPU 인프라에서 다양한 역할을 할 수 있습니다.

| Ceph 인터페이스 | GPU 인프라에서의 용도 |
|---|---|
| RBD | Kubernetes PVC, VM disk, 학습 workspace, DB성 workload |
| CephFS | 여러 Pod가 공유하는 dataset, checkpoint, shared workspace |
| RGW/S3 | 데이터셋 저장, 모델 artifact, checkpoint archive, backup |

### 8-2) Ceph RBD와 GPU workload

RBD는 block device입니다. Kubernetes에서는 ceph-csi-rbd를 통해 PVC로 제공합니다. RBD는 하나의 Pod 또는 하나의 VM이 block device처럼 사용하는 데 적합합니다.

GPU workload에서 RBD가 적합한 경우는 다음과 같습니다.

| 용도 | 적합성 |
|---|---|
| 단일 학습 Pod workspace | 적합 |
| 모델 cache disk | 적합 가능 |
| DB/metadata service | 적합 |
| 여러 Pod가 동시에 파일 공유 | 부적합. CephFS 검토 |
| OpenStack VM root/volume disk | 매우 적합 |

RBD는 기본적으로 block device이므로 동시에 여러 노드에서 read-write로 붙이는 구조에 주의해야 합니다. Kubernetes에서 일반적으로 RWO(ReadWriteOnce)로 많이 사용됩니다.

### 8-3) CephFS와 GPU workload

CephFS는 분산 파일시스템입니다. 여러 Pod가 같은 파일 경로를 공유해야 할 때 유용합니다. Kubernetes에서는 ceph-csi-cephfs를 통해 RWX(ReadWriteMany) PVC로 사용할 수 있습니다.

GPU workload에서 CephFS가 적합한 경우는 다음과 같습니다.

| 용도 | 적합성 |
|---|---|
| 여러 학습 Pod가 같은 dataset 공유 | 적합 |
| checkpoint 공유 | 적합 가능 |
| 공용 workspace | 적합 |
| 작은 파일 수백만 개 읽기 | MDS/metadata 병목 주의 |
| 초고속 학습 데이터 공급 | local NVMe cache와 조합 권장 |

CephFS는 metadata server인 MDS가 중요합니다. 작은 파일이 많고 directory traversal이 많으면 MDS가 병목이 될 수 있습니다. 이미지 데이터셋처럼 작은 파일이 많으면 CephFS metadata 성능을 반드시 봐야 합니다.

### 8-4) RGW/S3와 GPU workload

RGW는 Ceph의 object gateway입니다. S3 호환 API를 제공하므로 dataset, model artifact, checkpoint archive, backup, 배포 이미지 저장에 유용합니다.

GPU 학습에서는 원본 데이터를 RGW/S3에 저장하고, 학습 전에 local NVMe나 CephFS로 staging하는 구조를 많이 사용합니다. 직접 S3에서 매 batch마다 작은 object를 읽으면 latency와 API overhead가 커질 수 있습니다.

### 8-5) GPU 학습과 Ceph 병목

GPU 학습에서 Ceph 병목은 다음 위치에서 자주 발생합니다.

| 병목 | 설명 |
|---|---|
| 작은 파일 대량 읽기 | CephFS MDS metadata 병목 |
| 대용량 checkpoint write | OSD write latency 상승 |
| recovery/backfill 중 학습 | Ceph 내부 복구 트래픽이 학습 I/O에 영향 |
| public network 혼잡 | Ceph client I/O 지연 |
| cluster network 혼잡 | OSD replication/backfill 지연 |
| OSD nearfull/full | 쓰기 지연 또는 실패 |
| CephFS MDS failover | mount 일시 지연 |
| CSI mount/map 지연 | Pod ContainerCreating 지연 |

### 8-6) GPU workload에 적합한 Ceph 사용 패턴

실무에서는 GPU가 Ceph를 직접 느리게 읽는 상황을 줄이기 위해 다음 구조를 많이 검토합니다.

```text
원본 데이터 저장: RGW/S3 또는 CephFS
학습 시작 전: Local NVMe로 dataset staging/cache
학습 중: Local NVMe에서 빠르게 읽기
checkpoint: 주기적으로 CephFS/RGW에 저장
최종 모델: RGW/S3 또는 model registry에 저장
```

이 구조의 장점은 GPU가 매 batch마다 Ceph latency를 직접 기다리지 않는다는 것입니다. 단점은 데이터 staging 시간, local NVMe 용량 관리, cache consistency 문제가 추가된다는 것입니다.

### 8-7) Ceph와 GPU 네트워크 분리

GPU 클러스터에서는 스토리지 트래픽과 학습 통신 트래픽이 충돌할 수 있습니다. Ceph client I/O, OSD replication, NCCL, RDMA, 서비스 트래픽이 같은 NIC 또는 같은 switch fabric을 사용하면 병목 원인 파악이 어려워집니다.

가능하면 다음처럼 네트워크를 분리하는 것이 좋습니다.

| 네트워크 | 용도 |
|---|---|
| Kubernetes management network | API server, kubelet, control-plane 통신 |
| Pod/service network | 일반 Pod 통신 |
| Ceph public network | client가 Ceph에 접근하는 트래픽 |
| Ceph cluster network | OSD replication/recovery/backfill |
| GPU RDMA/NCCL network | 분산 학습 통신 |

모든 환경에서 물리적으로 완전 분리할 수는 없지만, VLAN, QoS, 별도 NIC, 별도 switch, SR-IOV/RDMA 인터페이스를 통해 분리하는 방향을 검토해야 합니다.

## 9. ceph-csi와 GPU Kubernetes 워크로드

### 9-1) ceph-csi의 역할

ceph-csi는 Kubernetes에서 Ceph RBD와 CephFS를 사용할 수 있게 해주는 CSI driver입니다. Kubernetes 사용자는 PVC를 만들고, ceph-csi가 Ceph pool 또는 CephFS subvolume에 실제 volume을 생성합니다.

GPU Pod도 일반 Pod와 마찬가지로 PVC를 사용할 수 있습니다.

```text
GPU Pod
→ PVC 요청
→ StorageClass가 ceph-csi-rbd 또는 ceph-csi-cephfs 사용
→ Ceph에 RBD image 또는 CephFS subvolume 생성
→ Node에서 mount/map
→ Pod 내부에서 dataset/checkpoint/workspace 사용
```

### 9-2) RBD StorageClass와 CephFS StorageClass 선택

| 요구사항 | 권장 |
|---|---|
| 단일 Pod가 block device처럼 사용 | RBD |
| 여러 Pod가 같은 경로 공유 | CephFS |
| VM disk/OpenStack Cinder 유사 구조 | RBD |
| shared dataset | CephFS 또는 RGW + local cache |
| checkpoint archive | CephFS 또는 RGW |
| 고성능 임시 scratch | Local NVMe 권장 |

### 9-3) PVC mount 장애와 GPU Pod 장애 연결

GPU Pod가 `ContainerCreating`에서 멈춰 있을 때 실제 원인은 GPU가 아니라 PVC mount 실패일 수 있습니다. 따라서 GPU Pod 장애를 볼 때는 GPU resource뿐 아니라 PVC 상태도 같이 봐야 합니다.

확인 흐름은 다음과 같습니다.

```text
Pod 상태 확인
→ Events에서 FailedMount/FailedAttach 확인
→ PVC Bound 여부 확인
→ PV와 StorageClass 확인
→ ceph-csi controller log 확인
→ ceph-csi node plugin log 확인
→ Ceph cluster health 확인
→ Ceph pool/image/subvolume 존재 여부 확인
→ node의 mount/map 상태 확인
```

### 9-4) GPU 학습과 VolumeSnapshot

학습 checkpoint는 애플리케이션 레벨에서 저장하는 경우가 많지만, PVC snapshot도 운영상 유용할 수 있습니다. Kubernetes CSI snapshot을 사용하면 RBD snapshot 또는 CephFS snapshot과 연계할 수 있습니다.

다만 snapshot은 백업과 동일하지 않습니다. 같은 Ceph 클러스터 내부 snapshot은 클러스터 장애나 pool 장애에 대한 완전한 보호가 아닙니다. 중요한 모델 artifact는 RGW/S3, 외부 백업, 원격 복제 등을 함께 고려해야 합니다.

### 9-5) Volume Expansion

학습 중 workspace나 checkpoint 저장소가 부족해질 수 있습니다. StorageClass에서 volume expansion을 허용하면 PVC 용량 확장이 가능합니다. 하지만 파일시스템 확장, RBD image resize, Pod 재시작 필요 여부, application 동작을 함께 고려해야 합니다.

## 10. GPU 학습/추론 워크로드별 Kubernetes + Ceph 설계

### 10-1) 단일 GPU 학습 Pod

단일 GPU 학습 Pod는 가장 단순한 구조입니다.

```text
GPU Node
→ GPU Operator
→ Pod requests nvidia.com/gpu: 1
→ PVC는 RBD 또는 CephFS 사용
→ 데이터셋은 CephFS/RGW에서 읽거나 local cache 사용
```

권장 구조는 다음과 같습니다.

| 항목 | 권장 |
|---|---|
| GPU 할당 | `nvidia.com/gpu: 1` |
| 학습 코드 | container image로 고정 |
| 데이터셋 | CephFS 또는 RGW에서 local NVMe로 staging |
| checkpoint | CephFS/RGW에 주기적 저장 |
| 로그 | stdout + object/log storage |
| 모니터링 | DCGM + Prometheus + Grafana |

### 10-2) 멀티 GPU 단일 노드 학습

한 서버에 GPU가 여러 장 있고 하나의 학습 job이 여러 GPU를 사용하는 구조입니다. 이때는 NVLink, PCIe topology, NCCL, CPU NUMA가 중요합니다.

```text
Single GPU Node
├─ GPU0
├─ GPU1
├─ GPU2
└─ GPU3

Pod requests nvidia.com/gpu: 4
NCCL이 GPU 간 통신 수행
```

Ceph는 데이터 공급과 checkpoint 저장을 담당합니다. 학습 데이터 읽기가 느리면 4개의 GPU가 모두 기다리게 되므로 local NVMe cache를 적극 검토합니다.

### 10-3) 멀티 노드 분산 학습

여러 GPU 노드가 하나의 job에 참여하는 구조입니다. 이때는 Kubernetes scheduling만으로는 부족할 수 있고, Kubeflow, Volcano, Kueue, Ray, PyTorchJob, MPIJob 같은 batch/distributed training 도구가 필요할 수 있습니다.

```text
GPU Node A: GPU 4장
GPU Node B: GPU 4장
GPU Node C: GPU 4장

총 12 GPU로 하나의 분산 학습 job 실행
```

중요 요소는 다음과 같습니다.

| 요소 | 이유 |
|---|---|
| NCCL | GPU 간 collective communication |
| RDMA/RoCE/InfiniBand | 서버 간 낮은 latency와 높은 bandwidth |
| SR-IOV/Multus | Pod에 고성능 network interface 제공 |
| CephFS/RGW | dataset/checkpoint 공유 |
| Local NVMe cache | 학습 중 I/O 병목 감소 |
| Scheduler | gang scheduling, queue, quota 필요 |

### 10-4) GPU 추론 서비스

추론 서비스는 학습과 다르게 latency, 동시성, autoscaling, 모델 로딩, rollout이 중요합니다.

```text
Ingress/Gateway
→ Inference Service Pod
→ GPU 또는 MIG slice 사용
→ 모델 artifact는 RGW/S3/CephFS에서 로드
→ 요청 처리 후 응답 반환
```

추론 서비스에서는 MIG나 Time-Slicing이 유용할 수 있습니다. 큰 GPU 하나를 작은 서비스 여러 개가 나누어 사용할 수 있기 때문입니다.

| 항목 | 학습 | 추론 |
|---|---|---|
| 핵심 지표 | throughput, epoch time | latency, QPS, concurrency |
| GPU 사용 | 장시간 고점유 | 요청량에 따라 변동 |
| 스토리지 | dataset, checkpoint | model artifact, cache |
| 네트워크 | NCCL/RDMA 중요 | API latency, model server 중요 |
| GPU 공유 | 보통 독점 선호 | MIG/Time-Slicing 검토 가능 |

## 11. GPU Scheduler: Kubernetes, Volcano, Kueue, Ray, Slurm

### 11-1) 왜 GPU Scheduler가 필요한가

GPU는 비싸고 한정된 자원입니다. 단순히 GPU가 있는 노드에 Pod를 올리는 것만으로는 충분하지 않습니다. 여러 사용자, 여러 팀, 여러 학습 job, 여러 추론 서비스를 운영하려면 queue, quota, priority, gang scheduling, fair sharing이 필요합니다.

### 11-2) 기본 Kubernetes Scheduler

기본 Kubernetes Scheduler는 `nvidia.com/gpu` 리소스가 남아 있는 노드에 Pod를 배치할 수 있습니다. 단순 GPU Pod 운영에는 충분할 수 있습니다. 하지만 분산 학습처럼 여러 Pod가 동시에 떠야 하는 job에는 한계가 있습니다.

예를 들어 worker Pod 8개가 모두 떠야 학습이 시작되는데, 일부만 뜨고 일부는 Pending이면 GPU가 낭비됩니다. 이런 경우 gang scheduling이 필요합니다.

### 11-3) Volcano

Volcano는 Kubernetes-native batch scheduler입니다. AI, big data, HPC workload에서 gang scheduling, queue, priority 등을 제공하기 위해 사용됩니다. 분산 학습 job을 Kubernetes에서 운영할 때 검토할 수 있습니다.

### 11-4) Kueue

Kueue는 Kubernetes에서 batch job queueing을 관리하기 위한 도구입니다. namespace/team별 quota, queue, admission control을 통해 한정된 GPU 자원을 더 체계적으로 사용할 수 있게 합니다.

### 11-5) Ray

Ray는 Python 친화적인 분산 컴퓨팅 프레임워크입니다. Ray cluster는 CPU/GPU 리소스를 추상화하고, Ray task/actor/train/tune/serve가 해당 리소스를 사용합니다. Kubernetes에서는 KubeRay를 통해 Ray cluster를 운영할 수 있습니다.

Ray는 다음에 적합합니다.

| 용도 | 설명 |
|---|---|
| 병렬 전처리 | 여러 CPU/GPU에서 데이터 처리 |
| 하이퍼파라미터 튜닝 | 여러 실험 병렬 실행 |
| 분산 학습 | Ray Train |
| 모델 서빙 | Ray Serve |
| Python 개발자 친화 | 코드 레벨에서 분산 처리 작성 |

### 11-6) Slurm

Slurm은 HPC에서 많이 사용하는 workload manager입니다. 연구소, 슈퍼컴퓨터, 대규모 GPU cluster에서 job queue, partition, fair-share, GPU allocation을 관리합니다.

Kubernetes와 Slurm은 경쟁 관계라기보다 용도가 다릅니다.

| 구분 | Kubernetes | Slurm | Ray |
|---|---|---|---|
| 주 목적 | 컨테이너 오케스트레이션 | HPC job scheduling | Python 분산 실행 |
| GPU 관리 | Device Plugin/Operator | GRES/partition | Ray resource |
| 강점 | 서비스 운영, MLOps, 추론 | Batch 학습, HPC, queue | 개발자 친화, ML workflow |
| Ceph 연동 | PVC/CSI | shared filesystem mount | object/file storage 사용 |

## 12. OpenStack, Ceph, Kubernetes, GPU 연계

### 12-1) OpenStack에서 GPU를 쓰는 방식

OpenStack에서 GPU를 사용하는 대표 방식은 다음과 같습니다.

| 방식 | 설명 |
|---|---|
| PCI Passthrough | 물리 GPU를 VM에 직접 할당 |
| vGPU | 물리 GPU를 가상 GPU로 나누어 VM에 할당 |
| SR-IOV NIC | VM에 고성능 NIC VF 할당 |
| Ceph RBD | VM disk, Cinder volume, Glance image backend |

OpenStack Nova compute node에서 GPU Passthrough를 사용하려면 BIOS IOMMU 설정, kernel boot parameter, VFIO binding, Nova PCI alias, flavor extra spec, placement 설정 등이 필요합니다. 구체 설정은 OpenStack 버전과 배포판에 따라 달라지므로 운영 환경 문서를 기준으로 해야 합니다.

### 12-2) OpenStack과 Kubernetes가 함께 있는 환경

회사에서 OpenStack과 Kubernetes를 모두 다룬다면 다음 구조가 가능합니다.

```text
Ceph Cluster
├─ RBD → OpenStack Cinder/Nova/Glance
├─ RBD → Kubernetes ceph-csi-rbd PVC
├─ CephFS → Kubernetes ceph-csi-cephfs PVC
└─ RGW → S3 API, artifact, backup

GPU Resources
├─ OpenStack VM에 GPU Passthrough/vGPU 제공
└─ Kubernetes Pod에 GPU Operator/Device Plugin으로 제공

Network
├─ OpenStack Neutron SR-IOV
└─ Kubernetes Multus/SR-IOV/RDMA
```

### 12-3) GPU VM과 GPU Pod 선택 기준

| 기준 | GPU VM | GPU Pod |
|---|---|---|
| 격리 | VM 격리가 강함 | 컨테이너 격리 |
| 운영 유연성 | OS 단위 커스터마이징 쉬움 | 배포/스케일링/자동화 유리 |
| 성능 | Passthrough 시 native에 가까움 | native에 가까움 |
| 멀티테넌시 | 프로젝트별 VM 제공에 유리 | namespace/quota 기반 플랫폼에 유리 |
| MLOps/서빙 | 별도 구성 필요 | Kubernetes 생태계 활용 유리 |
| 장기 학습 job | Slurm/VM도 가능 | Kubernetes batch scheduler 필요 |

## 13. GPU 모니터링과 Ceph/Kubernetes 통합 관측성

### 13-1) GPU 지표

| 지표 | 의미 |
|---|---|
| GPU Utilization | GPU 연산 사용률 |
| Memory Utilization | VRAM 사용률 |
| FB Memory Used/Free | 실제 VRAM 사용량 |
| Temperature | GPU 온도 |
| Power Draw | 전력 사용량 |
| SM Utilization | Streaming Multiprocessor 사용률 |
| Tensor Core Utilization | Tensor Core 사용률 가능 시 확인 |

### 13-2) Kubernetes 지표

GPU Pod 운영에서는 Kubernetes 지표도 같이 봐야 합니다.

| 지표 | 의미 |
|---|---|
| Pod Pending | GPU 부족 또는 scheduling 문제 |
| Pod OOMKilled | CPU/RAM OOM. GPU OOM과 다름 |
| ContainerCreating | image pull 또는 volume mount 문제 |
| PVC Pending | StorageClass/CSI/Ceph 문제 |
| Node Allocatable GPU | GPU 리소스 가용량 |
| Namespace GPU 사용량 | 팀/사용자별 quota 관리 |
| GPU Pod restart | driver/CUDA/app 문제 가능 |

### 13-3) Ceph 지표

GPU workload와 Ceph를 같이 운영하면 다음 지표가 중요합니다.

| 지표 | 의미 |
|---|---|
| Ceph Health | 전체 클러스터 상태 |
| OSD up/in | 저장소 노드 상태 |
| PG active+clean | 데이터 배치 정상 여부 |
| PG degraded/recovering | 복구/성능 저하 가능성 |
| OSD latency | 학습 I/O 지연 원인 |
| MDS latency | CephFS metadata 병목 |
| CephFS client count | 클라이언트 증가와 MDS 부하 |
| RGW request latency | S3 기반 dataset/artifact 접근 지연 |
| pool nearfull/full | checkpoint 실패 가능성 |
| recovery/backfill rate | 학습 I/O와 충돌 가능 |

### 13-4) 통합 대시보드에서 봐야 하는 것

실무에서는 GPU만 따로 보는 것보다 GPU, Pod, PVC, Ceph, Network를 한 화면에서 연결해 봐야 합니다.

예를 들어 GPU utilization이 낮을 때 다음을 같이 봅니다.

```text
GPU Utilization 낮음
→ GPU Pod는 Running인가?
→ 학습 프로세스는 살아 있는가?
→ VRAM은 사용 중인가?
→ CPU 사용률이 높은가?
→ PVC/CephFS read latency가 높은가?
→ Ceph recovery/backfill 중인가?
→ Network throughput/drop이 있는가?
→ NCCL timeout 또는 dataloader stall이 있는가?
```

## 14. 장애 대응 시나리오

### 14-1) GPU Pod가 Pending

가능 원인은 다음과 같습니다.

| 원인 | 설명 |
|---|---|
| GPU 리소스 부족 | 요청한 `nvidia.com/gpu`가 남아 있는 node 없음 |
| nodeSelector 불일치 | 특정 GPU 모델 label이 맞지 않음 |
| taint/toleration 불일치 | GPU node taint를 Pod가 toleration하지 않음 |
| MIG profile 부족 | 요청한 MIG 리소스가 없음 |
| Scheduler 정책 | queue/gang scheduling 조건 미충족 |

### 14-2) GPU Pod가 ContainerCreating에서 멈춤

GPU 문제처럼 보이지만 volume mount 문제일 수 있습니다.

| 원인 | 설명 |
|---|---|
| PVC mount 실패 | ceph-csi node plugin 오류 |
| RBD map 실패 | Ceph auth, network, image lock 문제 |
| CephFS mount 실패 | MDS 문제, secret, network 문제 |
| image pull 지연 | GPU image가 크고 registry가 느림 |
| container runtime GPU 설정 문제 | NVIDIA runtime/CDI 문제 |

### 14-3) Pod 안에서 GPU가 안 보임

| 계층 | 확인 내용 |
|---|---|
| Host | NVIDIA Driver 정상 여부 |
| Kubernetes Node | `nvidia.com/gpu` 리소스 표시 여부 |
| Device Plugin | Pod 상태와 로그 |
| Container Runtime | NVIDIA Container Toolkit/CDI 설정 |
| Pod Spec | GPU resource limit 지정 여부 |
| Security | privileged, device access, runtimeClass 설정 필요 여부 |

### 14-4) CUDA out of memory

CUDA OOM은 GPU VRAM 부족입니다. Kubernetes OOMKilled와 다릅니다. Kubernetes OOMKilled는 container memory limit 초과이고, CUDA OOM은 GPU memory 할당 실패입니다.

가능 원인은 다음과 같습니다.

| 원인 | 조치 |
|---|---|
| batch size 과다 | batch size 축소 |
| 모델이 큼 | 더 큰 GPU, model parallel, quantization 검토 |
| 여러 프로세스가 같은 GPU 사용 | GPU 점유 프로세스 정리 |
| MIG slice가 작음 | 더 큰 MIG profile 사용 |
| memory leak | 코드/프레임워크 사용 패턴 확인 |
| fragmentation | 프로세스 재시작, memory allocator 설정 검토 |

### 14-5) NCCL Timeout

멀티 GPU/멀티 노드 학습에서 흔합니다.

| 원인 | 설명 |
|---|---|
| 네트워크 불안정 | packet drop, MTU 불일치, firewall |
| 잘못된 interface 선택 | NCCL이 관리망 interface를 사용 |
| RDMA 설정 실패 | RoCE/InfiniBand/RDMA device 문제 |
| Pod 간 DNS/통신 문제 | Kubernetes network 문제 |
| 일부 worker 지연 | 느린 노드, 느린 스토리지, GPU 장애 |
| Ceph I/O 병목 | worker 중 일부가 데이터 로딩에서 지연 |

### 14-6) Ceph 병목으로 GPU Utilization 저하

증상은 GPU가 낮은 사용률을 보이고 학습 속도가 느린 경우입니다.

확인 순서:

1. GPU utilization과 VRAM 사용량 확인
2. CPU dataloader 사용률 확인
3. 학습 Pod의 PVC 종류 확인
4. CephFS/RBD/RGW latency 확인
5. Ceph health, PG 상태, OSD latency 확인
6. recovery/backfill/scrub 진행 여부 확인
7. 작은 파일 metadata 병목 여부 확인
8. local NVMe staging 또는 dataset sharding 검토

## 15. 실무 설계 패턴

### 15-1) 소규모 GPU Kubernetes 클러스터

```text
GPU Worker Node 1~3대
→ GPU Operator
→ ceph-csi-rbd / ceph-csi-cephfs
→ Prometheus + Grafana + DCGM Exporter
→ Namespace별 GPU quota
```

적합한 용도는 개발, 테스트, 소규모 학습, 내부 추론 서비스입니다.

### 15-2) GPU 추론 플랫폼

```text
Ingress/API Gateway
→ Inference Pod
→ GPU/MIG/Time-Slicing
→ Model artifact from RGW/S3/CephFS
→ Metrics: latency, QPS, GPU utilization
```

추론 플랫폼에서는 GPU를 통째로 독점하는 것보다 MIG 또는 Time-Slicing을 통해 활용률을 높이는 것이 중요할 수 있습니다.

### 15-3) 분산 학습 플랫폼

```text
GPU Nodes
→ GPU Operator
→ Multus + SR-IOV/RDMA Network
→ Ceph/RGW dataset storage
→ Local NVMe cache
→ Volcano/Kueue/Ray/Kubeflow Training Operator
→ DCGM + Ceph + Network monitoring
```

분산 학습에서는 GPU뿐 아니라 네트워크와 스토리지 병목이 핵심입니다.

### 15-4) OpenStack GPU VM + Kubernetes GPU Pod 혼합 환경

```text
Ceph Cluster 공용 사용
├─ OpenStack: VM disk via RBD, GPU Passthrough/vGPU
└─ Kubernetes: PVC via ceph-csi, GPU Pod via GPU Operator

Network
├─ OpenStack Neutron SR-IOV
└─ Kubernetes Multus/SR-IOV/RDMA
```

이 구조에서는 Ceph와 네트워크를 여러 플랫폼이 공유하므로 resource isolation, QoS, monitoring이 중요합니다.

## 17. 우선순위 학습 순서

### 17-1) 1단계: Kubernetes GPU 기본

먼저 GPU Operator, NVIDIA Device Plugin, NVIDIA Container Toolkit, `nvidia.com/gpu` 리소스, GPU Pod scheduling을 이해해야 합니다. 이 단계가 되어야 Kubernetes에서 GPU Pod가 왜 Pending인지, 왜 컨테이너 안에서 GPU가 안 보이는지 판단할 수 있습니다.

### 17-2) 2단계: Ceph CSI와 GPU 데이터 경로

ceph-csi-rbd, ceph-csi-cephfs, PVC, StorageClass, RBD, CephFS, RGW를 GPU workload와 연결해서 이해해야 합니다. GPU 학습에서 데이터가 어디서 읽히고, checkpoint가 어디로 쓰이며, Ceph가 병목이 되는지 볼 수 있어야 합니다.

### 17-3) 3단계: GPU 공유와 격리

MIG, Time-Slicing, MPS, GPU Passthrough, vGPU 차이를 알아야 합니다. 어떤 상황에서 GPU를 통째로 주고, 어떤 상황에서 나누어 쓸지 판단할 수 있어야 합니다.

### 17-4) 4단계: SR-IOV, Multus, RDMA

대규모 GPU 클러스터나 고성능 네트워크가 필요한 환경에서는 SR-IOV, Multus, RDMA, RoCE, InfiniBand, GPUDirect RDMA를 이해해야 합니다.

### 17-5) 5단계: Scheduler와 운영

Volcano, Kueue, Ray, Slurm, Kubernetes Scheduler의 차이를 알고, GPU 자원을 팀/사용자/job 단위로 어떻게 할당할지 설계해야 합니다.
