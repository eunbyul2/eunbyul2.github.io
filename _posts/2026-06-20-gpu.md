## 1. GPU 컴퓨팅 기초

### 1-1) CPU와 GPU 아키텍처 차이

CPU와 GPU는 둘 다 계산 장치지만 설계 목적이 다르다. CPU는 운영체제, 데이터베이스, 웹 서버, 네트워크 처리, 파일 시스템, 컨테이너 런타임처럼 다양한 종류의 작업을 안정적으로 처리하는 범용 프로세서다. 반면 GPU는 동일하거나 유사한 연산을 대량의 데이터에 반복 적용하는 병렬 처리 장치다. 딥러닝, 영상 처리, 행렬 연산, 시뮬레이션 같은 workload는 많은 데이터를 동시에 처리해야 하므로 GPU에 적합하다.

CPU는 상대적으로 적은 수의 고성능 코어를 가진다. 각 코어는 복잡한 제어 흐름, branch prediction, out-of-order execution, 큰 cache 계층을 통해 다양한 명령을 빠르게 처리한다. 그래서 웹 서버, DB, OS kernel, Kubernetes control plane처럼 복잡한 조건 분기와 I/O가 섞인 작업에 강하다.

GPU는 수많은 작은 연산 유닛을 가진다. GPU는 개별 코어 하나의 범용 처리 능력보다, 많은 스레드가 동시에 같은 형태의 명령을 실행하는 처리량을 극대화하도록 설계되어 있다. 그래서 한 번에 수천~수만 개의 스레드를 실행하면서 대규모 행렬곱, 벡터 연산, 이미지 픽셀 처리, 딥러닝 tensor 연산을 빠르게 처리한다.

인프라 엔지니어 관점에서 중요한 차이는 “GPU가 빠르다”가 아니라, **GPU가 빠르게 일하려면 데이터를 계속 공급해야 한다**는 점이다. CPU가 데이터를 준비하지 못하거나, 스토리지가 느리거나, CPU RAM에서 GPU VRAM으로 이동하는 경로가 느리거나, GPU 간 통신이 느리면 GPU는 계산하지 못하고 대기한다. GPU 인프라 운영의 핵심은 GPU 자체뿐 아니라 CPU, RAM, VRAM, PCIe, NVLink, 네트워크, 스토리지까지 하나의 pipeline으로 보는 것이다.

| 구분 | CPU | GPU |
|---|---|---|
| 설계 목적 | 범용 연산, 복잡한 제어 흐름 | 대량 병렬 연산, 높은 처리량 |
| 코어 수 | 적음 | 매우 많음 |
| 강점 | OS, DB, API, 네트워크, 파일 시스템 | 행렬곱, 벡터 연산, 딥러닝, 영상 처리 |
| 메모리 | 시스템 RAM 중심 | GPU 전용 VRAM 중심 |
| 병목 | CPU core, memory, I/O, lock | VRAM, memory bandwidth, PCIe, GPU 간 통신 |
| 운영 관점 | 프로세스, I/O, 네트워크, 커널 | GPU utilization, VRAM, 온도, 전력, ECC, CUDA stack |

### 1-2) CPU 구조

CPU는 일반적으로 core, cache, memory controller, PCIe controller, interconnect, socket 구조로 구성된다. 서버 CPU는 Intel Xeon, AMD EPYC 계열이 많이 사용된다. 고성능 GPU 서버에서는 CPU가 단순 보조 장치가 아니다. CPU는 데이터 로딩, 전처리, network stack, storage I/O, container runtime, scheduler agent, driver 호출을 담당한다.

GPU 서버에서 CPU가 부족하면 GPU 사용률이 낮아질 수 있다. 예를 들어 이미지 학습에서 CPU가 JPEG decode, augmentation, batch 구성 작업을 충분히 빠르게 처리하지 못하면 GPU는 데이터를 기다린다. 이 경우 GPU utilization이 낮게 나오지만 실제 병목은 GPU가 아니라 CPU와 데이터 pipeline이다.

인프라 엔지니어가 CPU에서 봐야 할 항목은 다음과 같다.

| 항목 | 의미 |
|---|---|
| Socket 수 | CPU 물리 소켓 수. NUMA 구조와 PCIe root complex에 영향 |
| Core/Thread 수 | 데이터 전처리, worker process, system daemon 처리 능력 |
| NUMA node | CPU, RAM, PCIe 장치의 근접성 판단 기준 |
| PCIe lane 수 | GPU, NIC, NVMe를 충분히 연결할 수 있는지 결정 |
| Memory channel | RAM 대역폭에 영향 |
| CPU-GPU topology | 특정 GPU가 어느 CPU socket에 가까운지 판단 |

### 1-3) GPU 구조

GPU는 여러 개의 Streaming Multiprocessor(SM)로 구성된다. 각 SM 안에는 CUDA Core, Tensor Core, warp scheduler, register file, shared memory/L1 cache 등이 들어간다. GPU workload는 수많은 thread로 쪼개져 실행되고, 이 thread들은 block과 grid 단위로 구성된다.

GPU 구조를 단순화하면 다음과 같다.

```text
GPU
├─ Global Memory / VRAM
├─ L2 Cache
├─ Streaming Multiprocessor(SM) 0
│  ├─ CUDA Core
│  ├─ Tensor Core
│  ├─ Warp Scheduler
│  ├─ Register File
│  └─ Shared Memory / L1 Cache
├─ Streaming Multiprocessor(SM) 1
└─ ...
```

인프라 엔지니어가 CUDA kernel을 직접 최적화하지 않더라도 이 구조를 알아야 하는 이유는 성능 지표를 해석하기 위해서다. 예를 들어 SM utilization이 낮으면 GPU compute unit이 충분히 사용되지 않는다는 뜻이고, memory bandwidth utilization이 높으면 연산보다 메모리 접근이 병목일 수 있다. Tensor Core utilization이 낮으면 AI workload가 Tensor Core를 제대로 사용하지 못하고 있을 수 있다.

### 1-4) Core vs CUDA Core

CPU core와 CUDA Core는 같은 의미가 아니다. CPU core는 독립적으로 복잡한 명령을 처리할 수 있는 강한 범용 코어다. CUDA Core는 GPU 내부에서 대량 병렬 연산을 수행하는 작은 연산 유닛이다. 따라서 CPU core 1개와 CUDA Core 1개를 성능으로 직접 비교하면 안 된다.

CUDA Core는 주로 FP32/INT 연산 같은 기본 산술 연산을 처리한다. 딥러닝에서 중요한 행렬 연산은 CUDA Core뿐 아니라 Tensor Core가 크게 관여한다. 최신 NVIDIA GPU에서는 Tensor Core가 FP16, BF16, TF32, FP8, INT8 같은 연산을 가속한다. 따라서 GPU 스펙을 볼 때 CUDA Core 수만 보면 안 되고, Tensor Core 성능, VRAM 용량, memory bandwidth, interconnect, 전력까지 함께 봐야 한다.

### 1-5) Thread, Warp, Block, Grid

CUDA 프로그래밍 모델에서는 GPU에서 실행되는 작은 실행 단위를 thread라고 한다. 여러 thread가 모여 block을 구성하고, 여러 block이 모여 grid를 구성한다. GPU kernel은 grid 단위로 실행되며, 각 thread는 자기 index를 기준으로 서로 다른 데이터를 처리한다.

```text
Grid
├─ Block 0
│  ├─ Thread 0
│  ├─ Thread 1
│  └─ ...
├─ Block 1
└─ ...
```

NVIDIA GPU에서 warp는 일반적으로 32개 thread의 묶음이다. 같은 warp에 속한 thread들은 SIMT 방식으로 명령을 실행한다. warp 단위 실행 구조 때문에 branch divergence가 발생하면 성능이 떨어질 수 있다. 예를 들어 같은 warp 안의 thread들이 서로 다른 조건문 경로를 타면 GPU는 각 경로를 순차적으로 처리해야 하므로 병렬 효율이 낮아진다.

인프라 엔지니어가 이걸 알아야 하는 이유는 애플리케이션 성능 병목을 개발자와 대화할 때 필요하기 때문이다. GPU 사용률이 낮을 때 단순히 “서버 문제”라고 볼 수 없고, kernel occupancy, memory access pattern, batch size, branch divergence 같은 코드 레벨 문제일 수도 있다.

### 1-6) Streaming Multiprocessor(SM)

SM은 NVIDIA GPU의 핵심 실행 단위다. SM은 warp를 스케줄링하고 CUDA Core/Tensor Core에 작업을 배치한다. GPU에는 여러 SM이 있고, 각 SM이 동시에 여러 warp를 실행한다. GPU 성능은 SM 수, SM 내부 연산 유닛, clock, memory bandwidth, Tensor Core 성능에 영향을 받는다.

운영 관점에서 SM utilization은 중요한 지표다. SM utilization이 낮으면 GPU의 compute unit이 충분히 사용되지 않는다는 뜻이다. 원인은 batch size가 작거나, 데이터 로딩이 느리거나, CPU 전처리가 느리거나, kernel launch overhead가 크거나, 애플리케이션이 GPU 병렬성을 충분히 활용하지 못하는 것일 수 있다.

### 1-7) SIMD와 SIMT

SIMD는 Single Instruction, Multiple Data다. 하나의 명령을 여러 데이터에 동시에 적용하는 방식이다. CPU의 vector instruction이나 GPU의 병렬 실행을 설명할 때 사용된다.

SIMT는 Single Instruction, Multiple Threads다. NVIDIA CUDA의 실행 모델을 설명하는 용어다. GPU는 여러 thread를 실행하지만, warp 단위에서는 같은 명령을 여러 thread가 실행한다. 각 thread는 자기 register와 thread index를 가지므로 프로그래밍 모델상으로는 독립 thread처럼 보인다.

SIMD와 SIMT의 차이를 단순화하면 다음과 같다.

| 구분 | SIMD | SIMT |
|---|---|---|
| 관점 | 명령이 여러 데이터에 적용 | 여러 thread가 같은 명령을 실행 |
| 프로그래밍 모델 | vector 연산 중심 | thread/block/grid 중심 |
| 대표 예 | CPU vector instruction | NVIDIA CUDA warp 실행 |
| 운영상 의미 | 병렬 데이터 처리 | warp, branch divergence, occupancy 이해 필요 |

### 1-8) Cache 구조와 Memory Hierarchy

GPU memory hierarchy는 성능 이해에 매우 중요하다. GPU에는 register, shared memory, L1 cache, L2 cache, global memory, constant memory, texture memory 등이 있다. 인프라 엔지니어가 모든 CUDA memory optimization을 할 필요는 없지만, VRAM 사용량과 memory bandwidth 병목을 이해하려면 계층 구조를 알아야 한다.

대표 계층은 다음과 같다.

| 계층 | 특징 | 운영 관점 |
|---|---|---|
| Register | thread별 가장 빠른 저장 공간 | kernel resource 사용량과 occupancy에 영향 |
| Shared Memory | 같은 block 내 thread들이 공유 | kernel 최적화와 관련 |
| L1 Cache | SM 근처 cache | memory access pattern 영향 |
| L2 Cache | GPU 전체 공유 cache | GPU memory traffic 해석에 중요 |
| Global Memory | VRAM의 큰 영역 | 용량 크지만 register/shared보다 느림 |
| Host RAM | CPU 시스템 메모리 | PCIe/NVLink 경유 필요 |
| Storage | NVMe/NFS/Ceph/Object | 데이터 로딩 pipeline 병목 가능 |

GPU에서 성능이 잘 나오려면 연산 유닛이 놀지 않도록 memory access를 효율적으로 해야 한다. 운영자는 application code를 직접 고치지 않더라도, GPU utilization이 낮고 memory throughput도 낮은지, memory throughput은 높은데 compute utilization이 낮은지, PCIe traffic이 병목인지 등을 지표로 판단해야 한다.

### 1-9) GPU가 AI에 적합한 이유

AI, 특히 딥러닝은 대규모 행렬 연산과 벡터 연산을 반복한다. 신경망의 forward pass와 backward pass는 대부분 tensor 연산이다. GPU는 수많은 thread가 동시에 tensor element를 처리하고, Tensor Core가 행렬곱을 가속하므로 AI workload에 적합하다.

대표 연산은 다음과 같다.

| 연산 | 설명 |
|---|---|
| 행렬곱 | 딥러닝 layer의 핵심 연산 |
| 벡터 연산 | activation, normalization, element-wise 연산 |
| Convolution | CNN, 이미지 모델, 영상 모델에 중요 |
| Attention | Transformer/LLM의 핵심 연산 |
| Reduction | loss 계산, gradient aggregation, AllReduce 등에 사용 |

AI workload는 precision도 중요하다. 반드시 FP32만 쓰는 것이 아니라 FP16, BF16, TF32, FP8, INT8, INT4 등 다양한 정밀도를 사용한다. 정밀도를 낮추면 메모리 사용량과 bandwidth 요구량을 줄이고 Tensor Core 가속을 활용할 수 있다. 다만 정밀도를 낮추면 정확도 손실이나 numerical instability가 생길 수 있어 framework와 모델 특성에 맞게 사용해야 한다.

| 정밀도 | 의미 | 주 용도 |
|---|---|---|
| FP32 | 32-bit floating point | 안정적인 학습, baseline |
| TF32 | NVIDIA Ampere 이후 Tensor Core용 19-bit 수준 형식 | FP32 코드에서 Tensor Core 활용 |
| FP16 | 16-bit floating point | mixed precision 학습/추론 |
| BF16 | bfloat16, exponent 범위가 넓음 | 대규모 학습에서 많이 사용 |
| FP8 | 8-bit floating point | 최신 대규모 GPU 학습/추론 최적화 |
| INT8 | 8-bit integer | 추론 quantization |
| INT4 | 4-bit integer | LLM quantization, 메모리 절약 |

Mixed Precision은 일부 연산은 낮은 정밀도로 빠르게 수행하고, 필요한 부분은 높은 정밀도로 유지하는 방식이다. 학습에서는 loss scaling, optimizer state 관리가 중요하고, 추론에서는 quantization calibration, accuracy validation이 중요하다. 인프라 엔지니어는 precision 선택이 VRAM 사용량, throughput, GPU 종류 선택, TensorRT/Triton 같은 추론 플랫폼 설계에 영향을 준다는 점을 알아야 한다.

## 2. GPU 서버 하드웨어

### 2-1) GPU 서버 구성요소

GPU 서버는 CPU 서버에 GPU만 추가한 장비가 아니다. GPU는 고전력, 고발열, 고대역폭 요구사항을 만들기 때문에 서버 설계 전체가 달라진다. 인프라 엔지니어는 GPU 모델명만 볼 것이 아니라 CPU, NUMA, socket, RAM, GPU, VRAM, PCIe, PSU, cooling, NIC, storage, BMC까지 전체 구성요소를 봐야 한다.

GPU 서버의 기본 데이터 흐름은 다음과 같다.

```text
Storage 또는 Network
→ CPU가 데이터를 RAM으로 로드
→ PCIe/NVLink 경로를 통해 GPU VRAM으로 이동
→ GPU가 VRAM 데이터로 연산
→ 결과를 VRAM, RAM, Storage, Network로 전달
```

이 흐름에서 어느 한 곳이 느리면 GPU는 idle 상태가 된다. 따라서 GPU 서버 운영은 GPU 하나만 보는 것이 아니라 전체 데이터 공급 pipeline을 보는 일이다.

### 2-2) CPU, NUMA, Socket

GPU 서버의 CPU는 GPU를 제어하고 데이터를 공급하는 역할을 한다. CPU socket이 여러 개인 서버에서는 NUMA 구조가 생긴다. NUMA는 CPU socket마다 가까운 RAM과 PCIe 장치가 다르다는 뜻이다. 어떤 GPU는 CPU0에 더 가깝고, 어떤 GPU는 CPU1에 더 가까울 수 있다.

NUMA를 고려하지 않으면 CPU0에서 실행되는 프로세스가 CPU1에 가까운 GPU와 RAM을 사용하면서 불필요한 inter-socket traffic이 생길 수 있다. 멀티 GPU 학습에서는 이 차이가 성능에 영향을 준다. Kubernetes에서는 Topology Manager, CPU Manager, Device Plugin topology 정보가 관련되고, Slurm에서는 CPU binding, memory binding, GPU binding 정책이 관련된다.

### 2-3) RAM

RAM은 GPU에 올라가기 전 데이터가 준비되는 공간이다. 데이터셋 cache, dataloader buffer, preprocessing 결과, OS page cache, container memory가 RAM을 사용한다. GPU가 많을수록 동시에 공급해야 할 데이터도 많아지므로 RAM 용량과 memory bandwidth가 중요하다.

RAM이 부족하면 swap이 발생하고, swap이 발생하면 GPU는 데이터를 기다린다. 학습 서버에서는 swap을 완전히 끄거나, 최소한 swap 발생을 심각한 장애 신호로 취급하는 경우가 많다. 단순히 GPU VRAM만 충분하다고 해서 학습이 빠르게 도는 것이 아니다.

### 2-4) GPU와 VRAM

GPU는 연산 장치이고 VRAM은 GPU 전용 메모리다. 모델 파라미터, input batch, activation, gradient, optimizer state, KV cache가 VRAM에 올라간다. 학습은 추론보다 VRAM을 훨씬 많이 사용한다. 같은 모델이라도 추론은 가능하지만 학습은 VRAM 부족으로 실패할 수 있다.

GPU 선택 시 확인할 항목은 다음과 같다.

| 항목 | 확인 이유 |
|---|---|
| VRAM 용량 | 모델과 batch가 올라가는지 결정 |
| Memory bandwidth | GPU core가 데이터를 읽고 쓰는 속도 |
| Tensor Core | AI 행렬 연산 가속 여부 |
| 지원 정밀도 | FP32, FP16, BF16, FP8, INT8 등 |
| ECC | 데이터센터 안정성, 메모리 오류 감지/수정 |
| TDP | 전력/냉각 설계 기준 |
| Form factor | PCIe인지 SXM인지 |
| MIG 지원 | GPU 분할 운영 가능 여부 |
| NVLink/NVSwitch | 멀티 GPU 통신 성능 |

### 2-5) PCIe와 Lane

PCIe는 CPU와 GPU, NIC, NVMe 같은 장치를 연결하는 표준 인터페이스다. GPU 서버에서 PCIe lane 수와 topology는 중요하다. GPU 여러 장, 고속 NIC, NVMe를 동시에 연결하려면 충분한 PCIe lane이 필요하다.

PCIe에서 중요한 개념은 lane, generation, switch, root complex다. lane은 데이터 전송 경로이며, x16 GPU는 16개 lane을 사용한다. PCIe generation이 높을수록 lane당 bandwidth가 높다. 하지만 실제 성능은 generation뿐 아니라 x16/x8 link width, PCIe switch 구성, CPU root complex 연결 구조에 영향을 받는다.

### 2-6) PSU와 전력

GPU 서버는 전력 요구량이 크다. 고성능 GPU 여러 장을 장착한 서버는 수 kW 단위의 전력을 사용할 수 있다. 인프라 엔지니어는 GPU TDP뿐 아니라 전체 시스템 전력, PSU 이중화, PDU 용량, rack당 전력 밀도, UPS 용량을 확인해야 한다.

전력 문제가 생기면 GPU reset, 서버 다운, 성능 제한, thermal throttling으로 이어질 수 있다. GPU 서버 도입 전에는 데이터센터 전력 여유가 있는지 반드시 확인해야 한다.

### 2-7) Cooling

GPU는 발열이 크다. 온도가 높으면 clock throttling이 발생하고 성능이 떨어진다. 장시간 고온 상태는 장애 위험도 높인다. 공랭 서버는 airflow, cold aisle/hot aisle, blanking panel, fan speed, inlet temperature가 중요하다. 고밀도 GPU rack은 수랭 또는 direct liquid cooling을 고려해야 한다.

운영자는 GPU temperature뿐 아니라 memory temperature, fan speed, power draw, throttle reason을 함께 봐야 한다. GPU 사용률은 낮은데 온도가 높다면 airflow나 fan 문제일 수 있고, 사용률이 높은데 clock이 낮다면 power limit 또는 thermal throttling을 의심해야 한다.

### 2-8) GPU 종류

GPU는 용도별로 선택해야 한다. 아래 구분은 일반적 기준이며, 구체 스펙과 가격은 시점별로 바뀌므로 도입 시 반드시 최신 벤더 자료를 확인해야 한다.

| GPU | 일반적 용도 |
|---|---|
| RTX 계열 | 개인/소규모 실험, 개발, 비용 민감 환경 |
| L4 | 저전력 추론, 영상 처리, edge/serving |
| L40S | 추론, 그래픽, 중간 규모 학습, 범용 AI 서버 |
| A100 | 데이터센터 학습/추론, MIG 지원, Ampere 기반 |
| H100 | 대규모 AI 학습/추론, Hopper 기반, FP8 등 최신 기능 |
| H200 | H100 계열 후속, 더 큰 HBM 용량/대역폭 중심 |
| B200 | Blackwell 기반 최신 대규모 AI 인프라용 GPU |

인프라 관점에서는 RTX와 데이터센터 GPU의 차이를 알아야 한다. 데이터센터 GPU는 ECC, 안정성, 장시간 workload, MIG, NVLink/SXM, 드라이버 지원, 서버 벤더 검증 측면에서 유리하다. RTX는 비용 효율이 좋지만 데이터센터 운영 기능과 안정성, 보증, 냉각 구조에서 한계가 있을 수 있다.

### 2-9) GPU Form Factor: PCIe GPU와 SXM GPU

PCIe GPU는 일반 확장 카드 형태다. 범용 서버에 장착하기 쉽고 비용이 상대적으로 낮다. 다만 고성능 GPU 간 interconnect나 냉각, 전력 공급 측면에서는 SXM 기반 서버보다 제한적일 수 있다.

SXM GPU는 NVIDIA 데이터센터 GPU에서 사용하는 고성능 모듈 형태다. 전용 서버 플랫폼이 필요하지만, NVLink/NVSwitch, 전력 공급, 냉각 측면에서 대규모 학습 서버에 적합하다. HGX/DGX 계열 서버에서 많이 볼 수 있다.

| 구분 | PCIe GPU | SXM GPU |
|---|---|---|
| 장착 방식 | PCIe card | 전용 모듈/서버 플랫폼 |
| 장점 | 범용성, 도입 용이성 | 고성능, 고대역폭 interconnect |
| 단점 | 통신/냉각/전력 한계 | 서버 선택 제한, 비용 높음 |
| 주요 용도 | 개발, 추론, 중간 규모 학습 | 대규모 학습, 고성능 AI 클러스터 |

### 2-10) GPU 서버 벤더

실무에서는 Dell, HPE, Lenovo, Supermicro, Gigabyte, ASUS, NVIDIA DGX/HGX 기반 시스템 등 다양한 벤더를 검토한다. 벤더 선택 시 단순 가격보다 다음을 봐야 한다.

| 전력 설계 | PSU 이중화, rack power 요구량 |
| PCIe topology | GPU/NIC/NVMe 연결 구조 |

| 유지보수 | 부품 교체, 현장 지원, SLA |

## 3. Linux와 GPU

### 3-1) Linux Kernel과 GPU

GPU 서버는 대부분 Linux 기반으로 운영된다. Linux에서 NVIDIA GPU를 사용하려면 GPU가 PCIe 장치로 인식되고, NVIDIA kernel module이 로드되고, user-space library가 정상적으로 연결되어야 한다.

핵심 구성요소는 다음과 같다.

| 구성요소 | 역할 |
|---|---|
| Linux Kernel | 장치 인식, memory, interrupt, driver module 관리 |
| NVIDIA Kernel Module | GPU 하드웨어 제어 |
| DKMS | kernel update 시 driver module 자동 재빌드 지원 |
| Device File | `/dev/nvidia0`, `/dev/nvidiactl`, `/dev/nvidia-uvm` 등 |
| User-space Library | CUDA runtime, nvidia-smi, framework가 사용하는 library |

### 3-2) DKMS

DKMS는 Dynamic Kernel Module Support다. NVIDIA Driver처럼 kernel module이 필요한 소프트웨어는 kernel version이 바뀌면 module을 다시 빌드해야 한다. DKMS를 사용하면 kernel update 시 module 재빌드를 자동화할 수 있다.

실무에서 자주 발생하는 장애는 kernel update 후 NVIDIA module이 새 kernel에 맞게 빌드되지 않아 `nvidia-smi`가 실패하는 것이다. 원인은 kernel header 누락, gcc 버전 불일치, Secure Boot module signing 문제, DKMS build 실패 등이 있다.

### 3-3) Device File: /dev/nvidia*

NVIDIA Driver가 정상 로드되면 `/dev/nvidia0`, `/dev/nvidia1`, `/dev/nvidiactl`, `/dev/nvidia-uvm` 같은 장치 파일이 생성된다. 애플리케이션은 user-space library를 통해 이 device file에 접근하여 GPU를 사용한다.

컨테이너에서 GPU를 사용하려면 host의 GPU device file과 필요한 library가 container에 적절히 전달되어야 한다. NVIDIA Container Toolkit은 이 과정을 자동화한다.

### 3-4) GPU 인식 과정

Linux에서 GPU가 사용 가능해지는 흐름은 다음과 같다.

```text
서버 부팅
→ BIOS/UEFI가 PCIe 장치 초기화
→ Linux kernel이 PCIe device enumeration 수행
→ NVIDIA Driver kernel module 로드
→ /dev/nvidia* device file 생성
→ nvidia-smi/DCGM/CUDA runtime이 GPU 접근
→ PyTorch/TensorFlow/컨테이너/Kubernetes workload가 GPU 사용
```

이 흐름을 이해하면 장애를 계층적으로 분리할 수 있다. PCIe 장치 자체가 안 보이면 하드웨어/BIOS/BMC 문제 가능성이 크고, PCIe 장치는 보이는데 `nvidia-smi`가 실패하면 driver/kernel module 문제 가능성이 크다. host에서는 보이는데 container에서 안 보이면 container runtime 문제, Kubernetes에서만 안 보이면 device plugin/operator/kubelet 문제 가능성이 크다.

### 3-5) GPU 관련 Linux 명령어

실무에서는 명령어를 외우는 것보다 어떤 계층을 확인하는지 아는 것이 중요하다.

| 명령어 | 확인 계층 | 목적 |
|---|---|---|
| `lspci` | PCIe/하드웨어 | GPU가 PCIe 장치로 보이는지 확인 |
| `lsmod` | Kernel module | NVIDIA module 로드 여부 확인 |
| `dmesg` | Kernel log | PCIe error, driver load error, Xid 확인 |
| `journalctl` | System service log | driver, container runtime, kubelet 로그 확인 |
| `nvidia-smi` | NVIDIA Driver | GPU 상태, driver version, process, memory 확인 |
| `nvidia-smi topo -m` | Topology | GPU-GPU, GPU-NIC, GPU-CPU 연결 관계 확인 |
| `dcgmi` | DCGM | GPU health, diagnostics, metric 확인 |

문서에는 명령어를 길게 나열하지 않지만, 실무에서는 위 명령어들이 계층별 진단의 기준점이 된다.

### 3-6) Secure Boot와 커널 모듈 서명

Secure Boot가 활성화된 서버에서는 서명되지 않은 kernel module 로드가 차단될 수 있다. NVIDIA Driver module이 서명되지 않았거나 등록되지 않은 키로 서명되어 있으면 module load가 실패한다. 조직 보안 정책상 Secure Boot를 유지해야 한다면 MOK 등록, module signing 절차를 운영 표준에 포함해야 한다.

## 4. NVIDIA Driver

### 4-1) Driver 구조

NVIDIA Driver는 kernel-space 구성요소와 user-space 구성요소로 나뉜다. kernel-space module은 GPU 장치 제어, memory management, interrupt handling, device file 제공을 담당한다. user-space library는 CUDA runtime, nvidia-smi, DCGM, PyTorch 같은 프로그램이 GPU에 접근할 수 있도록 한다.

구조는 다음과 같다.

```text
Application / Framework
→ CUDA Runtime / NVIDIA User-space Library
→ NVIDIA Kernel Module
→ PCIe
→ GPU Hardware
```

인프라 엔지니어는 user-space library와 kernel module version mismatch를 주의해야 한다. 패키지 설치가 꼬이면 kernel module은 A 버전인데 user-space library는 B 버전이 되어 `nvidia-smi`가 실패할 수 있다.

### 4-2) Driver 역할

NVIDIA Driver의 주요 역할은 다음과 같다.

| 역할 | 설명 |
|---|---|
| GPU 제어 | GPU 초기화, command submission, scheduling |
| VRAM 관리 | GPU memory allocation/free, address mapping |
| Interrupt 처리 | GPU 작업 완료, 오류, reset 등 event 처리 |
| Power/Clock 관리 | 전력 제한, 성능 상태, thermal 상태 관리 |
| Device file 제공 | `/dev/nvidia*`를 통한 user-space 접근 제공 |
| Error reporting | Xid error, ECC error 등 보고 |

Driver는 CUDA보다 아래 계층이다. CUDA를 설치해도 driver가 없거나 호환되지 않으면 GPU를 사용할 수 없다.

### 4-3) Driver 설치 방식

실무에서 driver 설치 방식은 운영 표준에 따라 달라진다.

| 방식 | 장점 | 단점 |
|---|---|---|
| OS package manager | 업데이트/제거/의존성 관리가 쉬움 | 최신 버전 반영이 늦을 수 있음 |
| NVIDIA runfile | 최신 버전 접근이 빠름 | 패키지 관리와 충돌 가능, 자동화 어려움 |
| GPU Operator | Kubernetes GPU stack과 함께 관리 가능 | 클러스터 운영 복잡도, 정책 검토 필요 |
| 벤더 이미지 | 검증된 조합 사용 가능 | 유연성 제한, 벤더 종속 가능 |

운영 환경에서는 무조건 최신 driver를 설치하는 것이 아니라, GPU 모델, CUDA runtime 요구사항, OS kernel, Kubernetes/GPU Operator 지원 여부를 기준으로 표준 driver version을 정하고 검증 후 배포해야 한다.

### 4-4) Driver Compatibility

NVIDIA CUDA Compatibility 문서에 따르면 CUDA 애플리케이션은 driver와 runtime 호환성을 만족해야 한다. 일반적으로 host driver가 너무 낮으면 최신 CUDA runtime으로 빌드된 container/application이 실행되지 않는다.

실무 원칙은 다음과 같다.

1. GPU 노드 driver version을 표준화한다.
2. container image의 CUDA runtime version을 명시적으로 관리한다.
3. PyTorch/TensorFlow image tag를 `latest`로 두지 않는다.
4. driver upgrade는 검증 환경에서 먼저 수행한다.
5. kernel update와 driver module 재빌드 실패를 방지한다.
6. 노드별 driver/runtime version drift를 모니터링한다.

### 4-5) Driver 장애

자주 발생하는 driver 장애는 다음과 같다.

| 장애 | 증상 | 원인 후보 |
|---|---|---|
| Driver mismatch | `nvidia-smi` 실패 | kernel module/user library 버전 불일치 |
| Kernel upgrade 문제 | 재부팅 후 GPU 안 보임 | DKMS build 실패, header 누락 |
| Secure Boot 문제 | module load 실패 | 서명되지 않은 module 차단 |
| Xid Error | GPU reset, job failure | GPU fault, driver bug, thermal, PCIe error |
| GPU disappeared | 특정 GPU 미인식 | PCIe/riser/power/BMC/hardware issue |
| Persistence mode 문제 | job 시작 지연 | GPU context 초기화 반복 |

### 4-6) Xid Error

Xid Error는 NVIDIA Driver가 보고하는 GPU 오류 코드다. Xid는 단일 원인만 의미하지 않는다. application fault, GPU memory error, PCIe error, thermal issue, driver bug, hardware fault 등 다양한 원인이 가능하다.

Xid 발생 시 확인해야 할 항목은 다음과 같다.

| 확인 항목 | 이유 |
|---|---|
| `dmesg`/journal | Xid code, 발생 시각, GPU UUID 확인 |
| DCGM health | GPU health 상태 확인 |
| ECC error | memory error 여부 확인 |
| temperature/power | thermal/power issue 확인 |
| BMC event log | PCIe, PSU, thermal hardware event 확인 |
| workload pattern | 특정 job에서만 발생하는지 확인 |
| driver version | known issue 여부 확인 |

반복되는 Xid는 운영 장애로 취급해야 한다. 단발성으로 넘기지 말고 GPU UUID, 노드, 시간, workload, driver version, temperature, ECC count를 기록해야 한다.

## 5. CUDA

### 5-1) CUDA Architecture

CUDA는 NVIDIA GPU를 범용 계산에 사용할 수 있게 하는 병렬 컴퓨팅 플랫폼이다. CUDA는 programming model, compiler, runtime, driver API, library ecosystem을 포함한다. 인프라 엔지니어가 CUDA kernel을 직접 작성하지 않더라도, CUDA runtime과 driver의 관계, version compatibility, container 내부 CUDA 구성은 반드시 이해해야 한다.

CUDA 실행 흐름은 다음과 같다.

```text
Application / PyTorch / TensorFlow
→ CUDA Runtime API 또는 CUDA Driver API
→ NVIDIA Driver
→ GPU Hardware
```

CUDA 프로그램은 CPU에서 실행되는 host code와 GPU에서 실행되는 device code로 나뉜다. GPU에서 실행되는 함수를 kernel이라고 한다. CPU가 kernel launch를 요청하면 GPU가 thread/block/grid 단위로 병렬 실행한다.

### 5-2) CUDA Runtime API와 CUDA Driver API

CUDA Runtime API는 개발자가 상대적으로 쉽게 CUDA 기능을 사용할 수 있게 해주는 고수준 API다. CUDA Driver API는 더 저수준 API로 context, module, function, memory를 더 직접 제어한다.

| 구분 | CUDA Runtime API | CUDA Driver API |
|---|---|---|
| 추상화 수준 | 높음 | 낮음 |
| 사용 난이도 | 상대적으로 쉬움 | 복잡함 |
| context 관리 | runtime이 자동 처리 | 명시적 관리 필요 |
| 주요 사용자 | 일반 CUDA 개발자, framework | runtime/framework 내부, 고급 개발자 |

인프라 엔지니어는 이 차이를 깊게 코딩할 필요는 없지만, 애플리케이션 로그에서 runtime error와 driver error가 다르게 나타날 수 있음을 알아야 한다.

### 5-3) CUDA Toolkit

CUDA Toolkit은 CUDA 개발에 필요한 compiler, library, header, tool을 포함한다. 대표 구성요소는 다음과 같다.

| 구성요소 | 역할 |
|---|---|
| nvcc | CUDA compiler |
| CUDA Runtime | CUDA 프로그램 실행 runtime |
| CUDA Driver API headers | driver-level programming API |
| cuBLAS | BLAS/행렬 연산 라이브러리 |
| cuFFT | FFT 라이브러리 |
| cuSPARSE | sparse matrix 라이브러리 |
| Nsight | profiling/debugging 도구 |

실무에서는 host에 Toolkit 전체를 설치하지 않아도 되는 경우가 많다. 컨테이너 이미지가 필요한 CUDA runtime/library를 포함하고, host는 NVIDIA Driver만 제공하는 구조가 흔하다. 다만 CUDA 코드를 컴파일해야 하거나 custom extension을 build해야 하면 Toolkit이 필요하다.

### 5-4) CUDA Version 관리

CUDA version 관리는 GPU 운영의 핵심이다. container image의 CUDA runtime이 host driver가 지원하는 범위를 벗어나면 실행 실패가 발생한다. 반대로 host에 최신 CUDA Toolkit이 있어도 애플리케이션이 container 내부 runtime을 사용하면 host Toolkit은 거의 의미가 없을 수 있다.

운영 기준은 다음과 같다.

| 항목 | 기준 |
|---|---|
| Host Driver | 클러스터 표준 version 유지 |
| Container CUDA Runtime | application/framework 요구사항에 맞춤 |
| PyTorch/TensorFlow | CUDA build tag 확인 |
| cuDNN/NCCL | framework image에 포함된 버전 확인 |
| Custom extension | build 시 사용한 CUDA와 runtime compatibility 확인 |
| Rollback | driver/image upgrade 실패 시 되돌릴 수 있어야 함 |

### 5-5) CUDA 장애 유형

CUDA 관련 장애는 application, runtime, driver, memory 문제로 나뉜다.

| 장애 | 의미 |
|---|---|
| CUDA out of memory | VRAM 부족 또는 fragmentation |
| invalid device ordinal | 지정한 GPU index가 없음 |
| no kernel image available | GPU architecture와 binary compatibility 문제 |
| driver version is insufficient | host driver가 CUDA runtime 요구사항보다 낮음 |
| illegal memory access | application kernel memory bug 가능성 |
| CUDA initialization error | driver/container/runtime 초기화 문제 |

인프라 엔지니어는 CUDA error를 모두 application 문제로 넘기면 안 된다. driver/runtime compatibility, container GPU device 노출, MIG profile, GPU health까지 확인해야 한다.

## 6. AI Library Stack

### 6-1) cuDNN

cuDNN은 CUDA 위에서 동작하는 딥러닝 연산 최적화 라이브러리다. Convolution, pooling, activation, normalization, RNN 관련 연산 등을 가속한다. PyTorch나 TensorFlow는 내부적으로 cuDNN을 사용한다.

인프라 엔지니어가 알아야 할 점은 cuDNN을 직접 호출하는 방법보다, cuDNN version이 CUDA version 및 framework version과 호환되어야 한다는 것이다. cuDNN support matrix를 기준으로 framework image를 고르고, 같은 클러스터에서 서로 다른 image가 과도하게 섞이지 않도록 관리해야 한다.

### 6-2) Convolution, Pooling, Activation

Convolution은 CNN과 영상/이미지 모델에서 핵심 연산이다. Pooling은 feature map을 축소하거나 요약한다. Activation은 ReLU, GELU, Sigmoid처럼 비선형성을 부여한다. 이런 연산은 GPU에서 대량 병렬 처리되며 cuDNN이 최적화된 kernel을 제공한다.

추론 서버에서 CNN 기반 모델의 latency가 높다면 단순 GPU 성능뿐 아니라 cuDNN version, TensorRT 최적화, batch size, input resolution, precision을 함께 봐야 한다.

### 6-3) NCCL

NCCL은 NVIDIA Collective Communications Library다. 멀티 GPU와 멀티 노드 분산 학습에서 GPU 간 collective communication을 최적화한다. 공식 NCCL 문서는 AllReduce, Broadcast, Reduce, AllGather, ReduceScatter 같은 collective operation을 제공한다.

대표 collective는 다음과 같다.

| 연산 | 의미 | 분산 학습에서의 예 |
|---|---|---|
| AllReduce | 모든 rank의 값을 합산/평균 후 모두에게 배포 | gradient 평균 |
| Broadcast | 한 rank의 값을 다른 rank에 전달 | 초기 parameter 배포 |
| Reduce | 여러 rank 값을 한 rank로 모음 | metric 집계 |
| AllGather | 각 rank 데이터를 모두 모아 모두에게 배포 | sharded tensor 재구성 |
| ReduceScatter | reduce 후 shard 단위로 분산 | ZeRO/FSDP 통신 최적화 |

NCCL은 GPU topology, NVLink, PCIe, NIC, RDMA, network interface를 고려해 통신 경로를 선택한다. NCCL timeout은 실무에서 흔한 장애이며, 원인은 방화벽, DNS, hostname, interface 선택, MTU, RDMA, driver, container network, GPU peer access, node 간 version mismatch 등 다양하다.

### 6-4) TensorRT

TensorRT는 NVIDIA의 추론 최적화 SDK다. 학습용 framework 모델을 추론에 적합하게 최적화하여 latency를 줄이고 throughput을 높인다. TensorRT는 layer fusion, precision calibration, kernel autotuning, engine build 등을 수행한다.

인프라 관점에서 TensorRT는 추론 서비스의 성능 최적화와 관련된다. TensorRT engine은 GPU architecture, TensorRT version, CUDA version, precision, input shape에 영향을 받는다. 따라서 engine artifact를 어떤 환경에서 build했고 어떤 GPU에서 serving하는지 관리해야 한다.

### 6-5) Triton Inference Server

Triton Inference Server는 NVIDIA의 모델 serving 서버다. TensorRT, ONNX Runtime, PyTorch, TensorFlow 등 다양한 backend를 지원하고, dynamic batching, model repository, concurrent model execution, metrics export를 제공한다. Kubernetes 기반 AI serving 플랫폼에서는 Triton을 많이 사용한다.

인프라 엔지니어는 Triton을 사용할 때 GPU memory, model loading, dynamic batching, instance group, Prometheus metrics, readiness/liveness probe, rolling update 전략을 알아야 한다.

## 7. GPU Memory Architecture

### 7-1) VRAM

VRAM은 GPU 전용 메모리다. GPU 연산은 대부분 VRAM에 있는 데이터를 대상으로 수행된다. CPU RAM에 있는 데이터를 GPU가 직접 같은 속도로 처리할 수 있는 것이 아니다. 일반적인 PCIe GPU 환경에서는 CPU RAM에서 GPU VRAM으로 데이터를 복사해야 한다.

AI workload에서 VRAM에는 다음이 올라간다.

| 항목 | 설명 |
|---|---|
| Model parameter | 모델 가중치 |
| Input batch | 현재 처리할 입력 데이터 |
| Activation | forward 중간 결과 |
| Gradient | backward 계산 결과 |
| Optimizer state | Adam 등 optimizer가 유지하는 상태 |
| KV cache | LLM 추론에서 token generation cache |
| Temporary buffer | framework/library 내부 작업 공간 |

### 7-2) GDDR와 HBM

GPU VRAM은 GDDR 또는 HBM 계열을 사용한다. RTX 계열은 주로 GDDR을 사용하고, 데이터센터 고성능 GPU는 HBM을 사용하는 경우가 많다. HBM은 높은 memory bandwidth와 데이터센터 workload에 유리한 특성을 제공하지만 비용이 높다.

| 구분 | GDDR | HBM |
|---|---|---|
| 특징 | 그래픽 카드에 널리 사용 | 고대역폭 메모리 |
| 장점 | 비용 효율, 범용성 | 매우 높은 대역폭 |
| 사용 예 | RTX, 일부 데이터센터 GPU | A100, H100, H200, B200 계열 |
| 운영 관점 | 비용 효율 | 대규모 학습/LLM에 유리 |

### 7-3) Register, Shared Memory, Global Memory

GPU memory hierarchy는 성능에 직접 영향을 준다. Register는 thread별로 가장 빠른 저장 공간이다. Shared memory는 같은 block 내 thread가 공유할 수 있는 빠른 메모리다. Global memory는 VRAM의 큰 영역이며 용량은 크지만 register/shared memory보다 느리다.

운영자는 CUDA 최적화를 직접 하지 않더라도, profiling 결과에서 memory-bound workload인지 compute-bound workload인지 이해해야 한다. GPU utilization은 낮고 memory bandwidth가 높으면 memory-bound일 수 있고, memory bandwidth는 낮고 SM utilization도 낮으면 데이터 공급이나 kernel launch 병목일 수 있다.

### 7-4) Unified Memory

Unified Memory는 CPU와 GPU가 하나의 주소 공간처럼 메모리를 사용할 수 있게 하는 CUDA 기능이다. 개발 편의성을 높이지만, 실제 데이터 이동은 필요하다. 페이지가 CPU와 GPU 사이를 이동하면서 page fault와 migration overhead가 생길 수 있다.

인프라 관점에서는 Unified Memory를 사용한다고 해서 PCIe/NVLink 병목이 사라지는 것이 아니라는 점이 중요하다. workload가 Unified Memory에 과도하게 의존하면 성능이 예측하기 어려울 수 있다.

### 7-5) GPUDirect

GPUDirect는 GPU와 다른 장치 간 데이터 이동을 최적화하기 위한 NVIDIA 기술군이다. GPUDirect RDMA는 NIC 같은 peer device가 GPU memory에 직접 접근할 수 있게 하여 CPU와 system memory를 거치는 복사를 줄인다. GPUDirect Storage는 storage와 GPU memory 간 데이터 경로를 최적화하는 기술이다.

| 기술 | 목적 |
|---|---|
| GPUDirect P2P | GPU 간 직접 데이터 교환 |
| GPUDirect RDMA | NIC/HCA와 GPU memory 간 직접 통신 |
| GPUDirect Storage | Storage와 GPU memory 간 데이터 경로 최적화 |

GPUDirect 계열 기술은 대규모 학습, 고성능 네트워크, 고속 스토리지 환경에서 중요하다. 다만 hardware, driver, kernel, NIC, topology 제약이 있으므로 지원 matrix를 확인해야 한다.

## 8. PCIe

### 8-1) PCIe 구조

PCIe는 CPU와 peripheral device를 연결하는 고속 직렬 인터페이스다. GPU, NVMe, NIC, HBA가 PCIe를 사용한다. PCIe 구조에서 중요한 용어는 lane, switch, root complex다.

| 용어 | 의미 |
|---|---|
| Lane | PCIe 데이터 전송 경로 단위 |
| x16/x8/x4 | 사용 lane 수 |
| Root Complex | CPU 쪽 PCIe controller |
| PCIe Switch | 여러 장치를 연결하기 위한 PCIe switch |
| Endpoint | GPU, NIC, NVMe 같은 PCIe 장치 |

GPU가 x16으로 연결되어야 하는데 x8로 동작하면 bandwidth가 제한될 수 있다. 고속 NIC와 GPU가 같은 root complex 아래에 있는지 여부도 GPUDirect RDMA 성능에 영향을 줄 수 있다.

### 8-2) PCIe Gen3/4/5/6

PCIe generation이 올라갈수록 lane당 bandwidth가 증가한다. 하지만 운영자는 generation 숫자만 볼 것이 아니라 실제 link speed와 link width를 확인해야 한다. BIOS 설정, riser, 케이블, slot 구성, hardware 불량으로 기대한 generation/width보다 낮게 동작할 수 있다.

| 세대 | 운영상 의미 |
|---|---|
| PCIe Gen3 | 구형 서버, bandwidth 제한 가능 |
| PCIe Gen4 | A100 PCIe, 최신 NIC/NVMe 환경에서 흔함 |
| PCIe Gen5 | H100 PCIe, 고속 NIC/NVMe와 함께 중요 |
| PCIe Gen6 | 차세대 고성능 서버에서 중요도 증가 |

### 8-3) GPU 병목: CPU ↔ GPU 전송

CPU RAM에서 GPU VRAM으로 데이터를 복사하는 경로가 느리면 GPU가 기다린다. 이 병목은 다음 상황에서 잘 드러난다.

| 상황 | 설명 |
|---|---|
| 작은 batch | GPU 계산 시간보다 데이터 전송/launch overhead가 큼 |
| CPU 전처리 과다 | GPU가 준비된 batch를 기다림 |
| PCIe link 저하 | x16이 아닌 x8/x4 또는 낮은 generation으로 동작 |
| NUMA 불일치 | 먼 CPU socket의 RAM에서 GPU로 데이터 이동 |
| Storage 병목 | 데이터가 RAM까지 늦게 도착 |

운영자는 GPU utilization만 보지 말고 CPU, RAM, dataloader, PCIe, storage metric을 같이 봐야 한다.

## 9. Multi GPU

### 9-1) Multi GPU가 필요한 이유

하나의 GPU로 모델이 올라가지 않거나, 학습 시간을 줄여야 하거나, throughput을 높여야 하면 Multi GPU를 사용한다. 단일 서버 안에 여러 GPU가 있을 수도 있고, 여러 서버의 GPU를 묶을 수도 있다.

Multi GPU에서는 GPU 간 통신이 중요하다. 각 GPU가 계산한 gradient, parameter, activation, shard를 주고받아야 하기 때문이다. 통신이 느리면 GPU가 서로 기다리면서 utilization이 떨어진다.

### 9-2) GPU Topology: PIX, PXB, PHB, SYS

NVIDIA 도구에서 GPU topology를 보면 PIX, PXB, PHB, SYS 같은 경로가 나온다. 이는 GPU 간 통신 경로가 얼마나 가까운지를 나타낸다.

| 구분 | 의미 | 성능 관점 |
|---|---|---|
| NVLink | GPU 간 전용 고속 링크 | 가장 좋음 |
| PIX | 같은 PCIe switch 아래 | 좋음 |
| PXB | 여러 PCIe bridge/switch 경유 | 중간 |
| PHB | PCIe Host Bridge 경유 | 느릴 수 있음 |
| SYS | CPU socket 간 interconnect까지 경유 | 가장 불리한 편 |

같은 GPU 8장 서버라도 topology에 따라 분산 학습 성능이 달라진다. 운영자는 `nvidia-smi topo -m` 결과를 읽고 GPU 간 통신 경로와 GPU-NIC 근접성을 판단할 수 있어야 한다.

### 9-3) NVLink

NVLink는 NVIDIA GPU 간 고속 interconnect다. PCIe보다 높은 bandwidth와 낮은 latency를 제공하여 GPU 간 직접 통신에 유리하다. Multi GPU 학습에서는 gradient 교환, tensor parallel, pipeline parallel 등에서 NVLink 성능이 중요해진다.

### 9-4) NVSwitch

NVSwitch는 다수 GPU 간 all-to-all 통신을 효율적으로 제공하기 위한 switch fabric이다. GPU가 8장 이상 들어가는 고성능 서버에서는 모든 GPU를 단순히 1:1 NVLink로만 연결하기 어렵다. NVSwitch는 GPU들이 서로 고속으로 통신할 수 있는 fabric을 제공한다.

NVSwitch 기반 서버에서는 Fabric Manager 같은 구성요소가 필요할 수 있다. GPU는 보이지만 NVSwitch fabric이 정상 초기화되지 않으면 멀티 GPU 성능이나 동작에 문제가 생길 수 있다.

### 9-5) NCCL Topology

NCCL은 GPU 간 통신을 위해 topology를 고려한다. 단일 노드에서는 NVLink, NVSwitch, PCIe 경로를 보고, 멀티 노드에서는 NIC, RDMA, InfiniBand/RoCE 구성을 본다. NCCL은 ring, tree 등 다양한 알고리즘을 사용해 collective operation을 수행한다.

운영자는 NCCL 장애 발생 시 다음을 확인해야 한다.

| 확인 항목 | 이유 |
|---|---|
| GPU topology | GPU 간 통신 경로 확인 |
| NIC topology | GPU와 NIC가 같은 PCIe root complex 근처인지 확인 |
| Driver/CUDA/NCCL version | node 간 version mismatch 방지 |
| Network interface | NCCL이 올바른 NIC를 선택했는지 확인 |
| RDMA 상태 | IB/RoCE가 정상인지 확인 |
| MTU/firewall | node 간 통신 차단 여부 확인 |

## 10. GPU 네트워크

### 10-1) Ethernet

Ethernet은 가장 일반적인 네트워크다. 일반 서버에서는 1G/10G가 흔하고, GPU 서버나 스토리지 서버에서는 25G/100G/200G/400G 이상을 사용한다. GPU 추론 서비스에서는 API traffic과 model artifact download, logging, storage access가 Ethernet을 사용한다.

대규모 분산 학습에서는 일반 TCP/IP Ethernet만으로는 latency와 CPU overhead가 병목이 될 수 있다. 이때 RoCE 또는 InfiniBand를 검토한다.

| 속도 | 일반 용도 |
|---|---|
| 10G | 소규모 서버, 관리망, 일반 서비스 |
| 25G | 중간 규모 서비스/스토리지 |
| 100G | GPU 서버, 고성능 스토리지, 클러스터 |
| 200G/400G | 대규모 GPU 클러스터, 고속 RDMA 환경 |

### 10-2) RDMA

RDMA는 Remote Direct Memory Access다. 원격 서버의 메모리에 CPU와 kernel network stack 개입을 줄이고 접근하는 기술이다. RDMA는 낮은 latency, 낮은 CPU overhead, 높은 throughput이 필요할 때 사용된다.

일반 TCP/IP 통신은 NIC, kernel network stack, socket buffer, system call, user-space copy를 거친다. RDMA는 이 경로를 줄이고 NIC/HCA가 memory access를 직접 처리하도록 한다. GPU 클러스터에서는 RDMA를 통해 GPU 간 통신 대기 시간을 줄이고 GPU idle time을 낮출 수 있다.

### 10-3) Zero Copy

Zero Copy는 데이터가 여러 버퍼를 거치며 복사되는 과정을 줄이는 개념이다. 네트워크나 스토리지에서 CPU memory를 여러 번 복사하면 CPU overhead와 latency가 증가한다. RDMA와 GPUDirect RDMA는 이러한 복사와 kernel 개입을 줄여 성능을 높인다.

Zero Copy가 항상 자동으로 되는 것은 아니다. hardware, driver, memory registration, application/library support가 필요하다. 운영자는 “RDMA 지원 NIC가 있다”와 “실제 workload가 RDMA/zero-copy 경로를 사용한다”를 구분해야 한다.

### 10-4) RoCE

RoCE는 RDMA over Converged Ethernet이다. InfiniBand 전용망이 아니라 Ethernet 위에서 RDMA를 구현한다. 기존 Ethernet 기반 데이터센터에 RDMA를 적용할 수 있다는 장점이 있지만, lossless network에 가까운 구성이 필요하다.

RoCE 운영에서 중요한 요소는 다음과 같다.

| 항목 | 설명 |
|---|---|
| PFC | Priority Flow Control, lossless Ethernet 구성에 사용 |
| ECN | Explicit Congestion Notification, congestion 제어 |
| QoS | RDMA traffic 우선순위 관리 |
| MTU | jumbo frame 설정 일관성 |
| Switch buffer | congestion과 packet loss 방지 |
| NIC firmware/driver | RDMA 안정성에 영향 |

RoCE는 잘 구성하면 강력하지만, 잘못 구성하면 packet drop, PFC storm, tail latency 증가, NCCL timeout 문제가 생길 수 있다.

### 10-5) InfiniBand

InfiniBand는 HPC와 대규모 GPU 클러스터에서 사용되는 고성능 네트워크다. 낮은 latency와 높은 bandwidth, RDMA를 제공한다. Ethernet과 프로토콜이 다르며, 전용 HCA, InfiniBand switch, subnet manager가 필요하다.

핵심 용어는 다음과 같다.

| 용어 | 의미 |
|---|---|
| HCA | Host Channel Adapter, InfiniBand 전용 NIC 역할 |
| LID | Local Identifier, IB subnet 내 주소 |
| Subnet Manager | IB fabric 탐색, LID 할당, routing 관리 |
| Fabric | IB switch와 HCA로 구성된 네트워크 전체 |
| Partition | IB fabric 내 논리적 격리 |
| Queue Pair | RDMA 통신을 위한 send/receive queue 쌍 |

InfiniBand는 단순히 빠른 랜카드가 아니다. Ethernet과 다른 fabric 운영 지식이 필요하다. 대규모 GPU 클러스터에서는 IB fabric health, cable/optics error, switch port counter, subnet manager 상태가 중요하다.

### 10-6) GPUDirect RDMA

GPUDirect RDMA는 NIC/HCA 같은 PCIe peer device가 GPU memory에 직접 접근할 수 있게 하는 NVIDIA 기술이다. 일반 경로에서는 GPU data가 CPU system memory를 거쳐 NIC로 이동할 수 있지만, GPUDirect RDMA는 이 복사를 줄인다.

운영 관점에서 확인할 사항은 다음과 같다.

| 항목 | 이유 |
|---|---|
| GPU와 NIC topology | 같은 PCIe root complex 근처가 유리 |
| Driver/NIC 지원 | NVIDIA driver, OFED/rdma-core, NIC firmware 필요 |
| IOMMU 설정 | 성능과 compatibility에 영향 |
| Container/Kubernetes 지원 | device plugin, RDMA device, security 설정 필요 |
| NCCL 사용 여부 | 실제 통신 library가 GPUDirect RDMA를 활용하는지 확인 |

## 11. GPU Storage

### 11-1) GPU와 스토리지 병목

GPU는 데이터를 빠르게 소비한다. 스토리지가 데이터를 충분히 빨리 공급하지 못하면 GPU가 idle 상태가 된다. AI 학습에서는 dataset read, preprocessing, checkpoint write, model artifact 저장이 모두 스토리지와 관련된다.

스토리지 병목은 데이터 패턴에 따라 다르다. 작은 파일이 수백만 개면 metadata operation이 병목이 되고, 큰 shard 파일을 순차적으로 읽으면 throughput이 중요하다. checkpoint는 대용량 write burst를 만든다.

### 11-2) Local NVMe

Local NVMe는 가장 빠르고 단순한 방식이다. 학습 데이터셋을 노드 로컬 NVMe에 staging하면 네트워크 스토리지 병목을 줄일 수 있다. 단점은 데이터 복제, 노드별 일관성, 용량 관리, 장애 시 데이터 재배치가 어렵다는 점이다.

### 11-3) NFS

NFS는 운영이 쉽고 많은 환경에서 기본 공유 스토리지로 사용된다. 하지만 대규모 GPU 학습에서는 병목이 되기 쉽다. 특히 많은 노드가 동시에 작은 파일을 읽으면 metadata server와 network 병목이 발생한다. 소규모 실험에는 적합하지만 대규모 GPU cluster의 주 데이터 경로로는 한계가 있다.

### 11-4) Ceph

Ceph는 object, block, file storage를 제공하는 분산 스토리지다. GPU 인프라에서는 dataset storage, checkpoint storage, model artifact storage, Kubernetes PVC, S3-compatible object storage 용도로 사용할 수 있다.

Ceph를 GPU 학습 경로에 사용할 때는 다음을 고려해야 한다.

| 항목 | 확인 내용 |
|---|---|
| 데이터 패턴 | 작은 파일 랜덤 read인지, 큰 shard sequential read인지 |
| CephFS metadata | 작은 파일 많을 때 MDS 병목 가능 |
| RBD | block storage로 사용 시 latency/throughput 확인 |
| RGW/S3 | object storage로 dataset/model artifact 관리 가능 |
| Replication/EC | 안정성과 성능 trade-off |
| Network | Ceph traffic과 GPU NCCL/RDMA traffic 분리 여부 |
| Checkpoint write | 대용량 write burst가 전체 cluster에 미치는 영향 |

GPU cluster와 Ceph를 같이 운영한다면 storage network와 training network를 분리하는 것이 좋다. 같은 NIC와 switch에서 Ceph replication traffic과 NCCL training traffic이 경쟁하면 성능이 불안정해질 수 있다.

### 11-5) Lustre

Lustre는 HPC 환경에서 많이 사용하는 병렬 파일시스템이다. 대규모 compute node가 동시에 대량 데이터를 읽고 쓰는 workload에 적합하다. GPU cluster에서 대규모 학습 데이터셋을 공유해야 할 때 검토된다. 운영 복잡도는 높지만, HPC/AI 대규모 학습에서는 NFS보다 적합한 경우가 많다.

### 11-6) BeeGFS

BeeGFS도 HPC와 AI workload에서 사용되는 병렬 파일시스템이다. 비교적 구축과 운영이 Lustre보다 접근하기 쉬운 경우가 있으며, metadata와 storage service를 분리해 확장할 수 있다. 대규모 학습 데이터 로딩에 적합한 구조를 만들 수 있다.

### 11-7) 데이터 로딩 병목

GPU 사용률이 낮을 때 가장 흔한 원인 중 하나가 데이터 로딩 병목이다. 확인할 항목은 다음과 같다.

| 확인 항목 | 의미 |
|---|---|
| GPU utilization | GPU가 계산 중인지 확인 |
| CPU utilization | 전처리 병목 여부 |
| Disk read throughput | storage에서 충분히 읽는지 |
| Metadata latency | 작은 파일 병목 여부 |
| Network throughput | 원격 스토리지 병목 여부 |
| Dataloader worker | framework 데이터 로딩 병렬성 |
| Batch preparation time | GPU로 batch가 제때 공급되는지 |
| Checkpoint write time | 학습 중 저장 때문에 멈추는지 |

## 12. Docker GPU

### 12-1) NVIDIA Container Toolkit

NVIDIA Container Toolkit은 container가 host의 NVIDIA GPU를 사용할 수 있게 해주는 도구다. Host에는 NVIDIA Driver가 설치되어 있어야 하며, container runtime은 NVIDIA Container Toolkit을 통해 GPU device와 필요한 library를 container에 전달한다.

구조는 다음과 같다.

```text
Host OS
→ NVIDIA Driver
→ NVIDIA Container Toolkit
→ Docker 또는 containerd
→ Container 내부 CUDA Runtime / PyTorch / TensorFlow
→ GPU 사용
```

### 12-2) Container Runtime

Docker는 단일 서버 테스트에서 많이 사용된다. Kubernetes에서는 containerd가 일반적이다. Docker에서 GPU가 보인다고 Kubernetes Pod에서 자동으로 GPU가 보이는 것은 아니다. Kubernetes에서는 containerd 설정, NVIDIA runtime/CDI, Device Plugin, Pod resource request가 모두 맞아야 한다.

### 12-3) GPU Passthrough

컨테이너에서 GPU를 사용하려면 host의 GPU device와 driver library가 container에 노출되어야 한다. 이 과정을 GPU passthrough라고 부를 수 있다. VM 환경에서는 PCIe passthrough, vGPU, SR-IOV 같은 더 복잡한 방식이 사용될 수 있다.

컨테이너 GPU 접근은 보안과 격리를 고려해야 한다. 무분별한 privileged container, hostPath mount, Docker socket mount는 위험하다. GPU 접근에 필요한 최소 device와 library만 노출하는 것이 원칙이다.

### 12-4) CUDA in Container

컨테이너 이미지는 CUDA runtime, cuDNN, PyTorch/TensorFlow를 포함할 수 있다. 하지만 host driver와 호환되어야 한다. 실무에서는 다음을 지켜야 한다.

| 항목 | 권장 기준 |
|---|---|
| Base image | NVIDIA CUDA/PyTorch 공식 image 또는 내부 표준 image |
| Tag | `latest` 금지, 명시적 version 사용 |
| Driver compatibility | host driver가 container CUDA runtime을 지원해야 함 |
| Image size | 불필요한 build tool 제거, registry 관리 |
| 보안 | 취약점 scan, root 사용자 최소화 |
| 재현성 | requirements/lockfile, image digest 관리 |

## 13. Kubernetes GPU

### 13-1) Device Plugin

Kubernetes는 기본적으로 GPU를 자동 관리하지 않는다. GPU, NIC, FPGA 같은 vendor-specific device는 Device Plugin을 통해 kubelet에 광고된다. NVIDIA Device Plugin은 NVIDIA GPU를 `nvidia.com/gpu` 같은 extended resource로 노출한다.

Pod는 GPU resource를 limits로 요청한다. 일반적으로 GPU는 CPU처럼 millicore 단위로 나누어 overcommit하지 않는다. GPU 공유가 필요하면 MIG, time-slicing, MPS 같은 별도 구성이 필요하다.

### 13-2) GPU Operator

NVIDIA GPU Operator는 Kubernetes에서 GPU 노드에 필요한 NVIDIA software stack을 자동화한다. Driver, NVIDIA Container Toolkit, Device Plugin, GPU Feature Discovery, DCGM Exporter, MIG Manager 등을 관리할 수 있다.

GPU Operator를 사용할 때 결정할 사항은 다음과 같다.

| 항목 | 결정 기준 |
|---|---|
| Driver 설치 방식 | Operator가 설치할지, host pre-install driver를 사용할지 |
| Toolkit 관리 | containerd/Docker runtime 설정 자동화 여부 |
| MIG 사용 | MIG mode/profile 관리 여부 |
| Monitoring | DCGM Exporter 활성화 여부 |
| Air-gapped | 사내 registry와 image mirror 필요 여부 |
| OS support | 노드 OS와 kernel 지원 여부 |

### 13-3) GPU Scheduling

Kubernetes scheduler는 Pod가 요청한 GPU resource를 만족하는 node에 배치한다. 하지만 단순 GPU 개수만 보고 배치하면 GPU 모델, MIG profile, topology, storage locality, network locality를 반영하지 못할 수 있다.

GPU workload 배치에 사용하는 기능은 다음과 같다.

| 기능 | 목적 |
|---|---|
| node label | GPU 모델, zone, network, MIG 여부 표시 |
| nodeSelector | 특정 GPU 노드 선택 |
| nodeAffinity | 더 유연한 노드 선택 조건 |
| taint/toleration | 일반 workload가 GPU node에 올라가지 않게 제어 |
| resource quota | namespace별 GPU 사용량 제한 |
| priority/preemption | 중요한 job 우선 배치 |
| Kueue/Volcano | batch job queue와 gang scheduling 지원 |

### 13-4) Node Label

GPU 노드에는 GPU 모델, memory, MIG 지원, driver version, network type 같은 정보를 label로 관리하는 것이 좋다. GPU Feature Discovery는 GPU 관련 label을 자동으로 생성할 수 있다.

예를 들어 L4는 추론, H100은 대규모 학습, A100 MIG는 공유 실험용으로 분리할 수 있다. label이 없으면 사용자는 특정 GPU를 안정적으로 선택하기 어렵다.

### 13-5) Taint/Toleration

GPU 노드는 비싼 자원이다. 일반 Pod가 GPU node에 스케줄링되면 GPU 서버의 CPU/RAM/스토리지를 사용하면서 GPU workload에 영향을 줄 수 있다. 따라서 GPU node에는 taint를 부여하고, GPU workload에만 toleration을 주는 구성이 일반적이다.

### 13-6) MIG

MIG는 Multi-Instance GPU다. 지원 GPU를 여러 개의 독립 instance로 나누어 사용할 수 있다. MIG instance는 전용 compute와 memory resource를 가진다. NVIDIA 공식 문서는 MIG가 지원 GPU를 여러 isolated instance로 partition하여 여러 사용자/workload가 효율적으로 사용할 수 있게 한다고 설명한다.

MIG는 특히 추론, 개발/테스트, 소형 실험에 유용하다. 하지만 모든 GPU가 MIG를 지원하는 것은 아니며, profile별로 사용할 수 있는 compute slice와 memory가 정해져 있다.

운영자가 알아야 할 MIG 항목은 다음과 같다.

| 항목 | 의미 |
|---|---|
| MIG mode | GPU를 MIG로 사용할지 여부 |
| GI | GPU Instance |
| CI | Compute Instance |
| MIG profile | 1g, 2g, 3g, 7g 등 compute/memory 조합 |
| MIG strategy | Kubernetes Device Plugin에서 MIG 노출 방식 |
| MIG Manager | GPU Operator에서 MIG 구성 관리 |

### 13-7) Time Slicing

GPU time-slicing은 여러 workload가 하나의 GPU를 시간 분할 방식으로 공유하게 하는 기능이다. NVIDIA GPU Operator 문서에 따르면 time-slicing은 GPU oversubscription을 가능하게 하고, workload들이 interleave 방식으로 실행된다.

Time-slicing은 MIG처럼 강한 하드웨어 격리를 제공하지 않는다. 따라서 성능 예측성과 보안 격리는 낮지만, 작은 추론/개발 workload를 많이 수용하는 데 도움이 될 수 있다.

### 13-8) Kubernetes GPU 운영 장애

| 증상 | 확인 지점 |
|---|---|
| Node에 GPU resource가 안 보임 | Device Plugin, GPU Operator, driver, kubelet |
| Pod Pending | GPU 부족, taint/toleration, nodeSelector 불일치 |
| Pod 안에서 GPU 안 보임 | Container Toolkit, runtimeClass, CDI, securityContext |
| MIG resource 안 보임 | MIG mode, profile, migStrategy, GPU Operator 설정 |
| DCGM metric 없음 | dcgm-exporter, 권한, Prometheus scrape 설정 |
| GPU 노드에 일반 Pod 배치 | taint/toleration 미구성 |
| 특정 GPU만 과부하 | scheduler 정책, quota, 사용자 쏠림 |

## 14. GPU Monitoring

### 14-1) nvidia-smi

`nvidia-smi`는 GPU 상태를 빠르게 확인하는 기본 도구다. Driver version, CUDA compatibility 표시, GPU name, temperature, power, memory usage, utilization, process 목록을 확인할 수 있다.

다만 `nvidia-smi`는 운영 모니터링의 전부가 아니다. 장기간 metric 수집, Prometheus 연동, alerting, Kubernetes Pod 단위 분석에는 DCGM Exporter가 더 적합하다.

### 14-2) DCGM

DCGM(Data Center GPU Manager)은 데이터센터 GPU 관리와 진단을 위한 NVIDIA 도구다. GPU health check, diagnostics, metric 수집에 사용된다. DCGM Exporter는 DCGM Go API를 사용해 GPU metric을 HTTP `/metrics` endpoint로 노출하고 Prometheus가 수집할 수 있게 한다.

### 14-3) Prometheus

Prometheus는 DCGM Exporter에서 GPU metric을 scrape하여 저장한다. Kubernetes에서는 dcgm-exporter를 DaemonSet으로 배포하고, Prometheus Operator ServiceMonitor로 수집하는 구성이 흔하다.

### 14-4) Grafana

Grafana는 GPU metric을 시각화한다. GPU node별 utilization, memory, temperature, power, ECC error, Pod별 사용량, MIG instance별 사용량 등을 dashboard로 구성한다.

### 14-5) AlertManager

AlertManager는 Prometheus alert rule을 기반으로 알림을 보낸다. GPU 운영에서는 다음 알림이 중요하다.

| Alert | 의미 |
|---|---|
| GPU temperature high | 냉각 문제, thermal throttling 위험 |
| GPU utilization low while job running | 데이터 공급 병목 가능성 |
| GPU memory high | OOM 위험 |
| ECC error 증가 | GPU memory 이상 가능성 |
| Xid error 발생 | GPU/driver/hardware fault 가능성 |
| GPU not visible | driver/module/hardware 문제 |
| DCGM exporter down | 모니터링 사각지대 발생 |
| Pod pending by GPU | GPU resource 부족 또는 scheduling 문제 |

### 14-6) GPU 주요 지표

| 지표 | 해석 |
|---|---|
| GPU Utilization | GPU compute 사용률. 낮으면 idle 또는 병목 가능 |
| SM Utilization | SM compute unit 사용률 |
| Tensor Utilization | Tensor Core 활용 정도 |
| Memory Utilization | VRAM 대역폭 사용 정도 |
| Memory Used/Free | VRAM 용량 사용량 |
| Temperature | GPU 온도 |
| Memory Temperature | HBM/GDDR 온도 |
| Power Draw | 현재 전력 소비 |
| Power Limit | 전력 제한 값 |
| Throttle Reason | thermal/power 등 성능 제한 원인 |
| ECC Error | corrected/uncorrected memory error |
| PCIe Throughput | CPU-GPU/NIC-GPU 경로 병목 파악 |
| NVLink Throughput | GPU-GPU 통신량 |

GPU utilization이 낮다고 항상 문제가 있는 것은 아니다. 추론 서비스는 요청이 없으면 낮을 수 있다. 하지만 학습 job이 실행 중인데 utilization이 낮다면 CPU, storage, dataloader, PCIe, NCCL, network 병목을 의심해야 한다.

## 15. 분산 학습

### 15-1) Data Parallel

Data Parallel은 같은 모델을 여러 GPU에 복제하고, 각 GPU가 서로 다른 mini-batch를 처리하는 방식이다. 각 GPU가 계산한 gradient를 AllReduce로 평균내고 모델을 동기화한다. 가장 기본적인 분산 학습 방식이다.

장점은 이해하기 쉽고 구현이 상대적으로 단순하다는 점이다. 단점은 모델 전체가 각 GPU VRAM에 올라가야 하므로 모델이 너무 크면 사용할 수 없다는 점이다.

### 15-2) Model Parallel

Model Parallel은 모델 자체를 여러 GPU에 나누어 올리는 방식이다. 모델이 단일 GPU VRAM에 들어가지 않을 때 사용한다. 레이어 단위, tensor 단위, pipeline 단위 등 여러 방식이 있다.

### 15-3) Pipeline Parallel

Pipeline Parallel은 모델의 layer를 여러 GPU에 나누고, micro-batch를 pipeline처럼 흘려보내는 방식이다. GPU 간 activation 전달이 필요하며, pipeline bubble을 줄이는 것이 중요하다.

### 15-4) Tensor Parallel

Tensor Parallel은 하나의 layer 내부 tensor 연산을 여러 GPU로 나누는 방식이다. 대규모 Transformer/LLM 학습에서 중요하다. GPU 간 통신량이 많아 NVLink/NVSwitch, InfiniBand, NCCL 성능이 매우 중요하다.

### 15-5) DDP

DDP(Distributed Data Parallel)는 PyTorch의 대표적인 분산 학습 방식이다. 각 process가 하나의 GPU를 담당하고 gradient synchronization을 수행한다. DDP는 NCCL backend를 많이 사용한다.

인프라 엔지니어는 DDP job이 실패할 때 다음을 확인해야 한다.

| 확인 항목 | 이유 |
|---|---|
| GPU 수와 process 수 | rank/world size 불일치 방지 |
| NCCL backend | GPU 통신 사용 여부 |
| Hostname/DNS | node 간 통신 가능 여부 |
| Network interface | NCCL이 올바른 NIC를 사용하는지 |
| Firewall/port | rendezvous와 NCCL 통신 차단 여부 |
| Driver/CUDA/NCCL version | node 간 일관성 |

### 15-6) FSDP

FSDP(Fully Sharded Data Parallel)는 모델 parameter, gradient, optimizer state를 여러 GPU에 shard하여 VRAM 사용량을 줄이는 방식이다. 큰 모델을 학습할 때 중요하다. 통신 패턴이 복잡해지고 network/NCCL 성능에 더 민감해질 수 있다.

### 15-7) DeepSpeed

DeepSpeed는 대규모 모델 학습 최적화를 위한 라이브러리다. ZeRO optimizer stage를 통해 optimizer state, gradient, parameter를 shard하여 VRAM 사용량을 줄인다. DeepSpeed를 사용하면 단일 GPU에 들어가지 않는 큰 모델도 여러 GPU/노드에 나누어 학습할 수 있다.

운영자는 DeepSpeed job이 많은 GPU와 network를 장시간 점유한다는 점, checkpoint가 매우 클 수 있다는 점, NCCL/RDMA/storage 성능이 중요하다는 점을 알아야 한다.

### 15-8) 분산 학습에서 인프라 병목

| 병목 | 증상 |
|---|---|
| GPU compute 부족 | GPU utilization 높고 step time 길음 |
| VRAM 부족 | OOM, batch size 축소 필요 |
| NCCL 통신 병목 | GPU utilization sawtooth, AllReduce 시간 증가 |
| Network 병목 | 멀티 노드 확장 효율 낮음 |
| Storage 병목 | GPU utilization 낮고 dataloader 대기 증가 |
| CPU 병목 | dataloader/preprocessing worker 과부하 |
| Checkpoint 병목 | 저장 시 job pause 증가 |

## 16. GPU Scheduler

### 16-1) Slurm

Slurm은 HPC cluster에서 널리 쓰이는 workload manager다. Slurm은 node, CPU, memory, GPU 같은 자원을 job 단위로 할당하고 queue, partition, QoS, fair-share 정책을 제공한다. Slurm에서 GPU는 GRES(Generic RESource)로 관리하는 경우가 많다.

Slurm은 연구소, HPC, 슈퍼컴퓨팅, 대규모 batch 학습 환경에 적합하다. 사용자는 job script를 제출하고, Slurm이 적절한 GPU node에 작업을 배치한다.

### 16-2) Ray

Ray는 Python 친화적인 분산 컴퓨팅 프레임워크다. Python function과 actor를 분산 실행하고, CPU/GPU resource를 task 단위로 지정할 수 있다. Ray Train, Ray Tune, Ray Serve 등을 통해 분산 학습, 하이퍼파라미터 튜닝, 모델 서빙을 구성할 수 있다.

인프라 엔지니어 관점에서 Ray는 GPU cluster 자원을 Python 개발자가 쉽게 사용할 수 있게 하는 실행 계층이다. Ray를 Kubernetes 위에서 운영하려면 KubeRay, autoscaling, head/worker pod, resource request, dashboard, logging을 관리해야 한다.

### 16-3) Kubernetes

Kubernetes는 container orchestration 플랫폼이다. GPU를 extended resource로 관리하고, Pod 단위로 할당한다. 추론 서비스, MLOps 플랫폼, 내부 AI 플랫폼, Kubeflow, KServe, Triton serving 등에 적합하다.

Kubernetes만으로 batch GPU job의 gang scheduling이나 queue 정책이 충분하지 않을 수 있다. 이때 Volcano나 Kueue 같은 도구를 함께 검토한다.

### 16-4) Volcano

Volcano는 Kubernetes 기반 batch workload scheduler다. AI, big data, HPC workload를 Kubernetes에서 더 잘 다루기 위해 gang scheduling, queue, priority 같은 기능을 제공한다. 분산 학습처럼 여러 Pod가 동시에 떠야 하는 workload에 유리하다.

### 16-5) Kueue

Kueue는 Kubernetes-native job queueing 시스템이다. 여러 namespace/team의 batch workload를 queue로 관리하고 resource quota, admission control을 제공한다. GPU가 제한된 환경에서 사용자별/팀별 공정한 자원 사용을 관리하는 데 유용하다.

### 16-6) Scheduler 선택 기준

| 환경 | 적합한 도구 |
|---|---|
| 연구소/HPC batch 학습 | Slurm |
| Python 기반 분산 처리/튜닝 | Ray |
| AI 서비스 추론/MLOps | Kubernetes |
| Kubernetes batch/gang scheduling | Volcano |
| Kubernetes quota 기반 job queue | Kueue |
| 모델 서빙 | Kubernetes + KServe/Triton |

## 17. GPU 장애 대응

### 17-1) GPU 안 잡힘

GPU가 아예 안 잡히면 가장 아래 계층부터 본다. PCIe 장치로 보이는지, BMC hardware event log가 있는지, BIOS 설정이 맞는지, riser/cable/power 문제가 없는지 확인한다. 그 다음 Linux kernel, NVIDIA driver module, device file 생성 여부를 본다.

### 17-2) Driver 장애

Driver 장애는 `nvidia-smi` 실패, kernel module load 실패, version mismatch, DKMS build 실패, Secure Boot module signing 문제로 나타난다. Kernel update 이후 발생하는 경우가 많다.

대응 기준은 다음과 같다.

1. PCIe 장치 인식 여부 확인
2. NVIDIA module load 여부 확인
3. Kernel version과 module build 상태 확인
4. User-space library와 kernel module version 일치 확인
5. Secure Boot 차단 여부 확인
6. 필요 시 표준 driver version으로 재설치

### 17-3) CUDA 장애

CUDA 장애는 application 실행 시 나타난다. 대표적으로 `driver version is insufficient`, `CUDA out of memory`, `invalid device ordinal`, `no kernel image available`, `illegal memory access` 등이 있다.

CUDA 장애는 application만의 문제가 아닐 수 있다. Driver compatibility, container CUDA runtime, GPU architecture, MIG profile, framework version을 함께 확인한다.

### 17-4) GPU OOM

GPU OOM은 VRAM이 부족할 때 발생한다. 원인은 모델이 너무 크거나, batch size가 크거나, 여러 프로세스가 같은 GPU를 공유하거나, memory leak이 있거나, MIG profile이 작거나, fragmentation이 발생했기 때문일 수 있다.

운영자는 OOM이 발생한 GPU, process, Pod, container, model, batch size, MIG profile, VRAM 사용량 변화를 기록해야 한다.

### 17-5) ECC Error

ECC Error는 GPU memory 오류다. Correctable error는 수정 가능하지만 반복 증가하면 하드웨어 이상 신호일 수 있다. Uncorrectable error는 workload 실패나 GPU reset으로 이어질 수 있다. ECC error가 증가하는 GPU는 교체 검토 대상이 될 수 있다.

### 17-6) Xid Error

Xid Error는 NVIDIA Driver가 보고하는 GPU 오류다. Xid code에 따라 application fault, memory error, PCIe error, GPU reset, thermal 문제 등을 의심할 수 있다. 반복 Xid는 반드시 장애 티켓으로 관리해야 한다.

### 17-7) NCCL Timeout

NCCL Timeout은 멀티 GPU/멀티 노드 학습에서 자주 발생한다. 원인은 network interface 선택 오류, DNS/hostname 문제, firewall, port 차단, MTU mismatch, RDMA 실패, node 간 version mismatch, 특정 rank hang, GPU fault 등이 있다.

대응 순서는 다음과 같다.

1. 모든 rank/node가 정상 실행 중인지 확인
2. node 간 network reachability 확인
3. NCCL이 선택한 NIC 확인
4. RDMA/InfiniBand/RoCE 상태 확인
5. MTU/firewall/security group 확인
6. Driver/CUDA/NCCL/PyTorch version 일치 확인
7. 특정 GPU의 Xid/ECC/thermal 문제 확인

### 17-8) RDMA 장애

RDMA 장애는 NCCL 성능 저하나 timeout으로 나타날 수 있다. HCA 상태, driver/firmware, OFED/rdma-core, subnet manager, RoCE PFC/ECN, MTU, switch counter, cable/optic error를 확인해야 한다.

### 17-9) GPU Hang

GPU Hang은 workload가 멈추거나 GPU가 reset되지 않고 응답하지 않는 상태다. 원인은 application kernel bug, driver bug, hardware fault, thermal, power, PCIe issue, long-running kernel watchdog 설정 등 다양하다. 운영자는 강제 프로세스 종료, GPU reset 가능 여부, 노드 drain/reboot 기준을 정해야 한다.

### 17-10) 장애 대응 체계

GPU 장애는 다음 계층으로 나누어 대응한다.

```text
Hardware/BMC
→ PCIe/Linux Kernel
→ NVIDIA Driver
→ CUDA Runtime
→ Container Runtime
→ Kubernetes Device Plugin
→ Application Framework
→ Distributed Communication
```

장애 보고서에는 최소한 노드명, GPU UUID, driver version, CUDA runtime, container image, workload, 발생 시각, Xid/ECC, temperature, power, Pod/job 정보, 재현 여부가 포함되어야 한다.

## 18. GPU 데이터센터

### 18-1) Rack 설계

GPU rack은 일반 서버 rack보다 전력 밀도와 냉각 요구가 높다. GPU 서버 1대가 수 kW를 사용할 수 있으므로 rack당 장착 가능한 서버 수는 전력과 냉각 용량에 의해 제한된다. 단순히 U 공간이 남는다고 서버를 더 넣을 수 없다.

### 18-2) Power

전력 설계에서는 서버 최대 소비전력, PSU 이중화 방식, PDU 용량, rack당 전력, UPS backup 시간, 전원 이중화 경로를 고려한다. GPU 서버는 peak power와 sustained power를 구분해서 봐야 한다. AI 학습은 장시간 full load를 만들기 때문에 평균 전력도 높다.

### 18-3) Cooling

GPU 데이터센터에서는 냉각이 핵심이다. 공랭에서는 cold aisle/hot aisle, airflow direction, blanking panel, rack 배치, fan speed, inlet temperature가 중요하다. 고밀도 GPU rack에서는 공랭만으로 한계가 있어 liquid cooling을 검토한다.

### 18-4) PDU

PDU는 rack 내 장비에 전력을 공급한다. GPU rack에서는 PDU 용량, outlet 수, phase balance, metering 기능, remote power control이 중요하다. Intelligent PDU를 사용하면 서버별 전력 사용량을 모니터링할 수 있다.

### 18-5) UPS

UPS는 정전이나 전원 품질 문제에서 장비를 보호한다. GPU 서버는 전력 소모가 커서 UPS 용량 산정이 중요하다. 모든 GPU workload를 장시간 유지하려는 목적보다는 안전한 shutdown, storage consistency, control plane 유지 목적일 수 있다.

### 18-6) Liquid Cooling

Liquid cooling은 고밀도 GPU 서버에서 중요성이 커지고 있다. Direct-to-chip cooling, rear-door heat exchanger, immersion cooling 등 방식이 있다. 운영자는 누수 감지, coolant 관리, 유지보수 절차, rack compatibility, facility 요구사항을 확인해야 한다.

### 18-7) AIDC

AIDC는 AI Data Center를 의미하는 맥락에서 사용된다. 일반 데이터센터보다 GPU/AI workload에 최적화된 전력, 냉각, 고속 네트워크, 대용량 스토리지, AI workload scheduler, 보안, 모니터링 체계를 갖춘 데이터센터로 이해하면 된다.

AIDC에서 중요한 설계 축은 다음과 같다.

| 축 | 내용 |
|---|---|
| Compute | GPU 서버, CPU, 메모리, accelerator |
| Network | InfiniBand/RoCE, Ethernet, management network |
| Storage | Object, parallel filesystem, NVMe cache |
| Power | 고밀도 rack power, UPS, PDU |
| Cooling | liquid cooling, airflow, thermal monitoring |
| Platform | Kubernetes, Slurm, Ray, MLOps |
| Observability | GPU/DCGM, network, storage, job metrics |
| Governance | quota, cost, security, audit |

## 20. 실무 학습 순서

### 20-1) 1단계: GPU 컴퓨팅 기초

CPU/GPU 차이, thread/warp/block/grid, SM, SIMT, Tensor Core, precision, VRAM 구조를 이해한다. 이 단계는 GPU가 왜 AI workload에 적합한지 이해하는 기반이다.

### 20-2) 2단계: GPU 서버 하드웨어

CPU, NUMA, RAM, PCIe, GPU, VRAM, PSU, cooling, BMC를 학습한다. GPU 서버는 고성능 연산 장비이면서 동시에 고전력/고발열 서버라는 점을 이해해야 한다.

### 20-3) 3단계: Linux와 NVIDIA Driver

PCIe 인식, kernel module, DKMS, `/dev/nvidia*`, driver compatibility, Xid/ECC error를 학습한다. `nvidia-smi` 실패를 계층적으로 분석할 수 있어야 한다.

### 20-4) 4단계: CUDA와 AI Library Stack

CUDA runtime/driver API, cuDNN, NCCL, TensorRT, Triton의 역할을 학습한다. 버전 호환성 문제를 판단할 수 있어야 한다.

### 20-5) 5단계: Docker와 Kubernetes GPU

NVIDIA Container Toolkit, containerd, Device Plugin, GPU Operator, MIG, time-slicing, taint/toleration, GPU quota를 학습한다.

### 20-6) 6단계: Monitoring과 장애 대응

DCGM, Prometheus, Grafana, AlertManager, Xid, ECC, OOM, NCCL timeout, RDMA 장애를 운영 사례 중심으로 학습한다.

### 20-7) 7단계: Multi GPU, Network, Storage

NVLink, NVSwitch, NCCL topology, InfiniBand, RoCE, GPUDirect RDMA, Ceph, Lustre, BeeGFS, data loading 병목을 학습한다.

### 20-8) 8단계: Scheduler와 AI 플랫폼

Slurm, Ray, Kubernetes, Volcano, Kueue, KServe, Kubeflow, Triton serving을 조직 workload에 맞게 조합하는 방법을 학습한다.
